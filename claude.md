# CLAUDE.md

## 项目定位
MioForever 是一个个人陪伴型 AI 助手项目。
当前状态是：设计已确认，模块待施工。
本文件是本项目的施工规范；若与全局 CLAUDE.md 冲突，以本文件为准。

## 最高原则
- 先审查方向，再进入实现。
- 先找已有方案、研究、开源实现，再决定是否自研。
- 不一头扎进代码。
- 不顺着设想直接写，先判断设想是否成立。
- 所有改动都要能追溯到当前蓝图中的某一阶段或某一决策点。
- 遇到不清晰不理解的信息必须先停下来向用户确认或商讨。

## 施工顺序
默认按以下阶段推进，不跳阶段：
1. 底盘
2. Mio 主引擎移植
3. 行为引擎
4. 检索精选注入
5. 记忆网 + 反馈闭环
6. 多 Agent 框架
7. GUI 重设计
8. 配置分享生态
9. 认知提炼

如果当前任务不属于当前阶段，不要强行展开，先说明阶段归属。

## 研究优先规则
当用户提出设想、架构、技术路线、Agent 循环、记忆系统、配置体系、GUI 方案时：
- 先查找类似实现、相关研究、开源项目、成熟方案
- 先比较现成方案，再决定自研
- 若存在更成熟或更简单的路线，必须主动提出
- 若信息不足，先补查，再设计，再实现
- 不把“看起来能做”当成“值得做”

## 审查规则
回答前必须检查：
- 这个方向是否已有成熟实现
- 当前方案是否过度复杂
- 是否存在更低成本路径
- 是否把推测当成事实
- 是否遗漏了蓝图里已有的待定点或依赖关系

## 蓝图约束
- 以 `MioForever工程设计蓝图.md` 为当前设计依据。
- 任何新增模块都要说明它属于哪一阶段。
- 任何技术选型都要说明替代方案与取舍。
- 任何实现都要区分：已确认、待验证、待设计、远期构想。
- 不要把未定事项写成定论。

## 当前重点
当前优先关注这些待定项：
- 底盘拆解对象：Pi SDK / HanaAgent / Hermes / Reasonix
- 零件验收标准：什么叫“能用”
- 记忆网落地方式：存储层、worker、队列
- GUI 技术选型：Electron 或 Tauri
- llm_wiki 的接入方式
- 配置包 `.mio-pack` 的解析与加载
- 多 Agent 框架和开放 API 的接口边界

## 输出要求
- 先结论，后依据。
- 方案要能落地，尽量给步骤。
- 涉及架构时，必须说明风险和代价。
- 涉及技术选型时，至少给出两个方案的比较。
- 涉及不确定内容时，明确标注不确定点。
- 完成后说明：做了什么、为什么这样做、下一步是什么。

## 工作方式
- 先定位蓝图与相关文件，再动手。
- 先拆问题，再给方案。
- 先判断是否需要查资料，再写实现。
- 不做无关重构，不扩大任务范围。
- 不跳过审查，不省略验证。

## 禁止事项
- 不直接写代码替代方案审查。
- 不把单点猜测包装成整体结论。
- 不跳过阶段顺序直接做后续模块。
- 不把项目当前蓝图以外的假设写成事实。

## 方向质疑
当用户提出一个方向时，不默认它正确。
必须先审查：
- 是否已有更成熟方案
- 是否存在更简单路线
- 是否值得继续做
- 是否与当前阶段目标一致
