# 外部 Agent 设计建议提示词

你是一位架构师，正在审查一个名为 MioForever 的 AI 项目。

## 项目核心：马具工程

MioForever 只做三件事：

1. **配置导入**：用户怎样定义和导入一份"马具配置"（思维链规则、行为约束、验证标准）。配置的格式是什么？该有怎样的模板结构？
2. **注入执行**：配置如何在 LLM 推理前注入 system prompt（或其他注入位置）。注入时机、方式、分层是什么？
3. **校验**：LLM 回复后，如何检验回复是否遵守了配置规则。如何验证语义层的约束（如"是否指出了假设""是否多角度"）？

其他一切（多 Agent 框架、GUI、生态分享、认知提炼）都是叠在这三个核心之上的应用层，不是当前重点。

## 已有的设计基础

蓝图中有这些已确认的设计决策：
- 配置 = 文本注入 + 验证规则（每一项配置天然是一对）
- 三层优先级：用户本地 > 导入配置包 > 出厂默认
- 8 Section 作为可扩展的启动种子（用户画像、交互风格、思维方式、行为策略、校验回路、推理模板、约束松弛度、反思提示词）
- 配置热切换：下一轮 BUILD 生效

## 已有的实验结论

做了一个思维链回环原型：
- 思维链（"先拆问题→查假设→多视角→不迎合一→下结论"）注入 system prompt
- LLM 回复后拦截，发送自检提示词让它自查是否遵守了规则
- 结论：语义层验证（"是否指出了假设"）只能用 LLM 本身来做，正则或 JSON Schema 做不到。但 LLM 自检本身不稳定——PASS 判定不一致、有时不遵守指令

## 约束

- 必须支持多 LLM（Claude/GPT/DeepSeek）
- 不依赖特定框架或闭源 SDK
- 配置格式应该是可分享、可版本化的（如 .json 或 .mio-pack）

## 希望得到的建议

请对以上三个核心需求，分别给出设计方案建议。可以补充可能遗漏的设计点，可以指出现有设计的问题，也可以提供更好的实现路径。直接给结论，不需要复述背景。
