# 进化 — 项目自循环完善

> 完整规格见 `docs/evolution-loop-spec.md` v3.0。
> 分类学是第一个接入进化循环的项目。

---

## 核心原则

| 原则 | 含义 |
|------|------|
| 主Agent只调度 | 立意确认后主Agent不做研究判断，只派发任务和汇报 |
| 搜索Agent相互隔离 | 各自独立，不知道其他Agent的存在和进展 |
| 策略差异保证独立性 | 每个搜索Agent持有不同策略标签，收敛才有意义——不是靠随机种子 |
| 审计Agent独立 | 必须是全新Agent，不能是搜索池中的任何一个 |
| 收敛才落地 | 未收敛不进入执行。满 3 轮仍不收敛 → 分歧上抛用户 |
| 备份先于执行 | 改动前必须留快照 |
| 只新增不替换 | ADR 和决策树都是 append-only，标注时间戳 |

## Phase 1 — 立意

### Step 1：理解与拓展（主Agent）

分两步走——先读目标，再对照现状。

**① 回验项目立意**

先读项目的主文档（index.md / README / CLAUDE.md 开头），提取项目自己声称的目标：

> 这个项目声称自己是什么？要达成什么？

这是整个进化循环的参照系。后续所有观察都围绕一个问题：**项目离自己的立意还有多远？**

**② 四诊对照**

拿着项目立意，做四诊：

| 诊 | 问什么 |
|----|--------|
| **望** | 对照立意，项目目前实际是什么状态？差距在哪？ |
| **闻** | 有哪些隐性需求——项目立意暗示了但还没做的事？ |
| **问** | 用户说"进化 X"，真正想解决的问题是什么？（不一定是字面意思） |
| **切** | 有哪些约束、禁区、前置条件？ |

产出：**拓展需求** = 用户意图 + 立意差距 + 边界条件

### Step 2：立意确认（主Agent ↔ 用户）

向用户反问，格式如下：

```
你的需求是不是：[拓展后的需求]？
特别想确认：[最不确定的那一条]
```

用户确认 → 形成**立意**。立意一经确认，全流程冻结，不得修改。
偏离立意的结论 = 错误，无论结论本身是否合理。

> **分界线**：立意确认后，主Agent只负责调度和汇报，不做方向判断。

---

## Phase 2 — 竞争搜索

### Step 3：独立搜索（子Agent × N，N ≥ 3）

主Agent同时派出 N 个搜索子Agent，每个Agent收到：
- 完整立意
- 唯一策略标签（各Agent标签必须不同）
- 必要的项目上下文

每个Agent不知道其他Agent的存在、策略、进展。（隔离上下文，不是空上下文）

**标准策略集**（N = 3 时取前三，N 更大时向后扩展）：

| 优先级 | 标签 | 搜索视角 |
|--------|------|---------|
| 1 | 成功案例型 | 同类问题被成功解决的案例，提取可复用模式 |
| 2 | 反例型 | 同类方案失败或带来副作用的案例，识别风险 |
| 3 | 边界条件型 | 技术/实现层面的约束与上限，判断可行性边界 |
| 4 | 竞品参照型 | 成熟方案的设计决策，提取可借鉴处 |
| 5 | 第一性原理型 | 从问题本质推导，不依赖案例 |

每个Agent产出格式：

```
策略标签：[...]
搜索结论：[基于此策略的发现]
推荐方向：[具体的方向建议]
置信度：高 / 中 / 低
主要风险：[1-3条]
```

---

## Phase 3 — 收敛审计

> ⚠️ 审计Agent必须是全新独立Agent，不能是Phase 2搜索池中的任何一个。

### Step 4：比对审计（审计Agent，可循环，最多3轮）

拿到所有搜索子Agent的产出，依次执行：

**① 合规检查**
- 每个Agent是否按所分配的策略标签执行？
- 有无明显偏题、逻辑跳跃或信息缺失？

**② 偏离审查**
- 各Agent的推荐方向是否与立意对齐？
- 标记所有偏离立意的结论（标记为无效，不参与方向判断）

**③ 收敛判断**

满足以下全部条件视为收敛：
- 有效方向（未被偏离标记）全部一致，或差异仅在实现细节层面
- 无有效方向明显互斥

| 判断结果 | 处置 |
|----------|------|
| ✅ 收敛 | 进入 Step 5 |
| ❌ 方向分歧 | 启动调整合Agent（见下） |
| ❌ 某Agent偏离立意 | 打回该Agent，带修正指令重搜，结果纳入本轮审计 |

**调整合Agent**（分歧时触发）：
- 接收所有分歧方向（合并上下文——不在分叉中持续分裂）
- 研究分歧根源，尝试形成统一方向或判定孰优孰劣
- 产出新的统一推荐 → 送回 Step 4 审计

**终止条件**：
- 收敛 → 正常退出进入 Step 5
- 满 3 轮仍未收敛 → 主Agent将分歧方向列表上报用户，由用户选择后继续

### Step 5：最终合规审计（同审计Agent）

对收敛结果做终检：
- 方向与立意完全对齐？
- 操作清单逻辑自洽、无遗漏？
- 是否存在未处理的关键风险？

通过 → 主Agent接收结果，进入汇报。

---

## Phase 4 — 汇报与执行

### Step 6：结果汇报（主Agent → 用户）

```
方向：[X]
选择理由：[Y]
要做的操作：
  1. [操作A]
  2. [操作B]
  ...
已知风险：[风险列表]
```

同步写入 **ADR 条目**（方向级决策，只写一次，不在执行阶段修改）：

```
时间戳：[timestamp]
立意：[确认的需求]
选定方向：[...]
选择理由：[...]
放弃的方向：[...] 及原因
信息来源：搜索Agent × N，策略标签：[列表]
```

等待用户确认。

---

用户确认后，顺序执行以下步骤：

### Step 7：备份

对当前项目状态留快照。
- 轻量项目：文件副本
- Git项目：commit 当前状态
- **备份完成后**，进入 Step 8

### Step 8：执行

主Agent按操作清单逐项执行。

执行过程中同步维护**决策树**（执行级，append-only + 时间戳）：

```
[timestamp] 执行：[操作X]
  → 选择 [方案A]，原因：[...]（备选：[方案B]，排除原因：[...]）
[timestamp] 遭遇：[异常或分叉]
  → 处置：[...]
```

## 产出边界速查

| 产出 | 谁写 | 何时写 | 是否可修改 |
|------|------|--------|-----------|
| 立意 | 主Agent + 用户 | Step 2 | ❌ 冻结 |
| 搜索结论 | 搜索子Agent × N | Step 3 | — |
| ADR | 主Agent | Step 6 | ❌ 只写一次 |
| 决策树 | 主Agent | Step 8（持续） | ❌ 只追加 |

**ADR vs 决策树的边界**：
- ADR：为什么走这条路（方向级）
- 决策树：走的过程中遇到什么、怎么选的（执行级）

## 和分类学技能的关系

```
进化（本技能）
  ├── 调用 writer（写配置规则）
  ├── 调用 tools-config（配 MCP/Hook/权限）
  ├── 调用 discovery（搜工具/技能）
  ├── 调用 session-finder（查会话找根因）
  └── 调用 session-to-memory（归档经验）

进化是调度者。分类学技能是它的工具。
进化自己也可以被进化——每次跑完循环，检查循环本身有哪里要修。
```

## 名词解释

| 术语 | 含义 |
|------|------|
| 进化 | 项目的自循环完善机制——从需求理解到执行沉淀的完整闭环 |
| 立意 | Step 2 用户确认后的精确目标。全流程冻结，后续所有步骤的锚点 |
| 策略标签 | 分配给搜索 Agent 的唯一搜索视角，用结构化多样性保证收敛有意义 |
| 隔离上下文 | 每个搜索 Agent 只知道自己的策略标签和立意，不知道其他 Agent 的存在 |
| 收敛 | 有效方向全部一致且无互斥，差异仅在实现细节层面 |
| 偏离 | 结论偏离立意。在审计中标记为无效，打回重搜 |
| 调整合 | 分歧时将不同方向合并到一个上下文，研究分歧根源，尝试统一 |
| ADR | 方向级决策记录——"为什么走这条路" |
| 决策树 | 执行级改动记录——"走的过程中遇到什么"，append-only + 时间戳 |
