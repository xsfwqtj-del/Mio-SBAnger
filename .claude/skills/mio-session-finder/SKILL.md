# Mio Session Finder — 会话追溯

> 从 CC 会话 JSONL 中搜索追溯信息。用于根因分析、设计决策回溯、死引用溯源。

## 触发条件

- 用户想知道某段代码/配置/决策是在哪个会话中引入的
- 发现死引用、不一致、设计冲突，需要追溯原始上下文
- 用户说"查会话""找会话""追溯""根因""是什么时候引入的"

## 执行流程

### 1. 定位会话目录

CC 会话按项目隔离存储。先确认当前项目，再定位对应目录：

```
C:\Users\<用户名>\.claude\projects\
```

项目目录命名规则：盘符路径转哈希。`D:\Mio-SBAnger` → `D--Mio-SBAnger`。
其他盘符同理：`C:\path` → `C--path`，`D:\path` → `D--path`。

**先查当前项目目录**（最新会话在这里）：

```bash
ls -lt "C:/Users/xsfwq/.claude/projects/D--Mio-SBAnger/"*.jsonl | head -10
```

**如果当前项目没有**，查 `D-----`（全局/临时项目目录）：

```bash
ls -lt "C:/Users/xsfwq/.claude/projects/D-----/"*.jsonl | head -10
```

**备选数据源**（按需）：
- 全局历史：`C:\Users\<用户名>\.claude\history.jsonl`
- daemon 日志：`C:\Users\<用户名>\.claude\daemon.log`
- 会话元数据：`C:\Users\<用户名>\.claude\sessions\`

### 2. 搜索策略

按用户问题选择搜索方式：

| 用户想找 | 搜索方式 | 说明 |
|---------|---------|------|
| 某文件/配置最早在哪引入 | grep 文件名 → 找最早的 Write 调用 | 用 `"file_path".*文件名` 匹配 |
| 某个设计决策的讨论 | grep 关键术语 | 用术语本身做关键词 |
| 死引用/不一致的根因 | grep 死引用路径 → 找写入该引用的 Write 调用 | 用不存在的路径做关键词 |
| 最近改了什么 | 按 mtime 排序列出会话文件 | `ls -lt` 列出最近 5 个会话 |

### 3. 提取上下文

找到匹配行后提取（JSONL 每行一条完整 JSON，可逐行解析）：

- **会话 ID** — `sessionId` 字段
- **时间戳** — `timestamp` 字段
- **操作类型** — `Write` / `Edit` / `Bash` 等工具调用的 `input` 参数
- **文件名** — `file_path` 字段（如果存在）
- **slug** — 会话的计划 slug（如 `jiggly-purring-ripple`）

**不读整个文件**：JSONL 文件可能 2-3 MB。用 `rg`/`grep` 搜索关键行，只在需要上下文时才读前后几行。

### 4. 输出

```markdown
## 会话追溯: {搜索关键词}

### 匹配记录

| 会话 | 时间 | 操作 | 目标 |
|------|------|------|------|
| {sessionId 前 8 位} | {timestamp} | {Write/Edit/Bash} | {文件路径或命令摘要} |

### 根因分析
- {基于会话内容和时间线的分析}

### 建议
- {基于发现的行动建议}
```

如果搜索结果是会话本身的讨论内容（而非文件写入），标注为"**讨论级**"——即此会话讨论了该话题但可能不是最终实现的会话。

## 关键原则

- **会话文件只读**：只搜索分析，不修改会话文件
- **优先最近会话**：按 mtime 排序，最新会话最可能包含当前问题的答案
- **不读全文**：用 grep/rg 搜关键行，避免把 MB 级 JSONL 灌入上下文
- **跨项目搜索**：一个问题可能涉及多个项目目录（当前项目 + D----- 全局目录）
- **区分讨论和实现**：会话中讨论了 ≠ 会话中实现了。确认有 Write/Edit 调用才算实现

## 边界

**负责**:
- 搜索 CC 会话 JSONL 找到特定信息
- 追溯配置/代码/决策的引入时间和上下文
- 分析根因（死引用、不一致、设计冲突的会话级来源）
- 列出最近会话的时间线和操作摘要

**不负责**:
- 不修改会话文件
- 不生成新配置或代码（那是 writer / tools 的事）
- 不代替 session-to-memory（它做归档提取，我做搜索追溯）
- 不做通用文本搜索（那是 Grep 的事）

**交接**:
- 找到根因后需要修配置文本 → "建议调 writer: 帮我写 {维度}"
- 找到根因后需要修工具配置 → "建议调 tools-config: 帮我配 {工具}"
- 需要从会话中提取记忆归档 → "建议调 session-to-memory"
- 不清楚项目当前状态 → "建议调 evolve: 检查项目"

## 名词解释

| 术语 | 含义 |
|------|------|
| JSONL | CC 会话存储格式。每行一条完整的 JSON 消息，包含 tool_use、assistant、user 等类型 |
| sessionId | CC 会话的唯一标识（UUID），在会话文件的每条消息中重复出现 |
| slug | 会话的简短标识（如 `jiggly-purring-ripple`），用于跨引用计划文件 |
| projects/ | CC 按 hash(项目路径) 隔离会话的存储目录 |
| D----- | 全局/临时工作目录的哈希名，不在特定项目下的会话存在这里 |
