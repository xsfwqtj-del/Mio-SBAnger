# Product Capability — 意图→工程契约

将产品意图转成显式工程约束。不是"该做什么"，而是"在实施之前必须成立什么"。

## 何时使用

- 蓝图中有模糊的模块定义（如"8 Section Profile""四层注入""四边权重"），需要精确化
- 功能跨多个模块/阶段，编码前需要能力契约
- 设计意图清晰，但架构、数据、生命周期、策略影响仍然模糊
- 反复在审查中重申相同的隐藏假设
- 需要可跨会话重用的持久化产物

## 针对 MioForever 蓝图的使用场景

蓝图审查报告识别出 5 矛盾 + 18 缺口（1 P0、10 P1、7 P2）。每个缺口填入时，先用此技能：
- P0 缺口 #1（8 Section Profile）→ 定义 8 个 Section 的接口契约
- P1 缺口 #2（四层注入）→ 定义 stable/volatile/brief/hard 的约束和不变量
- P1 缺口 #4（Hook 接口签名）→ 定义 7 个 Hook 点的参数/返回值契约
- P1 缺口 #10（ProfileData 中间表示）→ 定义 schema 和映射规则

## 不可协商规则

- 不要发明产品事实。未解决的问题显式标注。
- 区分用户可见承诺和实现细节。
- 标明什么是固定策略、什么是架构偏好、什么仍是开放的。
- 如果请求与已有仓库约束冲突，明确说而不是平滑处理。
- 优先一个可重用的能力产物，而非散落的临时笔记。

## 工作流

### 1. 重述能力
将需求压缩为一句精确陈述：谁、什么新能力、什么结果因它改变。
如果这句陈述薄弱，实现会漂移。

### 2. 解析能力约束
提取实施前必须成立的约束：
- 业务规则
- 范围边界
- 不变量
- 信任边界
- 数据所有权
- 生命周期转换
- 发布/迁移要求
- 失败和恢复预期

### 3. 定义实现契约
产出 SRS 风格的能力计划：
- 能力摘要
- 显式 non-goals（不属于本模块的）
- 参与者和接触面
- 所需状态和转换
- 接口 / 输入 / 输出
- 数据模型影响
- 安全 / 策略约束
- 可观测性和运维需求
- 阻塞实现的开放问题

### 4. 翻译为执行
精确交棒：
- 可直接实现
- 需先做架构审查
- 需先做产品澄清

## 输出格式

```text
能力
- 一段话重述

约束
- 固定规则、不变量、边界

实现契约
- 参与者
- 接触面
- 状态和转换
- 接口/数据影响

Non-goals
- 本模块显式不负责的

开放问题
- 阻塞实现的产品决策

交棒
- 下一步干什么
```
