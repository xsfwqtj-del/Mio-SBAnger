# Search First — 先搜再做

将 MioForever 蓝图"研究优先规则"系统化：先查找已有实现，再决定是否自研。

## 触发条件

- 开始新功能开发（可能已有现成方案）
- 添加依赖或集成
- 用户说"加个 X 功能"而你要开始写代码之前
- 创建新工具、辅助函数或抽象之前

## 决策矩阵

| 信号 | 行动 |
|------|------|
| 精确匹配、维护良好、MIT/Apache | **直接采用** |
| 部分匹配、基础不错 | **扩展** — 安装 + 薄包装 |
| 多个弱匹配 | **组合** — 2-3 个小包拼装 |
| 无合适方案 | **自研** — 但基于研究结果设计，不凭空造 |

## 工作流

### 0. 工具可用性预检

| 渠道 | 检查方式 | 无此渠道时 |
|------|---------|-----------|
| 仓库内搜索 | `rg --files` + 定向 `rg` | 仅说明检查了可见文件 |
| 包注册表 | 项目包管理器 --version | 用 web/docs 搜索 |
| GitHub | `gh auth status` | 只用公开 web 搜索 |
| 技能目录 | `ls .claude/skills/` | 说明无本地技能目录 |

### 1. 需求分析
定义需要什么功能、语言/框架约束。

### 2. 并行搜索
- npm/PyPI 包注册表
- MCP 服务器
- 本地已有技能
- GitHub 开源实现
- Web 搜索

### 3. 评估
按功能完整性、维护状态、社区、文档、许可证、依赖数评分。

### 4. 决策
→ 采用 / 扩展 / 组合 / 自研

### 5. 实施
安装包 / 配置 MCP / 写最小自定义代码。

## 针对 MioForever 蓝图的特殊规则

根据蓝图"研究优先规则"第 4 条："若信息不足，先补查，再设计，再实现"：
- 每个技术决策前必须走一轮 search-first
- 每条 choice 必须说明替代方案和取舍
- 不确定内容明确标注"不确定"，不写成定论

## 快速模式（内联）

写代码前心里过一遍：
1. 仓库内是否已存在？→ `rg` 检查
2. 是否是常见问题？→ 搜 npm/PyPI
3. 有没有 MCP？→ 检查已有 MCP 配置
4. 有没有已有技能？→ 检查 `.claude/skills/`
5. 有没有 GitHub 实现？→ GitHub 代码搜索
