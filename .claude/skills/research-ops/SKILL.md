# Research Ops

证据优先的研究操作流程。不是替代 `deep-research` 或 `WebSearch`，而是告诉你什么时候用哪个、怎么组合。

## 技能栈

根据情况拉入这些技能：
- `WebSearch` + `WebFetch` — 快速当前网络发现（首选）
- `deep-research` — 需要多源综合+引用的重研究（用户明确说"深度研究"才用）
- `search-first` — 编码前的"先搜现成方案"检查

## 何时使用

- 用户说"研究""查一下""对比""谁适合聊""最新的是什么"
- 答案依赖当前公开信息
- 用户已提供部分证据，需要纳入新的推荐
- 可能变成持续监控而非一次性查询

## 护栏

- 不要从陈旧记忆中回答当前问题——搜索很便宜
- 严格区分：
  - 来源事实
  - 用户提供的证据
  - 推断
  - 建议
- 不要为本地代码/文档就能回答的问题启动重研究

## 工作流

### 1. 从用户已给你的信息开始
将已有材料标准化为：
- 已有证据的事实
- 需验证
- 开放问题

不要从零重启分析。

### 2. 对提问分类
- 快速事实回答
- 对比或决策备忘录
- 深度研究报告
- 候选持续监控

### 3. 走最轻的有效证据路径
- 用 `WebSearch` + `WebFetch` 进行快速发现
- 需要综合或多源时升级到 `deep-research`（用户明确说才用）
- 本地代码/文档已有答案时不搜索

### 4. 报告时标注证据边界
对重要声明标注：
- 来源事实
- 用户提供的上下文
- 推断
- 建议

时效敏感答案必须包含具体日期。

### 5. 判断是否应持续跟踪
如果用户可能反复问同一研究问题，明确提出并建议监控层而非永远重复手动搜索。

## 输出格式

```text
问题类型
- 事实 / 对比 / 决策 / 监控候选

证据
- 来源事实
- 用户提供的上下文

推断
- 从证据中得出的结论

建议
- 答案或下一步
- 是否应转为持续监控
```

## 陷阱

- 不要把推断混入来源事实而不标注
- 不要忽略用户提供的证据
- 不要为本地仓库上下文能回答的问题走重研究
- 不要给时效敏感答案不加日期
