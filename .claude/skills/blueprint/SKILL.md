# Blueprint — 施工计划生成器

将一行目标转化为任何 Agent 都可以冷启动执行的分步施工计划。

## 何时使用

- 将大型功能拆分为多个 PR，有明确的依赖顺序
- 规划跨多次会话的重构或迁移
- 协调多个子 Agent 的并行工作流
- 任何会话间上下文丢失会导致返工的任务

**不适用**：单次 PR 可完成、少于 3 次工具调用、用户说"直接做"。

## 工作方式

5 阶段管线：

1. **Research（研究）** — 飞行前检查（git、gh auth、remote、默认分支），然后读取项目结构、已有计划和记忆文件收集上下文。
2. **Design（设计）** — 将目标拆分为单 PR 大小的步骤（通常 3-12 步）。分配依赖边、并行/串行排序、模型层级（最强 vs 默认）、每步回滚策略。
3. **Draft（草稿）** — 将自包含 Markdown 计划文件写入 `plans/`。每步包含上下文简报、任务列表、验证命令、退出标准——新 Agent 无需阅读前面步骤即可执行任一步。
4. **Review（审查）** — 委托最强模型子 Agent 进行对抗审查，对照清单和反模式目录。修复所有关键发现后才定稿。
5. **Register（注册）** — 保存计划、更新记忆索引、向用户展示步骤数和并行摘要。

Blueprint 自动检测 git/gh 可用性。有 git + GitHub CLI 时生成完整 branch/PR/CI 工作流。没有时切换到直接模式（原地编辑，无分支）。

## 关键特性

- **冷启动执行** — 每步包含自包含上下文简报，不需要先前上下文。
- **对抗审查门禁** — 每个计划由最强模型子 Agent 审查完整性、依赖正确性、反模式检测。
- **Branch/PR/CI 工作流** — 内置于每步。无 git/gh 时优雅降级为直接模式。
- **并行步骤检测** — 依赖图识别无共享文件或输出依赖的步骤。
- **计划变更协议** — 步骤可拆分、插入、跳过、重排、废弃，带正式协议和审计追踪。
- **零运行时风险** — 纯 Markdown 技能，无 hook、无 shell 脚本、无可执行代码。

## 示例

### 基本用法

```
/blueprint myapp "将数据库迁移到 PostgreSQL"
```

产出 `plans/myapp-将数据库迁移到-postgresql.md`，步骤如：
- Step 1: 添加 PostgreSQL 驱动和连接配置
- Step 2: 为每个表创建迁移脚本
- Step 3: 更新 repository 层使用新驱动
- Step 4: 添加针对 PostgreSQL 的集成测试
- Step 5: 移除旧数据库代码和配置

### 多 Agent 项目

```
/blueprint chatbot "将 LLM provider 提取为插件系统"
```

产出带并行步骤的计划（如"实现 Anthropic 插件"和"实现 OpenAI 插件"在插件接口步骤完成后并行运行），模型层级分配（接口设计步用最强模型，实现步用默认），每步后验证不变量（如"所有已有测试通过""核心无 provider 导入"）。

## 要求

- Claude Code（用于 `/blueprint` 斜杠命令）
- Git + GitHub CLI（可选——启用完整 branch/PR/CI 工作流；Blueprint 检测缺失并自动切换到直接模式）
