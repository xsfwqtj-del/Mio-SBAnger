# Architecture Decision Records

在编码会话发生时捕获架构决策。不让决策只存在于聊天记录、PR 评论或某人记忆中，而是产出与代码共存的结构化 ADR 文档。

## 何时激活

- 用户明确说"记录这个决策"或"ADR 这个"
- 用户在重要替代方案中做选择（框架、库、模式、数据库、API 设计）
- 用户说"我们决定用……"或"我们选 X 不做 Y 的原因是……"
- 用户问"为什么我们选了 X？"（读取已有 ADR）
- 讨论架构权衡的计划阶段

## ADR 格式

使用 Michael Nygard 的轻量 ADR 格式，适配 AI 辅助开发：

```markdown
# ADR-NNNN: [决策标题]

**日期**: YYYY-MM-DD
**状态**: proposed | accepted | deprecated | superseded by ADR-NNNN
**决策者**: [谁参与了]

## 上下文

什么情况促使了这个决策或变更？

[2-5 句描述情况、约束和推动力]

## 决策

我们提出的变更是什么？

[1-3 句清晰陈述决策]

## 考虑过的替代方案

### 替代方案 1: [名称]
- **优点**: [好处]
- **缺点**: [缺点]
- **为什么不选**: [被拒绝的具体原因]

### 替代方案 2: [名称]
- **优点**: [好处]
- **缺点**: [缺点]
- **为什么不选**: [被拒绝的具体原因]

## 后果

因为这个变更，什么变得更容易或更困难？

### 积极的
- [好处 1]
- [好处 2]

### 消极的
- [权衡 1]
- [权衡 2]

### 风险
- [风险及缓解措施]
```

## 工作流

### 捕获新 ADR

当检测到决策时刻：

1. **初始化（仅首次）** — 如果 `docs/adr/` 不存在，先问用户确认再创建目录、`README.md`（含索引表头）和空白 `template.md`。未获同意不创建文件。
2. **识别决策** — 提取核心架构选择
3. **收集上下文** — 什么问题促使了这个决策？有什么约束？
4. **记录替代方案** — 考虑过哪些其他选项？为什么被拒绝？
5. **陈述后果** — 权衡是什么？什么变得更容易/更难？
6. **分配编号** — 扫描 `docs/adr/` 中已有 ADR 并递增
7. **确认后写入** — 先展示 ADR 草稿给用户审查。仅在明确批准后才写入 `docs/adr/NNNN-decision-title.md`。用户拒绝则丢弃草稿。
8. **更新索引** — 追加到 `docs/adr/README.md`

### 读取已有 ADR

当用户问"为什么我们选了 X？"：

1. 检查 `docs/adr/` 是否存在 → 不存在则回复"此项目未找到 ADR。要开始记录架构决策吗？"
2. 存在则扫描 `docs/adr/README.md` 索引找相关条目
3. 读取匹配的 ADR 文件，展示上下文和决策部分
4. 未找到匹配则回复"未找到该决策的 ADR。要现在记录吗？"

### ADR 目录结构

```
docs/
└── adr/
    ├── README.md              ← 所有 ADR 的索引
    ├── 0001-use-nextjs.md
    ├── 0002-postgres-over-mongo.md
    ├── 0003-rest-over-graphql.md
    └── template.md            ← 手动使用的空白模板
```

### ADR 索引格式

```markdown
# Architecture Decision Records

| ADR | 标题 | 状态 | 日期 |
|-----|------|------|------|
| [0001](0001-use-nextjs.md) | 使用 Next.js 作为前端框架 | accepted | 2026-01-15 |

```

## 决策检测信号

### 显式信号
- "我们选 X"
- "我们应该用 X 而不是 Y"
- "这个权衡值得因为……"
- "记录为 ADR"

### 隐式信号（建议记录 ADR——不要未经用户确认自动创建）
- 比较两个框架或库并得出结论
- 做数据库 schema 设计选择并陈述理由
- 在架构模式间选择（单体 vs 微服务、REST vs GraphQL）
- 决定认证/授权策略
- 评估替代方案后选择部署基础设施

## 什么算好 ADR

### 要做
- **具体** — "使用 Prisma ORM"不是"使用一个 ORM"
- **记录为什么** — 理由比决策内容更重要
- **包含被拒绝的替代方案** — 未来开发者需要知道考虑过什么
- **诚实地陈述后果** — 每个决策都有权衡
- **保持简短** — ADR 应该在 2 分钟内可读完
- **使用现在时** — "我们使用 X"不是"我们将使用 X"

### 不要
- 记录琐碎决策 — 变量命名或格式化选择不需要 ADR
- 写论文 — 上下文部分超过 10 行就太长了
- 省略替代方案 — "我们就选了这个"不是有效理由
- 不标注就补录 — 如果记录的是过去的决策，注明原始日期
- 让 ADR 过时 — 被取代的决策应引用其替代者

## ADR 生命周期

```
proposed → accepted → [deprecated | superseded by ADR-NNNN]
```

- **proposed**: 决策在讨论中，尚未提交
- **accepted**: 决策生效并正在遵循
- **deprecated**: 决策不再相关（如功能已移除）
- **superseded**: 更新的 ADR 替代此 ADR（始终链接替代者）

## 值得记录的决策类别

| 类别 | 示例 |
|------|------|
| **技术选型** | 框架、语言、数据库、云服务商 |
| **架构模式** | 单体 vs 微服务、事件驱动、CQRS |
| **API 设计** | REST vs GraphQL、版本策略、认证机制 |
| **数据建模** | Schema 设计、规范化决策、缓存策略 |
| **基础设施** | 部署模型、CI/CD 管线、监控栈 |
| **安全** | 认证策略、加密方法、密钥管理 |
| **测试** | 测试框架、覆盖率目标、E2E vs 集成测试平衡 |
| **流程** | 分支策略、审查流程、发布节奏 |
