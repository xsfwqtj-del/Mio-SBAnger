# ADR-0001: 四层注入与 8 Section 配置模型的映射关系

**日期**: 2026-06-10
**状态**: proposed
**决策者**: 烬汐

## 上下文

MioForever 蓝图 (2026-06-10 版) 同时定义了两套配置注入模型：

- **四层注入**（二⑤ · 底盘配置注入管道）：`stable / volatile / brief / hard`，描述内容注入到 system prompt 的不同层次
- **8 Section Profile**（三.4 · 〇.B）：用户画像 / 交互风格 / 思维方式 / 行为策略 / 校验回路 / 推理模板 / 约束松弛度 / 反思提示词，面向用户的配置维度

两套模型分别出现在底盘设计和配置体系设计中，但没有定义它们之间的关系。实现底盘⑤的开发者会做出四层注入，实现三.4 的开发者会做出 8 Section 注入——两个结果互不兼容。这是配置体系的主干接口，歧义在这里等于整个配置分享层（五）施工前地基已裂。

2026-06-10 外部审查将此问题标记为 S2（会导致整个系统失败的根问题）。

## 决策

**采用双模型分层方案**：8 Section 作为面向用户的逻辑模型（配置视角），四层注入作为 BUILD 阶段的内部实现机制（注入视角），两者通过显式映射表关联。

映射关系：

| 四层注入 | 映射内容 | KV 缓存策略 |
|---------|---------|------------|
| **L0 hard** | 宪法层不可变规则（安全边界 · 平台约束） | 完全静态，编译期固定 |
| **L1 stable** | 骨架层——Mio 身份 · 工具描述 · 记忆指令 · 出厂默认值 | 版本控制，跨会话不变 |
| **L2 volatile** | 8 Section 当前激活值——用户画像 / 交互风格 / 思维方式 / 行为策略 / 校验回路 / 推理模板 / 反思提示词 | 会话级，BUILD 阶段组装 |
| **L3 brief** | 烬织检索精选简报（≤10 条 ≤1200 字符）+ 会话元数据 | 拼入 user message 末尾，不破坏 prefix 缓存 |

**不在四层注入中的 8 Section 项**：
- **约束松弛度**（strict / flexible / creative）——纯代码级参数，不注入 prompt，在 [5][6a] Hook 点消费
- **反思提示词**——注入 L2 volatile，但其对应的验证规则注册在 [5b] POST_THINK

## 考虑过的替代方案

### 替代方案 1: 废弃四层注入命名，全部用 8 Section 驱动 BUILD
- **优点**: 概念统一，只需理解一套模型；减少文档维护负担
- **缺点**: 丢弃四层注入的物理价值——KV 缓存命中策略依赖 stable/volatile 的分层边界；hard 层（宪法规则）和 stable 层（工具描述）的不可变性需要不同于 Section 配置的表达方式
- **为什么不选**: 四层注入解决的是"怎么注入才能最大化 KV 缓存命中"，8 Section 解决的是"用户可以配置什么"。两者是不同的关注点，用一个模型替代另一个会丢失物理层的优化信息。保留双模型、定义映射是更务实的方案

### 替代方案 2: 8 Section 直接映射到四层，每 Section 自选注入层
- **优点**: 最灵活，每个 Section 可以独立选择注入到哪一层
- **缺点**: 给配置创作者暴露了 KV 缓存机制——中阶用户需要理解"哪个 Section 放哪一层"才能做出正确的配置包；增加了配置格式的复杂度
- **为什么不选**: 配置创作者不应关心 KV 缓存。Section 的注入层级应该是 Pack Loader 的内部行为，由系统决定而不是由配置作者决定

## 后果

### 积极的
- 两个已有设计资产都不丢弃——四层注入保留其缓存优化价值，8 Section 保留其用户配置价值
- 实现者有明确的映射表可参照，不会做出互不兼容的实现
- 四层注入对配置创作者完全透明——写 .mio-pack 的人不需要知道 stable / volatile 的区别
- KV 缓存命中策略有明确的边界：L0+L1 完全静态跨会话不变，L2 会话级可变但保持在 system prompt 末尾，L3 拼入 user message 末尾

### 消极的
- 开发者需要同时理解两套概念——学习成本翻倍
- 如果未来需要新增 Section，必须同时回答"它映射到哪一层"——增加了设计步骤
- 如果 L2 volatile 的内容频繁变化（用户每轮切换配置），system prompt 的 KV 缓存会部分失效

### 风险
- **映射僵化风险**：如果某个新 Section 不适合现有的四层框架，需要扩展注入层定义。缓解：四层不是封闭的——如果出现合理的第五种注入需求，可以新增 L2.5 或调整 L2 的语义
- **实现偏差风险**：Pack Loader 开发者和 Hook 开发者可能对映射有不同理解。缓解：此 ADR 的映射表是规范性定义，所有相关模块以此表为准
- **与底盘选型耦合**（S1）：如果底盘最终选型有自己的 prompt 组装机制，本决策需要重新评估。缓解：底盘评估矩阵中增加"是否支持自定义分层注入"作为验收标准之一
