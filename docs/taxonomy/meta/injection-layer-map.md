# 注入层级映射

> 分类学第四轴：元属性 → 注入层级 | 引用 ADR-0001

---

## 一、定义

**每项配置注入到 system prompt 的哪一层。这决定了它在 KV 缓存中的位置和生命周期。**

注入层级对配置创作者透明——写 CLAUDE.md 的人不需要知道 L0/L1/L2/L3。但这个映射表对实现者（自己搭建 Agent 的人）是必要的。

---

## 二、四层注入模型

| 层 | 语义 | 生命周期 | KV 缓存 | 内容 |
|----|------|---------|---------|------|
| **L0 宪法** | 不可变的安全边界 | 编译期固定 | 100% 命中 | 平台约束 / 不可变的安全规则 |
| **L1 骨架** | Agent 的身份和工具描述 | 跨会话不变 | 版本控制 | Mio 身份 · 工具描述 · 记忆指令 · 出厂默认 |
| **L2 人格** | 用户的配置选择 | 会话级 | BUILD 组装 | 用户画像 · 交互风格 · 思维方式 · 行为策略 · 技能触发规则 · 推理模板 · 反思提示词 |
| **L3 上下文** | 当前会话的动态信息 | 每轮变化 | 拼入 user msg 末尾 | 记忆简报 · 会话元数据 |

---

## 三、9 维度 → 四层映射

| 维度 | 注入层 | 说明 |
|------|--------|------|
| **行为策略** | L0（安全红线）+ L2（操作纪律） | 安全红线在 L0 不可被覆盖，操作纪律在 L2 可调整 |
| **校验回路** | 不注入 prompt | 校验回路注册到 Hook 点（PRE_TOOL/POST_TOOL/POST_THINK），LLM 不直接感知 |
| **推理模板** | L2 | 输出结构属于人格层 |
| **工具可用性** | L1（工具清单）+ 代码参数（权限矩阵） | 工具清单在 L1 静态缓存，权限矩阵在 PRE_TOOL Hook 消费 |
| **技能触发规则** | L2（触发条件）+ 代码参数（注册表） | 触发条件注入 prompt 让 LLM 知道何时调用 |
| **用户画像** | L2 | 身份和背景随项目变化 |
| **交互风格** | L2 | 语气和表达方式 |
| **思维方式** | L2 | 思维原则 |
| **反思提示词** | L2 | 反思问题集 |

---

## 四、不在四层注入中的项

- **约束松弛度** — 纯代码参数，在 POST_THINK / PRE_TOOL Hook 点消费
- **工具权限矩阵** — 代码参数，在 PRE_TOOL Hook 消费
- **技能注册表** — 代码参数，由技能加载器消费

---

## 五、与 ADR-0001 的关系

本映射表继承 ADR-0001 的双模型分层方案：
- 8 Section（现 9 维度）= 逻辑模型（面向用户）
- 四层注入 = 物理模型（面向实现者）

详见：`docs/adr/0001-four-layer-injection-vs-8-section-mapping.md`

---

## 六、CC 源码验证

CC 源码泄露暴露的静态/动态分界处（`SYSTEM_PROMPT_DYNAMIC_BOUNDARY`）与本模型的 L1/L2 分界一致：

| CC 源码 | 本模型 |
|---------|--------|
| 静态段（全局缓存）— 身份/安全/行为规则/工具偏好 | L0 宪法 + L1 骨架 |
| 动态段 — CLAUDE.md / MEMORY.md / MCP / 环境 | L2 人格 + L3 上下文 |

CC 的静态段相当于我们的 L0+L1（跨会话不变），动态段相当于我们的 L2+L3（会话/每轮可变）。说明四层注入模型与生产级 CC 的实践一致。
