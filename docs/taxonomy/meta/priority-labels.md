# 优先级标签体系

> 分类学第四轴：元属性 → 优先级标签

---

## 一、定义

**当多个配置源（全局/项目/技能）、多个配置项之间产生冲突时，谁说了算。**

优先级标签是每项配置的元属性——它不是配置内容本身，而是描述"这项配置和其他配置的关系"。

---

## 二、三级标签

| 标签 | 语义 | 行为 |
|------|------|------|
| `[覆盖]` | 凌驾于其他项之上 | 冲突时此项胜出，忽略冲突项的相关部分 |
| `[补充]` | 叠加在其他项之上 | 冲突时合并两者，本项附加于其他项之后（**默认**） |
| `[建议]` | LLM 可自由选择 | 冲突时 LLM 根据上下文自行判断 |

---

## 三、按维度的推荐优先级

| 维度 | 推荐标签 | 理由 |
|------|---------|------|
| 行为策略（安全红线） | `[覆盖]` | 安全不可妥协 |
| 行为策略（操作纪律） | `[覆盖]` | 纪律 > 风格 |
| 校验回路 | `[覆盖]` | 验证规则不能被他项放松 |
| 推理模板 | `[补充]` | 模板可以叠加，但不能互相覆盖 |
| 工具可用性（权限矩阵） | `[覆盖]` | deny 列表不可被覆盖 |
| 工具可用性（纪律） | `[补充]` | 纪律可叠加 |
| 技能触发规则 | `[补充]` | 技能规则可按场景累加 |
| 用户画像 | `[补充]` | 全局画像 + 项目画像 |
| 交互风格 | `[建议]` | 风格冲突时 LLM 自行判断 |
| 思维方式 | `[补充]` | 思维原则可叠加 |
| 反思提示词 | `[补充]` | 反思问题可跨源合并 |

---

## 四、多源冲突消解

### 4.1 三层配置源

```
用户全局 (C:\Users\xsfwq\.claude\CLAUDE.md)  >  项目 (.\CLAUDE.md)  >  出厂默认
```

全局 CLAUDE.md 的定义有最高优先级——它是用户的核心规则。

### 4.2 冲突消解流程

```
配置 A 和 配置 B 冲突
  │
  ├── 同一维度？
  │     ├── 是 → 比较优先级标签
  │     │         [覆盖] > [补充] > [建议]
  │     │         同级 → 比较配置源
  │     │                  全局 > 项目 > 出厂
  │     │
  │     └── 否 → 先检查维度优先级
  │               行为策略 [覆盖] 交互风格
  │               校验回路 [覆盖] 推理模板
  │               如果维度互不冲突 → 各自生效
  │
  └── 同源同标签 → LLM 自行判断 + 记录审计
```

### 4.3 标签不可降级

`[覆盖]` 项不能被 `[建议]` 项覆盖。但如果用户自己写了一条 `[建议]` 而全局是 `[覆盖]`，以全局为准——用户的"建议"无法覆盖自己的"铁律"。

---

## 五、ECC 对标

ECC 的规则没有显式的优先级系统。34 条规则平等共存，冲突时由 LLM 判断。在实际使用中，这可能导致以下问题：

- TDD 规则 (强制 80% 覆盖率) 和快速原型技能 ("先做出能用的") 冲突
- 没有机制决定谁优先

本分类学的优先级标签系统可以直接补这个口。

---

## 六、使用建议

- 默认全部用 `[补充]` —— 大多数配置不需要互相覆盖
- 安全红线用 `[覆盖]` —— 而且写在全局 CLAUDE.md 里
- 只在明确知道"这一项必须胜过其他项"时才用 `[覆盖]`
- 不确定时「先补充、观察冲突、再升级」
