# ADR-0002: Agent 行为配置分类学框架设计

**日期**: 2026-06-11
**状态**: proposed
**决策者**: 烬汐

## 上下文

MioForever 蓝图（2026-06-10）定义了三层配置体系：8 Section Profile（用户画像/交互风格/思维方式/行为策略/校验回路/推理模板/约束松弛度/反思提示词）、四层注入（stable/volatile/brief/hard）、配置分享生态（.mio-pack / Pack Loader / 穿搭）。

2026-06-11 讨论归档对蓝图做了诚实推翻，结论是：所有超过"文本卡片"的复杂度（记忆网慢通道、ITN 集体智慧网络、多 Agent 框架、配置分享生态）都是过度设计。CLAUDE.md 就是答案。

但 CLAUDE.md 缺一个完整的分类学——没人系统性地定义过"一个 Agent 的行为配置应该覆盖哪些维度，每个维度怎么写才有效"。

本 ADR 记录分类学框架设计中的关键决策。

## 决策

### D1: 摒弃 8 Section 封闭框架，采用可扩展维度模型

**决策**：从 8 Section 扩展到 9 个内容维度（5 通用 + 4 个人），并在分类学层面明确——这 9 个不是封闭框架，后续可扩展。

**调整细节**：

| 原 8 Section | 分类学中的归宿 | 理由 |
|-------------|--------------|------|
| 用户画像 | 保留 → 个人维度 | — |
| 交互风格 | 保留 → 个人维度 | — |
| 思维方式 | 保留 → 个人维度 | — |
| 行为策略 | 保留 → 通用维度 | — |
| 校验回路 | 保留 → 通用维度 | — |
| 推理模板 | 保留 → 通用维度 | — |
| 反思提示词 | 保留 → 个人维度 | — |
| 约束松弛度 | **移除 → 第二轴（代码参数）** | 纯代码级参数，不注入 prompt。ADR-0001 已确认此定位 |
| — | **🆕 工具可用性** → 通用维度 | 工具用不了是当前 Agent 生态最痛的配置盲区 |
| — | **🆕 技能触发规则** → 通用维度 | 什么时候调什么技能、多技能冲突谁先触发 |

**替代方案**：
- **保持 8 Section 不变**：优点是和蓝图一致。缺点是约束松弛度不适合作 Section（蓝图自己都说了它"不注入 prompt"），且缺少工具和技能两个实际痛点维度。
- **扩展到 12+ 维度**：优点是最完整。缺点是分类过细会增加认知负担，且有些候选维度（如"成本预算偏好"、"模型选择偏好"）更适合作为某个维度的子项而非一级维度。

**为什么不选替代方案**：8 Section 留下了两个覆盖盲区——工具和技能。ECC 的 251 个技能 + 14 个 MCP 配置的规模证明这两个域不是"可选的附加项"，而是一级配置维度。9 个维度在完整性和简洁性之间达到了当前最优平衡。

### D2: 采用五轴而非单轴分类

**决策**：Agent 行为配置不是一维的标签分类，而是五轴的交叉定位。每项配置需要同时回答：控制什么（内容）、怎么生效（实现）、怎么检查（验证）、怎么描述（元属性）、在哪个节点注入（注入点）。

**核心理由**：蓝图"配置 = 文本注入 + 验证规则"的配对模型已经暗示了"内容"和"验证"是两个独立轴。五轴框架是这个洞察的自然扩展。

**替代方案**：
- **单轴分类**（只按内容维度分）：最直观，但不能解释为什么同一个内容维度（如行为策略）可以有不同验证方式（规则匹配 vs LLM 判定）。
- **三轴分类**（内容 + 实现 + 注入点）：蓝图〇章的 8 步流水线 + 配置注入管道的自然映射。但漏了验证轴（蓝图和审查报告都反复强调验证的重要性）和元属性轴（优先级标签是 ADR-0001 中定义的关键机制）。

### D3: 约束松弛度从"配置维度"降为"配置元参数"

**决策**：约束松弛度（strict/flexible/creative）不是 Agent 的行为内容，而是控制"验证规则有多严"的全局参数。放在第二轴（实现维度 → 代码参数）。

**依据**：ADR-0001 已经明确"约束松弛度——纯代码级参数，不注入 prompt，在 [5][6a] Hook 点消费"。蓝图〇.B.7 关键决策表也有"约束松弛度不注入 prompt——纯代码级参数，LLM 不需要感知"。

**连锁影响**：
- 分类学第一轴从 8 个内容维度变为 9 个（去了约束松弛度，加了工具可用性和技能触发规则，净 +1）
- 约束松弛度仍然存在，但分类位置变了——在第四轴（元属性）和第五轴（注入点 POST_THINK / PRE_TOOL）中描述
- "穿搭"（跨维度自由组合）不再包含约束松弛度——它是全局控制，不应该和用户画像/交互风格一样被"穿搭"

### D4: ECC 定位为一级参考源和验证集

**决策**：ECC（Everything Claude Code，affaan-m/ECC，197K+ stars）作为分类学的一级参考源。分类学的每个维度都应该能在 ECC 中找到对应的规则/技能/Agent。Phase 4 用 ECC 的 34 条规则做覆盖验证。

**依据**：
- ECC 是目前最大的 Agent 配置分享生态，代表了社区经过验证的"什么是重要的"
- 如果分类学的某个维度在 ECC 中找不到任何对应配置，该维度可能不存在（或者是 ECC 的覆盖盲区）
- 如果 ECC 中某个高频配置在分类学中找不到对应维度，分类学有覆盖盲区

**边界**：分类学不是 ECC 的文档。分类学做 ECC 没做的事——方法论、跨框架抽象、验证轴、注入点映射。

### D5: 产出物为文档，非软件

**决策**：分类学的产出物是结构化文档（Markdown），不是代码或平台。存放于 `docs/taxonomy/`。

**依据**：蓝图讨论归档的核心结论是"最简单的工具往往就是对的工具"。分类学本质上是"CLAUDE.md 的分类学"——如果 CLAUDE.md 是文本，分类学也应该是文本。

**放弃的产出形态**：
- 交互式网页（增加维护成本，和"知识成果"定位不符）
- CLI 工具（过度工程化，和蓝图推翻结论矛盾）
- 数据库/后端（回到"做平台"的误区）

## 后果

### 积极的
- 蓝图已确认的设计资产（配置=文本+验证、四层注入、8 Section 中 7 个）被继承而非丢弃
- 工具和技能两个实际痛点被纳入一级分类维度
- ECC 的存在让分类学有了最佳的验证集和校准源
- 五轴框架让"配置失效"的诊断有结构化的路径可循
- 文档形式的产出物与"简单工具"原则一致

### 消极的
- 9 维度 + 五轴比 8 Section 更复杂——学习成本增加了
- 跨框架抽象（Claude Code / Codex / Gemini CLI / Cursor）可能导致每个框架的实际用户觉得"不够具体"
- ECC 作为主要验证集可能导致分类学向 CC 生态倾斜，对非 CC 用户的覆盖不足

### 风险
- **维度膨胀风险**：9 个维度可能在 Phase 2 展开时发现需要拆分或合并。缓解：分类学框架标记为 v0.1，Phase 2 逐项展开中允许调整。
- **ECC 覆盖偏差**：如果 ECC 主要面向编程场景，分类学可能低估非编程场景（写作、学习、规划）的配置需求。缓解：Phase 4 验证时明确标注 ECC 的场景覆盖范围。
- **被诚实推翻**：分类学写完可能和蓝图一样被判定为过度设计。缓解：每个阶段产出后都检查——"如果只用 CLAUDE.md，这套分类学真的能帮人写出更好的 CLAUDE.md 吗？"
