> 版本：v0.1
> 日期：2026-06-11
> 状态：总纲 — 施工中

---

## 〇、为什么需要这个

### 当前局面

每个严肃使用 AI Agent 的人，最终都会做同一件事——写一段文本规则，塞进 system prompt，告诉 Agent "你应该这样做事"。

这段文本有各种名字：CLAUDE.md、AGENTS.md、.cursorrules、System Prompt、角色卡、World Info——名字不同，做的是一件事：**定义 Agent 的行为边界和风格**。

问题是：**没有一套共同语言来讨论"这段文本应该覆盖哪些维度、怎么写才有效"。**

每个人都在从零摸索。K 神的四原则是一个人经验总结。ECC 的 251 个技能是社区试错沉淀。酒馆的角色书是角色扮演领域的工程积累。但没人把它们放在一起，回答一个更基础的问题：

> "一个 Agent 的行为配置，到底要配什么？"

### 本分类学做什么

1. **定义 Agent 行为配置的完整维度空间**—— 9 个内容维度，五轴框架，不遗漏关键行为域
2. **为每个维度提供填写方法论**—— 有效写法 vs 无效写法，场景模板，真实样例
3. **告诉你怎么验证配置真的生效了**—— 对每种验证方式给出选择指南

### 本分类学不做什么

- 不给你现成的配置（那是 ECC 做的事）
- 不建平台或系统（这是知识成果，不是软件）
- 不假设特定 Agent 框架（跨 Claude Code / Codex / Gemini CLI / Cursor）
- 不为某一个人设计——技能、模板、出厂默认面向所有用户，不把个人偏好当通用设计

### 第一轴：内容维度

**配置控制 Agent 的什么行为？**

| # | 维度 | 类型 | 回答的问题 |
|---|------|------|-----------|
| 1 | 行为策略 | 通用 | "Agent 绝对不能做什么？必须做什么？" |
| 2 | 校验回路 | 通用 | "Agent 怎么自查自己的回答是否正确？" |
| 3 | 推理模板 | 通用 | "Agent 的回答应该按什么结构组织？" |
| 4 | 工具可用性 | 通用 | "Agent 能用哪些工具？每种工具怎么用？" |
| 5 | 技能触发规则 | 通用 | "什么情况下触发哪个技能/子 Agent？" |
| 6 | 用户画像 | 个人 | "Agent 在为谁服务？这个人的特征是什么？" |
| 7 | 交互风格 | 个人 | "Agent 怎么说话？语气、句式、节奏、幽默感？" |
| 8 | 思维方式 | 个人 | "Agent 用什么原则思考？决策倾向是什么？" |
| 9 | 反思提示词 | 个人 | "Agent 回答后应该自查什么？" |

通用维度回答"不管 Agent 为谁服务，这些规则都要有"。
个人维度回答"这个用户想让 Agent 以什么风格服务"。

### 第二轴：实现维度

**配置怎么让 Agent 感知到？**

| 实现方式 | 说明 | 例子 |
|---------|------|------|
| **文本注入** | 配置作为自然语言文本注入 system prompt 或 user message | "回复简洁，短句，动词先行" |
| **代码参数** | 配置作为非文本参数，在 Hook 点消费，LLM 不直接感知 | `constraint-looseness: strict` |

大多数配置是文本注入。约束松弛度是个例外——它是控制 Hook 判定强度的纯代码参数，LLM 不需要知道。

详见 ADR-0001（四层注入 vs 8 Section 映射）。

### 第三轴：验证维度

**怎么检查配置真的生效了？**

| 验证方式 | 速度 | 成本 | 适用场景 | 例子 |
|---------|------|------|---------|------|
| **规则匹配** | 毫秒级 | 零 | 确定性规则 | "是否包含 rm -rf"→正则匹配 |
| **LLM 判定** | 秒级 | 调 flash | 语义模糊规则 | "是否多角度分析"→flash 判断 |
| **格式校验** | 毫秒级 | 零 | 结构化输出 | JSON Schema / 模板字段检查 |
| **行为拦截** | 毫秒级 | 零 | 工具调用前 | PreToolUse Hook → deny + reason |
| **ECC 诊断** | 按需 | 零 | 配置失效排查 | 12 层 Agent 栈诊断 |
| **无验证** | — | — | 纯建议性配置 | "建议用中文回复" |

关键原则：**不信任 LLM 靠自觉。关键行为用验证兜底。**

### 第四轴：元属性

**配置项自身的描述性属性。**

| 元属性 | 说明 | 可选值 |
|--------|------|--------|
| **优先级标签** | 冲突时谁说了算 | `[覆盖]` / `[补充]` / `[建议]` |
| **注入层级** | 文本注入到 system prompt 的哪一层 | L0 宪法 / L1 骨架 / L2 人格 / L3 上下文 |
| **出厂默认** | 用户未设置时的 fallback 值 | 每维度一个默认值 |
| **技能清单** | 可用技能/子 Agent 的元数据 | 名称、触发条件、适用场景、来源 |

### 第五轴：注入点映射

**配置在 Agent 主循环的哪个节点生效？**

```
用户输入 → [BUILD] → [PRE_THINK] → [LLM 推理] → [POST_THINK] → [PRE_TOOL] → [工具执行] → [POST_TOOL] → [回复输出] → [ON_STOP]
              │            │              │            │            │                       │              │
              │            │              │            │            │                       │              │
           文本注入     检索注入      流式扫描    回复后判定   工具前拦截              工具后验证      会话落盘
          (system      (记忆简报)    (实时验证)   (格式/安全)  (门禁规则)             (一致性)       (JSONL)
           prompt)
```

每个配置项必须明确：它在哪个节点注入，在哪个节点验证。

## 二、九大内容维度概述

### 通用维度

#### 1. 行为策略
Agent 的"能做什么/不能做什么"清单。这是配置系统中最硬的部分——违反行为策略的后果不是"建议改进"，而是 BLOCK。

- **安全规则**：`rm -rf` → BLOCK，`git push -f` → CONFIRM
- **操作纪律**：不跳过审查、不凭空扩写、不猜
- **边界声明**：不做什么类型的任务、不越过的权限边界

#### 2. 校验回路
Agent 回答前的自查机制。不是"用户告诉 Agent 怎么做"，而是"Agent 自己检查自己的回答"。

- **V1 参数校验**：工具调用的参数类型/范围/必填字段
- **V2 语义合理性**：方向性质疑——Agent 有权对用户假设提出独立挑战
- **V3 一致性**：工具结果和调用意图是否一致
- **V4 结果回验**：关键事实是否可信

#### 3. 推理模板
Agent 的回答结构约束。"怎么组织输出"和"输出什么内容"同样重要。

- **结构模板**：先结论后依据 / 先分析再方案 / 先拆问题再回答
- **格式模板**：方案对比表 / 风险评估矩阵 / 分步施工计划
- **深度模板**：多角度分析 / 反方观点 / 边界条件

#### 4. 工具可用性
Agent 能调用什么工具，每种工具的权限、触发条件、使用纪律。

- **工具清单**：已安装的 MCP Server、插件、脚本
- **权限矩阵**：allow / deny / ask 三元
- **使用纪律**：什么情况下禁止调工具、调错怎么办、回退策略

#### 5. 技能触发规则
什么时候触发哪个技能/子 Agent。技能本质是"临时覆盖当前配置的一套配置"。

- **触发条件**：关键词匹配 / 语义匹配 / 强制手动
- **优先级**：多技能同时命中时谁先触发
- **恢复机制**：技能执行完后回到主配置还是覆盖主配置

### 个人维度

#### 6. 用户画像
Agent 为谁服务。这决定了 Agent 对用户背景的假设，影响解释的深度和推荐的方向。

- **身份**：角色（开发者/产品经理/学生）、专业领域、经验水平
- **背景**：技术栈偏好、工作环境、语言
- **当前状态**：正在做什么项目、什么阶段

#### 7. 交互风格
Agent 的"说话方式"。这是用户感知最强的一层——风格不对，内容全对也不满意。

- **语气**：直接/委婉、正式/随意、热情/冷静
- **长度**：简短/详细、先结论后展开/先过程后结论
- **语言**：中文/英文/混合、术语密度
- **节奏**：一次说完/分步展开、先问再答/直接答

#### 8. 思维方式
Agent 怎么处理信息、怎么做决定。这是"元认知"层——控制 Agent 的思考过程。

- **原则**：先判断再实现、先审查再行动、不确定就停下来问
- **启发式**：优先最小改动、默认不扩写、不猜不装懂
- **决策倾向**：保守/激进、务实/探索、跟随用户/主动质疑

#### 9. 反思提示词
Agent 回答后对自己的回答做最后一轮检查。这是"回答后的思维链"——不同于校验回路（代码级判定），反思提示词是 LLM 对自己的语义层检查。

- **安全检查**："我的回答有没有建议危险操作？"
- **完整性检查**："我有没有遗漏用户问题中的某一部分？"
- **风格检查**："我的回答是否符合交互风格中定义的语气和长度？"

## 三、与已有生态的关系

### ECC（Everything Claude Code）

**ECC 是什么**：affaan-m/ECC，197K+ stars，251 技能 + 63 子 Agent + 34 规则 + 79 命令 + 8 Hook。目前最成熟的 CC 配置分享生态。

**和本分类学的关系**：互补，不竞争。

| | ECC | 本分类学 |
|---|-----|---------|
| 给你什么 | 可直接安装的配置包（技能/Agent/规则/Hook） | 理解配置维度的框架 |
| 回答什么问题 | "用什么？" | "为什么这个有效？应该怎么设计自己的？" |
| 用户是谁 | 想要快速上手的 CC 用户 | 想要理解原理、自定义配置的用户 |
| 跨框架吗 | v2.0 开始跨，但主体仍是 CC | 跨框架（Claude Code / Codex / Gemini CLI / Cursor） |

ECC 是本分类学的**最佳验证集**。分类学的每个维度都应该能在 ECC 中找到对应的规则/技能。如果分类学找不到 ECC 中某个高频使用的配置对应的维度，说明分类学有覆盖盲区。

### 酒馆 / SillyTavern

**提供什么**：角色扮演领域的 prompt 构建管线——多层有序 prompt（system → 角色 → 动态 → 对话历史）、World Info 条件触发注入、宏替换系统（`{{user}}` `{{char}}`）、层级分离原则（规则/角色/场景/示例）。

**对本分类学的贡献**：
- "多层有序 prompt" → 注入层级（L0-L3）的参考范式
- "World Info 条件触发" → 技能触发规则的参考范式
- "宏替换系统" → 变量替换机制
- "层级分离原则" → 行为策略和角色描述的分离

### CC 马具系统（用户自建）

**提供什么**：生产环境中的 Hook 硬拦截机制——PreToolUse（Edit/Write/Grep/Bash 前检查）、PostToolUse（Bash 后检查）、Stop（会话结束提醒）。

**对本分类学的贡献**：
- 验证轴的"行为拦截"类的基础设施
- 证明 Hook 驱动的配置验证在生产中可用
- PreToolUse `deny + reason → LLM 自修正` 是经过验证的闭环模式

### K 神 CLAUDE.md 四原则

**提供什么**：一个人提炼出的元规则范式——先判断再实现、先审查再行动、不确定就明确说、优先最小改动。

**对本分类学的贡献**：
- "思维方式"维度的经典样例
- 证明：一个人可以写出覆盖 80% 场景的元规则
- 也证明：四原则不够——缺工具纪律、校验回路、交互风格等维度

## 四、如何使用本分类学

### 方式一：自检

对照 9 个内容维度，检查你当前的 CLAUDE.md 覆盖了哪些、漏了哪些。

**自检问卷**（完整版见 `templates/questionnaire.md`）：

1. 你明确告诉 Agent 不能做什么了吗？ → 行为策略
2. Agent 回答前会自查吗？ → 校验回路
3. 你规定了 Agent 的回答格式吗？ → 推理模板
4. 你的工具调用有权限控制和纪律吗？ → 工具可用性
5. 你的技能触发条件明确吗？ → 技能触发规则
6. Agent 知道你是什么样的人吗？ → 用户画像
7. Agent 用你喜欢的语气回复吗？ → 交互风格
8. Agent 的思考方式和你一致吗？ → 思维方式
9. Agent 回答后会自我检查吗？ → 反思提示词

### 方式二：填空

找到你想提升的维度，直接使用该维度的场景模板。每个维度文档的"场景模板"节提供了可直接填空使用的模板。

### 方式三：诊断

当 Agent 行为不符合预期时，用五轴框架定位问题：

1. **内容轴** — 是不是根本没配这个维度？
2. **实现轴** — 配了但注入方式不对？
3. **验证轴** — 配了但没有检查是否生效？
4. **元属性轴** — 配置被更高优先级的规则覆盖了？
5. **注入点轴** — 注入的节点不对？

### 方式四：从会话中归纳（session-to-memory → 分类学）

这是**从观察到配置的完整闭环**。不需要自建 Agent 平台——日常使用 CC 就是在运行实验：

```
使用 Agent → 产生会话 → session-to-memory → 11 类记忆 → 映射到 9 维度 → 填写配置模板 → 更新 CLAUDE.md
                                                   ↑
                                             分类学在这里
                                        (11类→9维 映射 + 填写方法论)
```

**11 类记忆 → 9 维度的映射规则**：

| session-to-memory 分类 | 映射到分类学维度 | 转化建议 |
|------------------------|-----------------|---------|
| 错误 — 被纠正的行为 | 行为策略 | 每次出错 → 问：缺少什么规则可以防止这个错误？ |
| 决策 — 选 A 不选 B | 思维方式 / 推理模板 | 每次决策 → 问：这个偏好应该固化为思维原则吗？ |
| 偏好 — 用户的风格/喜好 | 用户画像 / 交互风格 | 每次偏好表达 → 更新对应维度 |
| 模式 — 重复出现的规律 | 校验回路 | 连续 ≥3 次出现 → 需要自动化验证 |
| 排除 — 明确不做的方案 | 行为策略 / 出厂默认 | "不用 React" → 写入用户画像或行为策略边界 |
| 不变量 — 确认后不变的事实 | 用户画像 / 出厂默认 | 固化为配置的默认值 |
| 讨论 — 多次往返 + 结论 | 思维方式 / 推理模板 | 结论 → 提炼为思维原则或输出模板 |
| 问题 — 发现的待修项 | 校验回路 / 工具可用性 | 待修 → 是否需要新的检查规则？ |
| 知识 — 外部查到的信息 | 用户画像 | 对 Agent 行为有影响的 → 写入画像或默认值 |
| 配置 — settings/hooks 修改 | 工具可用性 / 技能触发规则 | 直接映射 |
| 使用 — CC 操作踩坑 | 工具可用性 / 行为策略 | "不要用这个命令" → 写入工具纪律 |

当你积累了足够的 session-to-memory 产出后，定期翻阅并按此映射表更新 CLAUDE.md。

## 五、与 MioForever 蓝图的关系

本分类学继承 MioForever 蓝图的设计资产，但重新定义了范围和目标：

| 蓝图中的设计 | 分类学中的位置 | 变化 |
|------------|--------------|------|
| 8 Section | 第一轴的 7 个维度（用户画像/交互风格/思维方式/行为策略/校验回路/推理模板/反思提示词） | 去掉了约束松弛度（移至第二轴），新增了 2 个通用维度 |
| 配置 = 文本 + 验证 | 第一轴（文本）+ 第三轴（验证）的配对关系 | 不变，分类学完整化了验证方式的选择逻辑 |
| 四层注入 | 第四轴（注入层级）+ 第五轴（注入点） | 继承 ADR-0001 双模型分层方案 |
| 穿搭/Mix & Match | 九维度自由组合 | 去掉了"Section 粒度"的约束，改为"任何维度都可以自由配置" |
| Pack Loader / .mio-pack | 不在范围内 | 被 CLAUDE.md 文本文件替代 |
| 配置分享生态 | 不在范围内 | ECC 已经做了这件事 |
| 记忆网 / 检索精选 | 不在范围内 | 属于记忆系统，不属行为配置 |
| 多 Agent 框架 | 不在范围内 | 单独的技术决策 |

### 继承的原则

- **研究优先**：先找已有方案，再决定自研 → ECC 是最佳参考
- **配置优先于代码**：大多数配置就是一段文本 → 分类学以文本注入为主
