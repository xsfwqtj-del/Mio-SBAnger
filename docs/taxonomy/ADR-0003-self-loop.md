# ADR-0003: 项目自循环完善机制设计

**日期**: 2026-06-11
**状态**: proposed
**决策者**: 烬汐

## 上下文

分类学（Agent 行为配置的 9 维度框架）建设完成后，面临一个后续问题：项目怎么持续变好？`mio-jinhua`（进化）技能最初是一份静态检查清单——逐项打勾，输出报告。但检查不是改进，靠人读了报告再手动修，循环断在"发现"和"行动"之间。

需要的不是一个检查工具，而是一个**让项目能自己持续完善的自循环机制**。

## 决策

### D1: 采用望闻问切 → 搜写装记验纠进循环

**决策**：进化不是一次性检查，是持续运转的闭环。结构为：

```
望(观察现状) → 闻(找异常异味) → 问(总结+提方向+用户确认) → 立意
  → 切(分析意图+拟进化路线) → 搜(找资料验证路线)
  → 写(出方案) → 装(实施) → 记(快照) → 验(验证) → 纠(纠错)
  → 进(经验沉淀→下次循环读到更准的信号) → 回到望
```

**每步职责**:

| 步 | 做什么 | 为什么需要 |
|----|--------|-----------|
| 望 | 不带预设观察项目现状 | 避免用模糊输入框死方向 |
| 闻 | 主动嗅探死引用/过期信息/冲突/空壳/腐烂 | 社区空白——AI 领域有"怎么改"没有"怎么发现该改" |
| 问 | 基于望闻信号总结现状、提 2-3 个方向，用户确认 | 立意不靠猜，不靠用户的模糊输入 |
| 立意 | 用户确认后的精确目标（不是独立步骤，是问的输出物） | 后续每一步都对它回验：还在范围内吗？ |
| 切 | 分析意图→拆解→扩搜索关键词→生成搜索议程 | 把"推测你不知道但该知道"系统化 |
| 搜 | 多源并发搜索+来源甄别+发散+对抗验证 | 搜不到不是失败，搜了没改才是 |
| 写 | 输出进化方案：怎么改、为什么、预期效果 | 搜的材料不整合就没有可执行性 |
| 装 | 实施。调 writer/tools/discovery/直接 Edit | 分类学技能是进化的工具，不是进化的本体 |
| 记 | 改前快照，为回退留底 | 社区空白——工程实用但学术不研究 |
| 验 | 确认改完生效 | 参考 TIDE 三维指标(效率/适应/记忆) |
| 纠 | 不生效→回退→重搜→重改 | 没有退路的改进是赌博 |
| 进 | 经验沉淀→session-to-memory→下次望信号更准 | 闭环最后一公里——没它循环不闭合 |

### D2: 立意置于问之后，而非循环输入

**决策**：立意不是循环的前提，是望闻问充分观察之后的产物。

**核心理由**：进化循环要解决的问题之一是"用户输入不够精确"。用户在说"进化"时，不知道哪个方向最值得进——把不精确的输入当立意，锚点本身就是偏的。

**问的机制**：
1. 基于望+闻的发现总结现状
2. 提 2-3 个方向（每个方向有依据——基于观察发现的问题）
3. 用户选一个，选中的方向 = 立意

立意不是猜的、不是自说自话的、也不是盲目接受用户原始输入的。是观察 + 确认的。

### D3: 记(快照)和进(沉淀)是工程价值，不是学术价值

**决策**：保留记和进，不因为学术领域不研究就砍掉。

**记**——改前记录旧状态。太简单（一行 JSON / 几行 Markdown），学术不研究。但实际使用中，改坏了退不回去是最常见的挫败来源之一。

**进**——经验沉淀进入下次循环。GPT Deep Research 通过 RL 训练实现反馈（一次几十万美金），TIDE 有记忆索引但只是衡量指标。我们的场景不需要 RL——每次会话的教训写下来，下次望直接读到就够了。复杂度远低于学术方案，效果在个人使用场景等价。

### D4: 分类学技能是进化的工具，不是进化的本体

**决策**：进化是自循环机制的承载者。分类学的 writer/tools/discovery/session-finder/session-to-memory 是它调用的工具。

**依据**：之前混淆了进化（项目怎么持续变好）和分类学技能链（装→记→验→纠）。分离后：进化负责诊断+研究+调度，分类学技能负责执行特定类型的改动。

## 替代方案

### 替代方案 1: GPT Deep Research 模式（纯搜索+合成）

只做搜+荐，不含装验证纠进。用户读了推荐后自己动手。

**淘汰理由**：断在"行动"前。分类学的 3 处死引用是发现了，但如果只是推荐"你应该修"，下次会话谁还记得？必须闭环到自己能动手改。

### 替代方案 2: 中医 AI 四诊模式（只诊断不行动）

望闻问切 → 诊断报告 → 用户自己处理。

**淘汰理由**：中医场景中诊断和治疗是分开的（诊断在 AI，治疗在医生）。但项目自完善场景中，诊断+治疗+康复+记忆应该在一起——因为"动手改"是这个循环最重要的价值。

### 替代方案 3: r-web-plan 模式（精密七步方案流程）

r-web-plan 已经是成熟的多步骤方案写作流程。为什么不直接用它？

**保留理由但不采用**：r-web-plan 适合 2+ 决策点的大型方案。进化循环大多数场景是轻量级改进（修一个死引用、更新一个过期计数），不需要完整方案。进化循环中有"写"步骤——需要时可以调用 r-web-plan，但日常轻量改进不走完整方案流程。

**互补关系**：进化的"写"步骤 → 判断改动复杂度 → 轻量（直接改）→ 重量（调用 r-web-plan 写完整方案）。

## 后果

### 积极的
- 项目不再依赖人发现了问题再去手动修——循环发现→自己研究→自己改
- 立意机制解决了"模糊输入 vs 精确行动"的矛盾
- 记的快照让每次改动都有退路
- 进的沉淀让每次循环都比上一次更聪明
- 分层清晰：进化(调度) / 分类学技能(工具) / r-web-plan(重型方案)

### 消极的
- 12 步循环看起来复杂，但大部分步骤在轻量场景下是 O(1) 操作
- 需要验证循环是否真的比"发现了问题记下来下次再说"更好——尚未 A/B 测试

### 风险
- **循环过长**：轻量改一个路径引用的次数 vs 走完完整十二步 → 需要明确"哪些步骤可以跳"
- **立意漂移**：用户确认的方向在搜和写的过程中可能因新发现而变化 → 需要回验机制
- **进堆积**：每次循环都存经验，memory 文件会膨胀 → 需要压缩和淘汰策略
