# 错误对照自查表

> Agent 出现了什么问题 → 查表 → 对应的配置维度 + 修复方案。
> 不是事后日志，是事前诊断工具。

---

## 症状: Agent 盲目动手，不审查方案

**典型表现**: 用户说"帮我写个 X"，Agent 直接开始写代码，不查已有方案、不判断可行性。

**根因**: 思维方式维度缺失"先审查再行动"原则。

**修复**:
```markdown
思维方式:
  - 原则: 先判断方向，再进入代码。如果发现"一头扎进代码"，停下来。
  - 原则: 先找已有方案，再决定自研。不重新发明轮子。
```

**验证**: 下次提出一个"看似简单但已有成熟方案"的任务，看 Agent 是否先搜索再动手。

---

## 症状: Agent 在不确定时猜答案

**典型表现**: 用户没说清楚技术栈，Agent 假设用 React；用户没说文件路径，Agent 自己猜了一个。

**根因**: 思维方式维度缺少"不确定就停下来问"。

**修复**:
```markdown
思维方式:
  - 不确定就明确说明，不要猜。列出需要的假设，请用户确认。
```

**验证**: 给一个信息不完整的任务，看 Agent 是直接开始还是先确认。

---

## 症状: Agent 擅自动了不该动的东西

**典型表现**: 删除了用户没让删的文件、改了用户没让改的配置、push 到了不该 push 的分支。

**根因**: 行为策略维度缺少安全红线 + PRE_TOOL Hook 未配置。

**修复**:
```markdown
行为策略:
  安全红线:
    - 绝不执行 rm -rf 或类似递归删除命令
    - 绝不执行 git push -f 到 main/master
    - 绝不修改 .git 目录中的文件
    - 绝不修改 settings.json 而不先验证 JSON 有效性
```
+ 配置 PRE_TOOL Hook:
```json
{"matcher": "Bash", "hooks": [{"command": "python guard.py"}]}
```

**验证**: 让 Agent 执行一次可能危险的操作，看是否被拦截。

---

## 症状: Agent 犯了同样的错误 ≥2 次

**典型表现**: 上次 Agent 删了不该删的文件，你纠正了。下次它又犯了同样的错误。

**根因**: 校验回路维度缺失——纠正后没有把教训固化为配置规则。

**修复**: 每次纠正后立即映射到分类学维度 → 更新 CLAUDE.md → 如果是确定性规则，加 Hook 拦截。

**验证**: 下次类似场景，看 Agent 是否被规则自动拦截。

---

## 症状: Agent 的回答结构混乱

**典型表现**: 先分析一大堆，最后才说结论。或者方案和风险混在一起。

**根因**: 推理模板维度缺失或写得模糊。

**修复**:
```markdown
推理模板:
  方案类: 先结论（一句），后理由（3-5条），再风险（1-2条）
  诊断类: 根因 → 证据 → 修复 → 预防
```

**验证**: 提一个方案类问题，看回复结构是否符合模板。

---

## 症状: Agent 的回答太长 / 太短 / 语气不对

**典型表现**: 回复用"你好！很高兴为你服务"开头；只说"可以"不解释为什么；用了太多 emoji。

**根因**: 交互风格维度缺失或写得太模糊。

**修复**: 不写"简洁专业"，写具体的行为指令：
```markdown
交互风格:
  不: 不问候、不客套、不 emoji、不说"希望这对你有帮助"
  语气: 直接。用"做X"而非"也许可以考虑做X"
```

**验证**: 日常对话中观察回复风格是否符合预期。

---

## 症状: Agent 在回答后从不反省

**典型表现**: 给了方案但没检查是否有遗漏；推荐了危险操作但没警告。

**根因**: 反思提示词维度缺失。

**修复**:
```markdown
反思提示词:
  回答前检查:
    1. 我的回答有没有建议危险操作？
    2. 用户的问题我全回答了吗？有没有遗漏？
    3. 我有没有把推测包装成事实？
```

**验证**: 提一个包含多个子问题的问题，看 Agent 是否全部覆盖。

---

## 症状: 装了一堆 MCP 但 Agent 不调

**典型表现**: 装了 `mcp__github`，但 Agent 用 Bash 直接调 gh CLI；装了 `mcp__context7`，但 Agent 用 WebSearch。

**根因**: 工具可用性维度缺失——工具清单只写了名字，没写"什么时候用、什么时候不用"。

**修复**:
```yaml
工具可用性:
  mcp__github:
    适用: "创建 PR" "查看 Issue" "推送文件"
    优先: 有 mcp__github 时优先用它，不用 Bash gh CLI
  mcp__context7:
    适用: "这个 API 怎么用" "最新版本参数"
    纪律: 每问最多查 3 次
```

**验证**: 提一个明显应该用 MCP 的任务，看 Agent 是否调了正确的工具。

---

## 症状: 技能该触发却不触发

**典型表现**: 用户说"debug"，但 Agent 没有进入调试模式；用户说"新功能"，但没有先做方案规划。

**根因**: 技能触发规则维度缺失——技能有描述，但没有明确的触发条件 + 优先级。

**修复**:
```yaml
技能触发规则:
  debugging:
    触发: "bug" "报错" "失败" "修一下" → 自动触发
    优先级: 高（覆盖默认行为）
  brainstorming:
    触发: "新功能" "设计方案" → 自动触发
    不触发: bug 修复、纯代码审查
```

**验证**: 用触发关键词发起对话，看技能是否自动激活。

---

## 症状: Agent 调用工具陷入死循环

**典型表现**: 调工具 → 不满意 → 重调 → 还不满意 → 再重调 → ... 无限循环。

**根因**: 校验回路维度缺少回环终止条件。

**修复**:
```yaml
校验回路:
  工具回环: max=10（连续同一工具调用超过 10 次 → 强制停止）
  拦截回环: max=3（违规→纠正→重调 最多 3 次）
```

**验证**: 给一个可能需要多轮工具调用的任务，观察是否在合理轮次内结束。

---

## 症状: 配置改坏了，Agent 行为退化了

**典型表现**: 加了一条新规则后，Agent 变得过于保守 / 频繁误解规则 / 在某些场景下行为异常。

**根因**: 配置进化机制缺失——改了配置但没有审计记录，不确定是哪个改动导致的问题。

**修复**: 每次改 CLAUDE.md 记录：
```markdown
## YYYY-MM-DD 更新
- 触发: {原因}
- 维度: {哪个维度}
- 旧: {改之前}
- 新: {改之后}
- 预期: {预期效果}
```

**验证**: 回滚到旧版本看问题是否消失，确认是配置改动导致。

---

## 症状: Agent 和用户偏好不匹配

**典型表现**: Agent 推荐了 Vue，但用户只用 React；Agent 用了英文回答，但用户偏好中文。

**根因**: 用户画像维度缺失。

**修复**:
```markdown
用户画像:
  角色: {你的角色}
  技术栈: {你的技术栈偏好}
  语言: 中文
  当前项目: {项目简介 + 当前阶段}
```

**验证**: 提一个涉及技术选型的问题，看 Agent 是否优先推荐你的偏好。
