# 交互风格

> 维度类型：个人 | 实现方式：文本注入 | 验证方式：LLM 判定 + 规则匹配

---

## 一、定义

**控制 Agent 怎么说话——语气、句式、节奏、语言、长度。**

这是用户感知最强的一层配置。内容全对但风格不对 → 用户不满意。风格对但内容有瑕疵 → 用户可能忽略。

### 控制什么

- **语气**：直接/委婉、正式/随意、热情/冷静
- **长度**：简短/详细、先结论后展开/先过程后结论
- **语言**：中文/英文/混合、术语密度
- **节奏**：一次说完/分步展开、先问再答/直接答

### 不控制什么

- 输出的结构组织（那是推理模板的事）
- 是否必须遵守风格（那是约束松弛度的事）

## 三、填写方法论

### 3.1 有效写法

**有效写法**：具体 + 有反例 + 不需要 LLM 自己揣摩。

```
交互风格:
  语气: 直接。不说"也许可以考虑..."，说"用这个方案，因为..."
  长度: 先结论后展开。第一句话就是答案，后面才是理由。
  语言: 中文为主。技术术语保留原文（API、TypeScript、KV cache）。
  节奏: 分步展开。复杂问题先给步骤概览，再逐一深入。
  不: 不要客套话（"你好！很高兴为你服务"），不要模糊措辞（"可能""也许""大概"）。
```

**为什么有效**：每一条都有一个"这样做"和一个"不这样做"的具体指示。LLM 不需要从"直接"这个词推断具体行为——它知道"直接"意味着"不说'也许可以考虑'"。

**无效写法**：

```
# ❌ 差
交互风格: 简洁专业、友好热情
```

"简洁专业"和"友好热情"本身就是矛盾的。而且没有具体指令，LLM 会按照自己的训练偏好在两者之间随机波动。

### 3.2 CC 源码中的交互风格

Claude Code 的 sysprompt 静态段对交互风格有精确的控制：

```
- "Lead with the answer or action, not the reasoning"
- "Keep your text output brief and direct"
- "If you can say it in one sentence, don't use three"
- "最终总结 ≤ 4 行，除非必要不详细说明"
- "最小化 token，避免不必要的开场白、结尾总结和 emoji"
- internal monologue ≤ 25 词，最终回复 ≤ 100 词
```

**CC 的风格是"极简命令式"**。这不是每个用户都喜欢的——有人希望 Agent 多说一点，有人希望 Agent 更热情。这就是交互风格作为个人维度的原因。

### 3.3 语气和内容的优先级

当交互风格和内容要求冲突时：

```
交互风格 vs 行为策略: 行为策略优先。
  例: "简洁"不能成为跳步审查的借口。
  → 行为策略 [覆盖] 交互风格

交互风格 vs 推理模板: 推理模板优先。
  例: "简短"不能跳过模板要求的"风险分析"段。
  → 推理模板 [覆盖] 交互风格

交互风格 vs 用户画像: 用户画像影响交互风格。
  例: "资深开发者" → 可以跳过基础概念解释。
```

## 四、验证方式

### 规则匹配（长度检查）

```python
def check_response_length(response, style_config):
    if style_config.get("max_length") and len(response) > style_config["max_length"]:
        return FLAG, f"回复长度 {len(response)} 超过限制 {style_config['max_length']}"
    return ALLOW
```

### LLM 判定（语气检查）

偶尔用 flash 模型判断一回合复的风格是否偏离：

```
检查: Agent 的回复是否符合定义的交互风格？
  - 语气是否直接？
  - 是否有不必要的客套？
  - 技术术语是否使用正确？
评分: 1-5，<3 → FLAG
```

**注意**：风格检查不要每轮都做——成本高且价值低。每 10 轮抽样一次即可。

## 五、场景模板

### 编程场景（极简风格）

```
交互风格（编程·极简）:

  语气: 直接。不用"也许""可能""或许可以考虑"。
  长度: 先结论后依据。第一句就是答案。
  语言: 中文。技术术语保留原文。
  节奏: 分步。复杂任务先给步骤概览。
  不: 不问候、不总结、不 emoji、不说"希望这对你有帮助"。
```

### 编程场景（教学风格）

```
交互风格（编程·教学）:

  语气: 耐心。解释每一步为什么这样做。
  长度: 详细。不只是给结论，给思维过程。
  语言: 中文。首次出现的技术术语加英文标注。
  节奏: 先问再答。不确定用户水平时先确认理解程度。
  不: 不假设用户已经知道某个概念。
```

### 写作场景

```
交互风格（写作·助力型）:

  语气: 建议性。用"可以考虑""另一种写法是"而非"应该这样写"。
  长度: 精要。改什么说什么，不用说教。
  节奏: 改完再总结。先给出修改，最后一句说改了什么。
  不: 不重写全文（除非要求），不评论用户的写作风格。
```

## 六、ECC 对标

ECC 没有独立的"交互风格"配置——它的交互风格隐含在 Agent 的技能描述中（planner 用 planner 的风格，code-reviewer 用 code-reviewer 的风格）。

这意味着 ECC 的交互风格是分散的、不统一的——同一个用户在同一个会话中可能遇到 planner 的详细风格和 code-reviewer 的简洁风格交替。

**本维度补充的**：统一的交互风格规则，跨技能/子 Agent 生效。

## 七、常见陷阱

| 陷阱 | 为什么错 | 怎么改 |
|------|---------|--------|
| 只有正面指令，没有反例 | "直接"→ LLM 不清楚什么叫"不直接" | 每条指令配一个"不"的例子 |
| 风格和推理模板冲突 | "简洁" vs "风险分析必须≥3 条" | 明确优先级：模板 [覆盖] 风格 |
| 风格太极端 | "永远不要超过 3 句话" → Agent 被迫在复杂问题上偷工减料 | 加例外规则：复杂问题可以超出长度限制 |
| 用文学化描述风格 | "像朋友一样聊天" → LLM 理解不一致 | 用行为化语言："直接给答案""不要客套" |

## 八、真实样例

### 用户的交互风格

来自生产环境 CLAUDE.md：

```
输出习惯:
- 先结论，后依据。
- 需要决策时，直接给推荐项。
- 发现前提缺失、逻辑冲突、风险较高时，先指出问题。
- 任务尽量拆成可验证步骤，再推进。
- 回答里要区分事实、推测和建议。
```

### CC 源码中的交互风格

```
- "Lead with the answer or action, not the reasoning"
- "If you can say it in one sentence, don't use three"
- "Keep your text output brief and direct"
- "最小化 token，避免不必要的开场白、结尾总结和 emoji"
- internal monologue ≤ 25 词，最终回复 ≤ 100 词
```

这两套风格的共同点：**直接、简洁、结构化**。区别在于用户的版本多了"指出问题"和"区分事实/推测"——这是"交互风格"和"思维方式"的交叉区域。
