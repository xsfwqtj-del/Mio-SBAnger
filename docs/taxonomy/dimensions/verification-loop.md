# 校验回路

> 维度类型：通用 | 实现方式：文本注入 + 代码参数 | 验证方式：规则匹配 + LLM 判定 + 行为拦截

---

## 一、定义

**控制 Agent 怎么自查——在回答之前、回答之后、工具调用之前、工具调用之后，用什么规则检查、怎么处理违规。**

校验回路和行为策略配成一对：行为策略说"不能做什么"，校验回路说"怎么检查你真的没做"。

### 控制什么

- 校验的节点——在 Agent 主循环的哪些环节做检查
- 每级校验的检查内容和方法
- 违规后的处置动作（BLOCK / CONFIRM / FLAG / ALLOW）
- 约束松弛度对处置动作的影响

### 不控制什么

- 具体的规则内容（那是行为策略、工具可用性的事）
- Agent 输出后的自我反思（那是反思提示词的事——LLM 层面的自查）

### 校验回路 vs 反思提示词

| | 校验回路 | 反思提示词 |
|---|---------|-----------|
| 谁执行 | 代码/规则引擎 | LLM 自己 |
| 速度 | 毫秒级 | 秒级（调 flash） |
| 确定性 | 确定 | 不确定（LLM 可能忽略） |
| 适用场景 | 可格式化的规则 | 语义模糊的检查 |

---

## 二、注入位置

校验回路不直接注入 system prompt，而是注册到 Hook 点：

| Hook 点 | 校验内容 | 实现方式 |
|---------|---------|---------|
| PRE_TOOL | V1 参数校验 + V2 语义合理性 | 代码参数（正则/规则匹配） |
| POST_TOOL | V3 一致性 + V4 结果回验 | 代码参数 + 可选 flash LLM |
| POST_THINK | 回复安全/格式/行为合规 | 代码参数 |
| ON_STOP | 会话完整性 | 代码参数 |

## 三、填写方法论

### 3.1 四级校验模型 (V1-V4)

```
V1 参数校验       V2 语义合理性      V3 一致性         V4 结果回验
    │                  │                │                │
    ▼                  ▼                ▼                ▼
 类型/范围检查      方向性质疑       意图 vs 结果     关键事实核实
 格式检查            危险操作拦截     是否跑偏         是否需要二次确认
 ────────────────    ────────────     ────────────     ────────────
 本地毫秒级         本地毫秒级        调 flash LLM      调 flash LLM
 100% 确定性         80% 确定性         70% 确定性         60% 确定性
```

### V1 参数校验

检查工具调用的参数是否符合预期格式：

```yaml
V1 参数校验规则:
  Write:
    - file_path 必须是绝对路径
    - content 不为空
    - 文件不在 .git/ 目录下

  Bash:
    - command 不包含 "rm -rf" "DROP" "force push" 等危险模式
    - command 长度 < 10000 字符

  WebFetch:
    - url 必须是 HTTPS（或允许的 HTTP 域名）
```

### V2 语义合理性

方向性质疑——Agent 有权对用户假设提出独立挑战：

```yaml
V2 语义合理性规则:
  - 如果工具调用和对话意图明显不匹配 → CONFIRM
    例: 用户要求"审查代码"，Agent 调了 Write → "确认要修改代码而非审查？"

  - 如果工具调用存在已知风险 → BLOCK
    例: Bash 中使用了未经确认的环境变量 → "此环境变量未确认，先确认值"

  - 如果工具调用暗示 Agent 误解了用户意图 → 注入纠正提示词并打回 LLM
```

**V2 是校验回路中最被低估的一层。** 大多数 Agent 只做了 V1（参数类型检查），不做 V2（方向性质疑）。V2 的本质是：**Agent 不能用户说什么就做什么——它必须独立判断方向是否正确。**

### V3 一致性

工具执行完后，检查结果和调用意图是否一致：

```yaml
V3 一致性检查:
  触发: POST_TOOL
  检查: 工具结果是否符合调用时的意图？

  例:
    - 意图是"读取文件内容"，结果是 "Permission denied"
      → 不是一致性失败，是执行失败。把错误信息传给 LLM 继续。

    - 意图是"查找 .ts 文件"，结果返回了 .js 文件
      → 一致性失败。注入纠正: "返回了 .js 文件而非 .ts 文件，请确认搜索模式。"
```

### V4 结果回验

对于关键事实（文件存在性、API 返回内容、命令执行结果），做二次确认：

```yaml
V4 结果回验:
  触发: POST_TOOL（仅对标记为"关键"的结果触发）
  检查: 工具返回的事实是否可信？

  例:
    - Bash 返回 "tests passed" → 不需要 V4（通过即可信）
    - Bash 返回 "file not found" → V4: 用 Read 工具确认文件真的不存在
    - WebFetch 返回的数据 → V4: 用另一个来源交叉验证
```

### 3.2 约束松弛度对校验的控制

校验的严格程度由约束松弛度控制：

```
             strict         flexible         creative
            ────────        ────────         ────────
V1 参数     强制检查        强制检查          强制检查（不变）
V2 语义     违规→BLOCK      违规→CONFIRM      违规→FLAG
V3 一致性   调 flash LLM    调 flash LLM      不检查
V4 回验     调 flash LLM    关键操作才会       不检查
```

**关键原则**：V1（参数校验）不受松弛度影响——参数校验是安全底线，任何时候都应该做。

### 3.3 回环终止条件

三个回环都必须有硬上限（来自蓝图审查 S4 问题）：

```yaml
回环终止:
  工具回环: Agent 可以多轮调工具 → 但最多 10 轮
    超过 → 强制输出当前结果 + 标注"未完成"

  拦截回环: 违规 → 注入纠正 → 重调 LLM → 最多 3 次
    超过 → 强制返回 + 审计日志

  判定回环: POST_THINK 违规 → 打回 LLM → 最多 3 次
    超过 → CONFIRM 用户（不再自动打回）
```

## 四、验证方式

校验回路本身也需要验证——谁检查检查者？

### 4.1 审计日志

每一条校验动作（ALLOW/BLOCK/CONFIRM/FLAG）记录到日志：

```json
{
  "hook": "PRE_TOOL",
  "tool": "Bash",
  "args": "rm file.txt",
  "check": "V1 参数校验",
  "rule": "dangerous-command-rm",
  "action": "BLOCK",
  "looseness": "strict",
  "timestamp": "2026-06-11T12:00:00Z"
}
```

定期审计：拦截率是否过高（误拦）？是否过低（漏拦）？纠正提示词是否有效（打回后 LLM 是否改进）？

### 4.2 静默审计

部分校验只审计不拦截（尤其是 V3/V4），用于监控校验回路本身是否需要调整。

## 五、场景模板

### 编程场景（strict 级）

```yaml
校验回路（编程 · strict）:

  PRE_TOOL:
    V1 参数:
      - 所有工具: 参数类型/必填字段检查
      - Write: 文件路径检查（不在 .git/、不是系统文件）
      - Bash: 危险命令正则匹配
    V2 语义:
      - Write 被调但用户说的是"读" → CONFIRM
      - 编辑文件数 > 5 → CONFIRM "确认要批量修改这么多文件？"

  POST_TOOL:
    V3 一致性: 对 Bash/WebFetch 结果做一致性检查
    V4 回验: 对"文件不存在""API 返回空"做二次确认

  回环终止:
    工具回环 max=10, 拦截回环 max=3, 判定回环 max=3
```

### 写作场景（flexible 级）

```yaml
校验回路（写作 · flexible）:

  PRE_TOOL:
    V1 参数: Write/Read 的文件路径检查
    V2 语义: 不检查（写作场景下 Agent 的自由度更高）

  POST_TOOL:
    V3 一致性: 不检查
    V4 回验: 只在文件写入失败时做
```

## 六、ECC 对标

| ECC 模块 | 对应本维度 |
|----------|-----------|
| Hooks (PreToolUse/PostToolUse) | PRE_TOOL / POST_TOOL 校验点 |
| AgentShield (102条静态分析规则) | V1 参数校验的自动检查版 |
| Hooks → verify loop (Lint→Test→Build) | V3 一致性检查——执行结果是否和预期一致 |
| Rules → 安全规则 | V2 语义合理性的规则来源 |

ECC 的校验偏向开发和代码质量（lint/build/test 自动化验证回路），而本维度的四级校验模型更通用——覆盖所有工具调用和回复。

## 七、常见陷阱

| 陷阱 | 为什么错 | 怎么改 |
|------|---------|--------|
| 只有 V1，没有 V2 | Agent 参数正确但方向错误 → 放行 | 加 V2 语义合理性——方向性质疑 |
| V3/V4 总是触发 | 每次工具调用都调 flash → 成本爆炸 | V3/V4 只对"关键操作"触发 |
| 回环没有终止条件 | Agent 在"拦截→纠正→再拦截"中无限循环 | 拦截回环 max=3，超出强制返回 |
| 校验太严 → Agent 什么都做不了 | strict 级下所有灰色地带都 BLOCK | 灰色地带动作用 CONFIRM（不是 BLOCK） |
| 校验回路和反思提示词混淆 | 用 LLM 做确定性检查（应该用规则匹配） | 确定性检查 = 校验回路，语义模糊检查 = 反思提示词 |

## 八、真实样例

### 样例 1：生产级 PRE_TOOL 校验

来自 `D:\mio\.claude\settings.json` 的 CC Hook 配置：

```json
{
  "PreToolUse": [
    {
      "matcher": "Edit|Write",
      "hooks": [{"command": "python gitnexus-guard.py"}]
    },
    {
      "matcher": "Bash",
      "hooks": [{"command": "python gitnexus-guard.py"}]
    }
  ],
  "PostToolUse": [
    {
      "matcher": "Bash",
      "hooks": [{"command": "python gitnexus-stale.py"}]
    }
  ]
}
```

**对应到校验回路**：
- `gitnexus-guard.py` = V2 语义合理性（Edit/Write 前提醒 impact 分析）
- `gitnexus-stale.py` = V3 一致性（git 操作后检查索引是否过期）

### 样例 2：状态机自检

来自 UserPromptSubmit Hook 注入的状态检查：

```
回答前：
1. 当前对话在干什么？和上轮状态变了没？
2. 该进 Plan Mode 吗？该建 Task 吗？该调 Skill 吗？
3. 回复末尾标注状态
```

这是 V2 语义合理性的文本注入版——在 Agent 回答前强制它做方向性质疑。
