# 技能触发规则

> 维度类型：通用 | 实现方式：文本注入 + 代码参数 | 验证方式：规则匹配 + ECC 诊断

---

## 一、定义

**控制 Agent 在什么情况下触发哪个技能/子 Agent，多项技能冲突时谁优先，技能执行后怎么恢复。**

技能（Skill）是一套临时的行为配置覆盖——它替换或增强 Agent 在特定场景下的行为。技能触发规则是这个"切换机制"的控制层。

### 控制什么

- 技能清单——有哪些可用技能、每个技能做什么
- 触发条件——什么输入/场景触发哪个技能（关键词？语义？手动？）
- 优先级——多个技能同时命中时谁先触发
- 恢复机制——技能执行完后回到主配置，还是覆盖主配置直到会话结束

### 不控制什么

- 技能内部的实现（那是技能文件本身的事）
- 技能的质量（那是技能开发者的责任）
- 技能间的数据传递（那是 Agent 框架的实现细节）

### 为什么这是一个独立的配置维度

ECC 有 251 个技能。用户自己的技能目录里有十几个技能。一个会话可能命中 3-5 个技能。

没有触发规则 → Agent 要么不触发技能（浪费），要么随机触发（混乱），要么全触发（上下文爆炸）。

技能触发规则是 Agent 的"调度器"配置——和操作系统的进程调度一样重要。

## 二、注入位置

| 注入层 | 注入内容 | 注入方式 |
|--------|---------|---------|
| L1 骨架层 | 技能清单（名称 + 一句话描述 + 触发条件） | BUILD 阶段注入 system prompt |
| L2 人格层 | 技能使用偏好（"优先用技能，不要自己蛮干"） | BUILD 阶段注入 system prompt |
| 代码参数 | 技能注册表（路径 + 触发匹配器 + 优先级） | 技能加载器消费 |

## 三、填写方法论

### 3.1 技能清单的写法

**有效写法**（结构化，LLM 能直接匹配）：

```yaml
# ✅ 好：每个技能有触发条件、适用场景、不适用场景
可用技能:

  sys-debug:
    触发: "遇到 bug" "测试失败" "行为异常"
    做什么: 系统性调试——先定位根因，再出修复方案
    不触发: 代码审查、功能开发、文档修改
    执行后: 回到主配置

  deep-research:
    触发: 用户明确说"深度研究" "全面调研"
    做什么: 多源搜索 → 对抗验证 → 引用报告
    不触发: 简单的"查一下""搜索"请求
    执行后: 回到主配置
    注意: 成本高（~3M tokens/次），不要在简单问题上触发

  brainstorming:
    触发: "新功能" "设计方案" "创意" "怎么做"
    做什么: 探索意图 → 多角度方案 → 筛选推荐
    不触发: bug 修复、代码审查、纯执行任务
    执行后: 覆盖主配置直到会话结束（设计思维保持激活）
```

**无效写法**（只有名字，没有触发条件）：

```
# ❌ 差：LLM 不知道什么时候该用
可用技能：debugging, research, brainstorming, planning, review
```

**原则**：技能清单不是目录，是路由表。LLM 需要知道"当前输入匹配哪个技能"的判断依据。

### 3.2 触发条件的层次

触发条件可以有三层精度：

| 层 | 精度 | 机制 | 例子 |
|----|------|------|------|
| **精确匹配** | 最高 | 关键词/正则 | 输入含"debug" → sys-debug |
| **语义匹配** | 中等 | LLM 判断意图 | 输入是"为什么结果不对" → 虽然不含"debug"，语义上是调试 → sys-debug |
| **手动触发** | 完全精确 | 用户显式调用 | `/deep-research` → 强制触发 |

**推荐策略**：
- 高频技能（每天触发 5+ 次）→ 精确匹配为主，降低判断成本
- 低频高成本技能（deep-research）→ 语义匹配 + 确认（"这需要深度研究，约 3M tokens，继续吗？"）
- 危险技能（安全扫描、系统操作）→ 手动触发为主，不自动匹配

### 3.3 优先级的写法

当多个技能同时匹配时，谁说了算：

```yaml
# ✅ 好：明确的优先级规则
技能优先级:
  1. 用户手动触发 > 自动匹配   # 用户明确说了算
  2. 高成本技能 > 低成本技能   # 先确认要不要花这个钱
  3. 危险技能 > 安全技能       # 先拦再放，不是先放再查
  4. 精确匹配 > 语义匹配       # 确定性高的优先
  5. 先注册 > 后注册           # 同优先级按加载顺序

  当无法自动判断时: 列出候选技能让用户选
```

### 3.4 恢复机制

技能执行完后 Agent 回到什么状态：

```
                    ┌──────────────────────────┐
                    │     主配置（CLAUDE.md）    │
                    │  思维方式 + 行为策略 +     │
                    │  交互风格 + ...           │
                    └──────────┬───────────────┘
                               │
                    技能触发 → │
                               ▼
                    ┌──────────────────────────┐
                    │   技能覆盖层（临时）       │
                    │  部分维度被覆盖/增强       │
                    │  · 推理模板 → 技能力模板    │
                    │  · 行为策略 → 技能纪律      │
                    └──────────┬───────────────┘
                               │
                    技能完成 → │ ← 恢复机制决定这里
                               │
              ┌────────────────┼────────────────┐
              ▼                ▼                ▼
        回到主配置        会话级覆盖        用户手动恢复
       （默认行为）      （brainstorming）   （手动 /reset）
```

```yaml
# ✅ 好：明确每种技能的恢复行为
恢复策略:
  默认: 技能执行完 → 立即回到主配置
  例外:
    brainstorming: 会话级覆盖（设计思维需要持续激活）
    agent-architect: 会话级覆盖（架构决策影响整个会话）
    安全扫描: 立即回到主配置（安全是检查点，不是持续状态）
```

## 四、验证方式

### 4.1 规则匹配：技能触发日志

记录每次技能触发的决策依据：

```python
def log_skill_trigger(user_input, matched_skills, chosen_skill, reason):
    """
    记录：为什么选了技能 A 而不是技能 B？
    用于事后分析触发规则的准确性。
    """
    return {
        "input_intent": classify_intent(user_input),
        "candidates": [s.name for s in matched_skills],
        "chosen": chosen_skill,
        "reason": reason,
        "timestamp": now()
    }
```

**关键指标**：
- 触发准确率 = 用户接受技能结果 / 总触发次数
- 技能冲突率 = 多技能同时匹配 / 总触发次数
- 漏触发率 = 应该触发但没触发 / 应该触发的总次数

### 4.2 ECC 诊断：触发失效排查

当技能"明明有但不触发"时：

- **Layer 1 (System prompt)**：技能清单正确注入到 system prompt 了吗？→ 检查 BUILD 阶段的注入日志
- **Layer 6 (Tool selection)**：LLM 是否识别到了技能触发条件？→ 检查"语义匹配"是否过于模糊
- **Layer 3 (Long-term memory)**：是否有旧的废弃技能仍在 system prompt 中？→ 检查技能清单是否过期

## 五、场景模板

### 编程场景

```yaml
技能触发规则（编程场景）:

  技能清单:
    tdd:
      触发: "新功能" "加一个" "实现" 且项目启用 TDD
      做什么: 先写测试 → 用户确认 → 写实现
      优先级: 高（覆盖默认的"直接写代码"行为）
      恢复: 立即回到主配置

    code-review:
      触发: "审查" "review" "检查这段代码"
      做什么: correctness bugs + simplification cleanups
      优先级: 中
      恢复: 立即回到主配置

    debugging:
      触发: "bug" "报错" "失败" "不对" "修一下"
      做什么: 系统调试 → 定位根因 → 出修复方案
      优先级: 高
      恢复: 会话级覆盖（调试状态下保持谨慎）

  冲突解决:
    - "新功能" 匹配 tdd 和 brainstorming → tdd 优先（更具体）
    - 多个技能同时匹配 → 列出让用户选（不要猜）

  手动技能:
    /reset: 恢复主配置，清除所有技能覆盖
    /skills: 列出当前可用技能
```

### 写作场景

```yaml
技能触发规则（写作场景）:

  技能清单:
    outline:
      触发: "写大纲" "规划结构" "目录"
      做什么: 先生成结构框架，用户确认后再填充内容
      优先级: 高
      恢复: 立即回到主配置

    deep-research:
      触发: 用户明确说"研究" "调研" + 话题有足够深度
      做什么: 多源搜索 → 验证 → 引用报告
      优先级: 中
      恢复: 立即回到主配置
      注意: 触发前确认成本（"这需要深度研究，约需X tokens，继续？"）

  冲突解决:
    - "研究一下怎么写这个主题" → 优先 outline（写作为主），研究辅助
```

## 六、ECC 对标

| ECC 模块 | 对应本维度 |
|----------|-----------|
| Skills (251 个) | 3.1 技能清单 |
| Agents (63 个子 Agent) | 技能清单中的"会话级覆盖"型技能 |
| Commands (79 个, `/plan` `/tdd`) | 3.2 手动触发 |
| Hooks (SessionStart) | 技能加载器——会话启动时注册可用技能 |

**ECC 缺少的**：技能优先级规则和冲突解决策略。251 个技能中可能有 10+ 个同时匹配一个输入（如"审查"可能匹配 code-review / security-review / agent-audit），ECC 没有在配置层定义谁优先。

**本维度补充的**：触发条件的层次（精确/语义/手动）、优先级规则、恢复机制。

## 七、常见陷阱

| 陷阱 | 为什么错 | 怎么改 |
|------|---------|--------|
| 技能描述太短或太模糊 | LLM 不知道技能做什么 → 不触发 | 每个技能至少写：触发条件 + 做什么 + 不触发条件 |
| 所有技能平等，没有优先级 | 多个技能同时匹配 → 随机选 → 不可预测 | 定义优先级规则（高成本>低成本，精确>语义） |
| 技能执行完不恢复主配置 | 技能的行为规则持续生效 → 污染后续对话 | 默认立即恢复，例外情况（如设计模式）显式声明 |
| 技能清单太长（50+ 技能全注册）| 上下文窗口被技能描述占满 | 按场景分组，只注册当前场景相关的技能 |
| 触发了高成本技能但没确认 | Deep research 消耗 3M tokens，用户不知道 | 高成本技能（>100K tokens）触发前确认 |
| 技能和 Agent 功能重叠 | 用户分不清"是技能做的还是 Agent 自己做的" | 技能开始和结束时明确声明（"我现在进入调试模式…调试完成，回到正常模式"） |

## 八、真实样例

### 样例 1：编程 Agent 的技能触发规则

来自用户生产环境中的技能配置。以下规则确保 Agent 不会在简单的"查文件"请求上触发重量级的 brainstorming 技能：

```
技能使用规则：
1. 技能是用 Skill 工具触发的——看到 <skill-trigger> 或用户说"/技能名"时才触发
2. 不为了让回答"显得完整"而多触发技能
3. 技能触发后，技能本身的指令优先于主配置
4. 同一个会话中同一技能最多触发 3 次——超过则不再自动匹配
```

### 样例 2：ECC 的技能注册模式

ECC 的技能通过 `/plugin` 系统管理——安装、启用、禁用、卸载。技能元数据（名称、描述、触发条件）集中在 `manifest` 中：

```yaml
# ECC 技能元数据模式
skill:
  name: debugging
  description: 系统调试——先定位根因再出方案
  triggers: [bug, 报错, 失败, 异常, 修一下]
  priority: high
  recovery: session  # 调试状态下保持谨慎，直到会话结束
  cost: low
```

**可以向 ECC 学什么**：元数据集中管理是好的。**ECC 可以补什么**：触发条件目前只是关键词列表，缺少语义匹配和优先级规则。
