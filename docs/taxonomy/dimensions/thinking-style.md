# 思维方式

> 维度类型：个人 | 实现方式：文本注入 | 验证方式：LLM 判定 + 无验证

---

## 一、定义

**控制 Agent 怎么处理信息、怎么做决定、用什么原则思考。**

这不是"Agent 输出什么"（推理模板），也不是"Agent 不能做什么"（行为策略）。这是 Agent 的**元认知层**——在输出之前，它怎么想。

### 控制什么

- **原则**：先判断再实现、先审查再行动、不确定就停下来问
- **启发式**：优先最小改动、不扩写不做无关优化、不猜不装懂
- **决策倾向**：保守还是激进、务实还是探索、跟随用户还是主动质疑
- **信息处理方式**：先拆问题再回答、先找已有方案再决定自研

### 不控制什么

- 具体的禁止/强制操作（那是行为策略的事）
- 输出的语气和长度（那是交互风格的事）
- 回答的结构格式（那是推理模板的事）

### 为什么这是个人维度

不同的人思考方式差异巨大。一个人可能偏好"先结论后依据"，另一个人可能偏好"先展开再收敛"。好的 Agent 应该匹配用户的思维方式，而不是强加一种"正确"的思维模式。

**但**：有些思维原则是跨用户通用的——比如"不确定就停下来问"。这些通用原则可以放在行为策略或推理模板中。

## 三、填写方法论

### 3.1 有效写法：原则 + 启发式 + 例子

**有效写法**（原则清晰，有具体可操作的启发式）：

```
思维方式：
  - 原则：先判断再实现。
    在写任何代码之前，先判断方案是否成立。
    如果发现自己在顺着设想直接往下写，必须先停下来。

  - 原则：不确定就明确说明。
    如果信息不足以做出判断，不要猜。明确说"我缺少以下信息…"
    如果用户的问题有歧义，列出可能的理解，请用户澄清。

  - 启发式：优先最小改动。
    改 3 行能解决的，不改 30 行。
    发现关联问题时告知用户，不要顺手一起改。

  - 决策倾向：务实优先。
    先用最简单的方法解决问题，再考虑是否需要优化。
    不做"以防万一"的设计。
```

**无效写法**（太抽象，LLM 不知道怎么做）：

```
# ❌ 差
思维方式：理性、严谨、高效、创造性思维
```

"理性"对 LLM 没有操作意义——它需要的是具体的行为指令。

### 3.2 思维原则的粒度

一个好的思维原则有三个部分：

```
原则: {判断标准}
动作: {当标准满足时，做什么}
反例: {什么情况下不适用}
```

**实例**：

```
原则: 先审查再行动
动作: 接到任务后，先检查——有没有成熟方案？有没有更低成本路径？有没有遗漏的前提？
反例: 简单的信息查询（"这个 API 怎么用"）不需要走完整审查流程。

原则: 优先最小改动
动作: 修改范围不超过完成任务所需的最小文件集。发现关联问题时标注而非直接改。
反例: 安全漏洞修复可以扩大范围（因为关联问题可能是同一漏洞的不同入口）。
```

### 3.3 CC 源码中的思维方式规则

Claude Code 源码泄露暴露的静态段中包含以下思维原则（CC 认为对所有用户通用的）：

```
- "Read and understand existing code before suggesting modifications"
- "Prefer editing existing files over creating new ones"
- "Never assume a library is available — check first"
- "Don't add error handling for scenarios that can't happen"
- "Delete unused code completely rather than adding compatibility shims"
```

这些是 CC 认为所有编程 Agent 都应该遵守的思维原则。它们的特点是：**反直觉但正确**——新手可能会"加个 try-catch 以防万一"，但好的 Agent 知道不应该。

## 四、验证方式

### 4.1 LLM 判定

思维方式是语义模糊的，不适合规则匹配。用 flash 模型做定期抽样检查：

```
检查点: 每 5 轮对话后
检查内容:
  - Agent 在回答前是否做了足够的审查？
  - Agent 是否在不确定时猜测了信息？
  - Agent 是否做了不必要的扩写？
判定方式: flash LLM 看最近 5 轮对话，给出 1-5 分
阈值: <3 分 → FLAG（审计记录），不 BLOCK
```

**注意**：思维方式的验证成本高且不精确。建议在早期不做频繁验证，而是定期复盘时人工检查。

### 4.2 无验证（自检代替他检）

替代方案：在反思提示词中让 Agent 自己检查：

```
反思提示词（思维方式部分）：
"我在回答之前是否做了足够的审查？
 我有没有在不该猜的地方猜测了？
 我有没有把推测包装成事实？"
```

这比外部验证更有效——因为 LLM 自己知道自己的思考过程。

## 五、场景模板

### 编程场景

```
思维方式（编程）:

  原则:
    - 先判断方向，再进代码。如果发现自己"一头扎进代码"，停下来。
    - 先找已有方案。不要自己实现一个已经被标准库解决的问题。
    - 不确定就停下来问。不要猜用户的环境、偏好、意图。

  启发式:
    - 最小改动：改 3 行能解决的不改 30 行。发现关联问题标注而非顺手改。
    - 先读再改：编辑文件前必须先用 Read 确认当前内容。
    - 先拆后答：复杂问题先拆成子问题，逐一回答。

  决策倾向:
    - 务实：先用最简单的方法。优化留到真正需要时。
    - 保守：不确定的技术选型 → 选最成熟的那个，不选最新的那个。
    - 跟随用户：用户说"不改 A"，就不要想办法绕过这个约束。
```

### 写作场景

```
思维方式（写作）:

  原则:
    - 先理解用户的写作目的，再动笔。
    - 修改前先读已有内容。不全文重写只需要微调的内容。

  启发式:
    - 最小改动：只改需要改的部分。
    - 保持原文风格：不要把自己的写作风格强加给用户。

  决策倾向:
    - 务实：用户要的是"能用的文字"，不是"优美的文字"。优先完成，再润色。
```

## 六、ECC 对标

| ECC 模块 | 对应本维度 |
|----------|-----------|
| Rules → TDD (80% 覆盖率) | 思维原则——先测试再实现 |
| Rules → Plan Before Execute | 思维原则——先规划再执行 |
| Agents → planner agent | 思维原则的独立实例化——把"先规划"交给专门的子 Agent |
| Skills → brainstorming | 思维方式的一种预设——探索性思考 |

ECC 的 Rules 中有些规则（如 TDD）既是思维原则也是行为策略。区别在于：思维原则是"我倾向于这样想"，行为策略是"我必须这样做"。

## 七、常见陷阱

| 陷阱 | 为什么错 | 怎么改 |
|------|---------|--------|
| 原则太抽象（"理性""严谨"） | LLM 不知道"严谨"和"不严谨"的具体区别 | 每一条原则配一个具体例子 |
| 所有场景同一套思维 | 编程需要的思维方式（先审查）和写作需要的（先理解目的）不同 | 按场景区分思维方式 |
| 思维方式写太长 | 超过 10 条原则 → LLM 记不住 → 重要原则被淹没 | 前 3 条是最重要的，其余按优先级排列 |
| 混淆思维方式和行为策略 | "不删文件"不是思维方式，是行为策略 | 思维方式 = 怎么说思考，行为策略 = 怎么做操作 |

## 八、真实样例

### 样例 1：用户的全局思维方式

来自 `C:\Users\xsfwq\.claude\CLAUDE.md`：

```
默认工作方式：
- 先理解上下文，再开始动手。
- 不确定就明确说明，不要猜。
- 优先最小改动，不做无关重构。
- 多种方案同时列出，说明取舍，不要只顺着一个方向写。

主动反思（回答前要检查）：
- 我是否在顺着用户的设想直接往下写
- 我是否遗漏了已有方案或相关研究
- 我是否把"看起来可行"误当成"真的可行"
- 是否存在更成熟、更低成本的路径
```

### 样例 2：CC 源码中的思维约束

从 CC 静态段中提取的通用思维约束（这些是 Anthropic 认为所有 Agent 都该有的）：

```
- 读已有代码后再修改（不盲改）
- 优先编辑已有文件（不新建）
- 不假设库已安装（先检查）
- 删掉没有用的代码（不加兼容层）
- 不预判错误场景加过度防护（不加无用的 try-catch）
```
