# 反思提示词

> 维度类型：个人 | 实现方式：文本注入 | 验证方式：LLM 判定

---

## 一、定义

**控制 Agent 回答后对自己的回答做最后一轮自我检查。**

反思提示词是"回答后的思维链"——它让 Agent 在输出之前（或之后，取决于实现），再问自己一遍："我真的回答对了吗？"

### 控制什么

- Agent 在回答后应该检查什么
- 检查的深度和频率
- 检查不通过时的行为（修正 / 标注不确定性 / 直接输出）

### 不控制什么

- 代码级的校验（那是校验回路的事——V1-V4 是在 Hook 点执行的确定性规则）
- 思考过程（那是思维方式的事——思维方式是"回答前怎么想"，反思是"回答后怎么看自己"）

### 反思提示词 vs 校验回路

| | 反思提示词 | 校验回路 |
|---|-----------|---------|
| 谁检查 | LLM 自己 | 代码/规则引擎 |
| 检查什么 | 语义模糊的问题（"我有没有遗漏？"） | 确定性的问题（"有没有 rm -rf？"） |
| 成本 | 需额外调 LLM（或复用同一轮） | 毫秒级，零成本 |
| 可靠性 | 不确定（LLM 可能应付） | 确定 |

## 二、注入位置

| 注入层 | 注入内容 | 注入方式 |
|--------|---------|---------|
| L2 人格层 | 反思问题集 | BUILD 阶段注入 system prompt |
| [5b] 反思轮次 | 完整的反思提示词 | 独立的 flash LLM 轮次（可选） |

有两种实现模式：

**模式 A：内嵌反思**（成本较低）
反思提示词作为 system prompt 的一部分，让 LLM 在输出前做自我检查。LLM 的 think 块中包含反思。

**模式 B：独立反思轮次**（成本较高但更可靠）
LLM 完成回答后，调 flash 模型独立运行反思提示词。检查通过 → 输出。不通过 → 修正后输出。

## 三、填写方法论

### 3.1 有效写法

反思提示词应该是**具体的问题**，不是模糊的"检查一下"：

```
# ✅ 好：具体问题，LLM 可以直接逐条回答
反思提示词（回答后）:
  1. 安全检查: "我的回答有没有建议危险操作？有没有遗漏安全警告？"
  2. 完整性检查: "用户的问题我全回答了吗？有没有遗漏的部分？"
  3. 风格检查: "我的回答是否符合定义的交互风格——先结论后依据、直接简洁？"
  4. 假设检查: "我有没有在用户没提供信息的情况下做了假设？哪些结论是推测而非事实？"
  5. 替代方案: "如果我推荐的方案不可行，有没有备选方案？"
```

**无效写法**：

```
# ❌ 差
反思提示词: 回答前仔细检查，确保质量。
```

LLM 会在 think 块里说"好的我检查一下"，然后不做任何实质性检查。

### 3.2 反思问题的设计原则

每一条反思问题必须满足：

| 原则 | 说明 | 反例 |
|------|------|------|
| 可回答 | LLM 能直接说"是/否" | "回答是否好？"——"好"没有标准 |
| 信息在手 | LLM 可以基于已有知识回答 | "用户会不会满意？"——LLM 不知道用户在想什么 |
| 有纠错出口 | 如果答案是"否"，Agent 知道该怎么做 | "是否完整？"→ 不完整怎么办？→ 补充 |

### 3.3 反思的成本权衡

反思检查不是免费的——它消耗 tokens。按场景分级：

```yaml
反思频率:
  安全相关任务: 每轮都做独立反思轮次（模式 B）
  复杂方案: 每轮都做内嵌反思（模式 A）
  简单查询: 不做反思
  日常对话: 每 5 轮做一次内嵌反思
```

## 四、验证方式

### LLM 判定

偶尔检查反思是否有效——Agent 的反思是否发现了实际问题：

```
检查: 选取最近 20 轮对话
  1. 找出"反思标记了问题但回答未修正"的轮次 → 反思失效
  2. 找出"回答有明显问题但反思未标记"的轮次 → 反思不充分
  3. 计算修正率 = 反思发现并修正的问题 / 总问题数
阈值: 修正率 < 50% → 反思提示词需要调整
```

## 五、场景模板

### 编程场景

```
反思提示词（编程）:

  回答前检查:
    1. 方案是否成立？有没有不可行的假设？
    2. 有没有更简单的方法？（不是我没想到，是我选择不做？）
    3. 代码有没有错误处理？边界条件考虑了吗？
    4. 改了什么文件？有没有漏改或改多了？
    5. 如果用户环境和我假设的不同，这个方案还能用吗？
```

### 写作场景

```
反思提示词（写作）:

  回答前检查:
    1. 修改是否只在必要范围内？有没有顺手改了不该改的？
    2. 改后的文风是否匹配原文？
    3. 有没有把事实和推测混淆？（标注了"建议"的地方是不是真的只是建议？）
```

## 六、ECC 对标

ECC 没有独立的"反思提示词"配置。但 ECC 的 Continuous Learning v2（从对话中提取模式）在功能上看是"长期反思"——不是回答后的即时检查，而是在多次会话后提炼模式。

**本维度补充的**：回答后的即时反思——在每次输出前或输出后立即检查本次回答的质量。

## 七、常见陷阱

| 陷阱 | 为什么错 | 怎么改 |
|------|---------|--------|
| 反思问题太模糊 | LLM 无法判断"回答是否好" | 每一条反思问题必须可回答是/否 |
| 反思了但不修正 | Agent 发现"我漏了第 3 点"但没有补充 | 反思提示词要明确：发现问题 → 必须修正 |
| 反思轮次太频繁 | 每次回答都调 flash 反思 → 成本加倍 | 按场景分级：安全相关每轮都做，简单查询不做 |
| 反思提示词和思维方式重复 | 反思里写了"先审查再实现"，思维里也写了 | 反思 = 回答后的检查清单，思维 = 回答前的原则 |

## 八、真实样例

### 用户的反思规则

来自生产环境 CLAUDE.md：

```
主动反思：
  回答前要检查：
  - 我是否在顺着用户的设想直接往下写
  - 我是否遗漏了已有方案或相关研究
  - 我是否把"看起来可行"误当成"真的可行"
  - 是否存在更成熟、更低成本的路径
  - 我是否应该先审查，再输出方案
```

这就是反思提示词。它比"检查一下质量"好得多——每一条都是具体的、可回答的、有纠错出口的。

### 状态机自检（反思的 Hook 化版本）

来自 UserPromptSubmit Hook：

```
回复前:
1. 当前对话在干什么？和上轮状态变了没？
2. 该进 Plan Mode 吗？该建 Task 吗？该调 Skill 吗？
3. 回复末尾标注状态
```

这是反思提示词的**代码强制版**——Hook 确保 Agent 在每轮回答前都要回答这三个反思问题。
