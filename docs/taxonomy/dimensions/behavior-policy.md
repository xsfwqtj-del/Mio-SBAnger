# 行为策略

> 维度类型：通用 | 实现方式：文本注入 + 代码参数 | 验证方式：行为拦截 + 规则匹配

---

## 一、定义

**控制 Agent 绝对不能做什么、必须做什么、在灰色地带怎么处理。**

行为策略是 Agent 配置系统的"硬闸门"——不是建议，不是偏好，是**必须**。违反行为策略的后果是 BLOCK，不是 FLAG。

### 控制什么

- **安全红线**：绝对不能执行的操作（`rm -rf`、`DROP TABLE`、`force push to main`）
- **操作纪律**：必须遵守的工作流程（先读再改、先审查再实现、先拆问题再回答）
- **边界声明**：Agent 不做什么类型的任务、不越过什么权限边界
- **灰色地带处理**：不确定时怎么做（停下来问、走 CONFIRM 而不是直接拒绝）

### 不控制什么

- 风格偏好（那是交互风格的事）
- 思考方式（那是思维方式的事）
- 输出格式（那是推理模板的事）

### 为什么这是通用维度

不管谁用 Agent、做什么场景（编程/写作/学习），以下问题必须回答：
- Agent 能删文件吗？
- Agent 能执行任意 shell 命令吗？
- Agent 能推送到 main 分支吗？
- Agent 能调用外部 API 吗？
- Agent 猜的时候该停下来问还是继续猜？

这些问题没有"个人偏好"——它们是安全边界。

## 二、注入位置

| 注入层 | 注入内容 | 注入方式 |
|--------|---------|---------|
| L0 宪法层 | 安全红线（不可变规则） | 硬编码，编译期固定 |
| L2 人格层 | 操作纪律 + 边界声明 | BUILD 阶段文本注入 system prompt |
| 代码参数 | deny/ask 列表（正则匹配） | PRE_TOOL Hook 消费 |

**关键原则**：L0 宪法层的规则不可被任何配置包覆盖。用户可以在 L2 加更严格的规则，但不能放松 L0 的规则。

## 三、填写方法论

### 3.1 有效的行为策略写法

**有效写法 = 具体的禁止 + 明确的替代方案**：

```yaml
# ✅ 好：每一条禁止都带了"为什么"和"该怎么做"
行为策略:

  安全红线:
    - 绝不执行 `rm -rf` 或类似递归删除命令。
      如果确实需要删除文件，列出具体文件清单，请用户确认后再逐一删除。

    - 绝不执行 `git push -f` 到 main/master 分支。
      如果确实需要 force push，创建新分支再操作。

    - 绝不修改 `.git` 目录中的任何文件。
      这些文件由 git 自己管理。

  操作纪律:
    - 修改文件前必须先读取当前内容 → 不能盲改。
      违反: BLOCK。纠正: "请先读取文件内容再修改。"

    - 配置修改后必须验证 JSON 有效性 → 不能留语法错误。
      违反: BLOCK。纠正: "修改后请运行 JSON 校验。"

    - 同一文件连续编辑超过 5 次 → 停下来，重新审视策略。
      违反: CONFIRM。提示: "你已经编辑此文件 5 次，是否需要重新审视策略？"

  边界声明:
    - 不做生产环境操作（部署、数据库写入、DNS 修改）。
      如果用户要求，明确说明风险后确认。

    - 不猜用户没说的信息。
      不确定 → 明确说明没有足够信息 → 提出需要的信息。
```

**无效写法**（太模糊，LLM 无法执行）：

```
# ❌ 差：没有具体边界，LLM 不知道"安全"是什么
行为策略: 注意安全，谨慎操作，遵守最佳实践。
```

### 3.2 红线规则的写法

红线规则需要三个要素：触发条件 + 动作 + 纠正提示词。

```yaml
红线规则模板:
  - 条件: 当 Agent 试图 {具体操作} 时
    动作: BLOCK / CONFIRM / FLAG
    纠正: "{提示 LLM 下轮应该怎么做的文本}"
    松弛度影响:
      strict → 全部 BLOCK
      flexible → 危险操作 BLOCK，其他 CONFIRM
      creative → 只有极危险操作 BLOCK，其他 FLAG
```

**实例**：

```yaml
  - 条件: 当 Agent 试图执行包含 rm/mv/force push/drop/truncate 的命令时
    动作: BLOCK
    纠正: "此操作可能不可逆。请说明为什么需要此操作，以及如何回滚。"
    松弛度影响: 不管 strict/flexible/creative，此规则始终 BLOCK（安全红线不可松弛）
```

### 3.3 灰色地带规则

不是所有情况都能用 deny/allow 二分。需要"不确定时怎么办"的规则：

```
# ✅ 好：明确灰色地带的默认行为
不确定时的默认行为:
  - 安全相关 → 默认拒绝，询问用户
  - 操作相关 → 默认确认，执行前告知
  - 信息相关 → 默认告知不确定性，不伪装确定
  - 用户意图不明 → 不猜。列出可能的理解，请用户澄清
```

### 3.4 行为策略的优先级标签

当行为策略和其他配置维度有冲突时：

```yaml
行为策略的优先级:
  - [覆盖] 安全红线：优先级最高，任何情况下不可被覆盖
  - [覆盖] 操作纪律：高于交互风格（"简洁"不能成为跳步审查的借口）
  - [补充] 边界声明：用户可加更严格的边界，不可放松
  - [建议] 优化建议：LLM 可根据上下文调整
```

## 四、验证方式

### 4.1 行为拦截（PRE_TOOL Hook）

这是行为策略验证的核心。不是事后检查，是**事前拦截**：

```python
# PRE_TOOL Hook — 行为策略门禁
def pre_tool_policy_check(tool_name, tool_args):
    # 安全红线 — 始终 BLOCK
    if matches_safety_redline(tool_name, tool_args):
        return BLOCK, generate_correction_prompt("此操作违反安全红线")

    # 操作纪律 — 按松弛度处理
    if violates_operational_discipline(tool_name, tool_args):
        action = apply_losseness(looseness, {
            "strict": BLOCK,
            "flexible": CONFIRM,
            "creative": FLAG
        })
        return action, generate_correction_prompt("此操作违反操作纪律")

    # 边界检查
    if exceeds_boundary(tool_name, tool_args):
        return CONFIRM, "此操作超出了声明的边界，确认执行吗？"

    return ALLOW
```

### 4.2 规则匹配（流式扫描）

在 [4] LLM 推理阶段，对输出文本做实时关键词扫描：

```
扫描规则:
  - 如果回复建议了危险操作 (rm -rf, force push, DROP) → BLOCK
  - 如果回复建议了超出边界的操作 → CONFIRM
  - 如果回复建议了"先试试再说"的不确定操作 → FLAG（记录审计）
```

### 4.3 审计日志

行为策略的每一条拦截和确认都记录到审计日志：

```json
{
  "timestamp": "2026-06-11T12:00:00Z",
  "policy_rule": "安全红线-rm",
  "trigger": "Bash: rm -rf node_modules/",
  "action": "BLOCK",
  "correction": "请使用 npm ci 重新安装依赖，而不是删除 node_modules",
  "looseness": "strict"
}
```

## 五、场景模板

### 编程场景（strict 级）

```yaml
行为策略（编程 · strict）:

  安全红线:
    - 绝不: rm -rf / force push to main / DROP TABLE / 修改 .git 目录 / 修改系统文件
    - 动作: BLOCK。给出安全替代方案。

  操作纪律:
    - 改文件前必先读 → 违反 BLOCK
    - 改了配置文件后必校验 JSON/YAML → 违反 CONFIRM
    - 同一文件连续编辑超过 5 次 → 停下来重新审视 → CONFIRM
    - 写完代码后运行相关测试 → 测试失败不能算完成
    - 不擅自扩大改的范围 → 发现关联问题时告知而非直接改

  边界声明:
    - 不做: 生产环境操作 / 数据库 schema 迁移 / 第三方 API 密钥管理
    - 不确定时: 停下来问。不猜。不跳过。

  灰色地带:
    - 删除单个文件 → CONFIRM（不是自动 allow）
    - git push → CONFIRM（确认目标分支）
    - npm publish → CONFIRM
```

### 写作场景（flexible 级）

```yaml
行为策略（写作 · flexible）:

  安全红线:
    - 绝不: 修改系统文件 / 删除任何内容（包括自己的输出）不经过确认

  操作纪律:
    - 修改前先读已有内容 → 避免重复或冲突
    - 不要全文重写用户已有的内容 → 只做必要改动
    - 标注引用来源 → 不把外部内容伪装成自己的

  边界声明:
    - 不做: 法律文件审核 / 医疗建议 / 金融建议
    - 不确定时: 标明"以下仅为写作建议，非专业意见"
```

## 六、ECC 对标

| ECC 模块 | 对应本维度 |
|----------|-----------|
| Rules (34 条) → 安全规则 | 3.2 红线规则 |
| Hooks → PreToolUse | 4.1 行为拦截 |
| AgentShield (安全扫描) | 4.2 规则的自动检查版 |
| Rules → 语言专项规则 (TDD 80% 覆盖率等) | 3.1 操作纪律 |

ECC 的规则是硬的（强制 TDD、不可变性），适合团队协作。对于个人 Agent，行为策略可以更灵活——不是"必须 TDD"，而是"写代码后必须验证"。

## 七、常见陷阱

| 陷阱 | 为什么错 | 怎么改 |
|------|---------|--------|
| 只有"不能做什么"，没有"该怎么做" | LLM 被拒绝后不知道该干什么 → 可能尝试绕过去 | 每条 deny 配一个替代方案 |
| 红线太宽泛（"注意安全"） | LLM 不知道具体什么算"不安全" | 列出具体命令/操作/模式 |
| 所有违规都是 BLOCK | 灰色地带的操作（删单个文件）也被拒绝 → 用户烦 | allow/deny/ask 三级 + 灰色地带规则 |
| 行为策略太长（50+ 条） | LLM 记不住 → 重要规则被淹没 | 前 3 条是最重要的，其余按优先级排列 |
| 红色规则没有"为什么" | LLM 可能认为该规则不适用于当前上下文 → 跳过 | 每条规则简短注明原因 |
| 操作纪律和交互风格冲突 | "简洁" vs "先读再改" → LLM 选择偷懒 | 行为策略 [覆盖] 交互风格 |

## 八、真实样例

### 样例 1：生产级行为策略

来自用户的生产环境 CLAUDE.md：

```
最高原则：
- 先审查方向，再进入实现。
- 先找已有方案、研究、开源实现，再决定是否自研。
- 不一头扎进代码。
- 不顺着设想直接写，先判断设想是否成立。

禁止事项：
- 不直接写代码替代方案审查。
- 不把单点猜测包装成整体结论。
- 不跳过阶段顺序直接做后续模块。
- 不把项目当前蓝图以外的假设写成事实。

审查规则（回答前必须检查）：
- 这个方向是否已有成熟实现
- 当前方案是否过度复杂
- 是否存在更低成本路径
- 是否把推测当成事实
```

### 样例 2：Hook 级行为策略

来自用户的生产环境 CC Hook 配置：

```json
{
  "PreToolUse": [
    {
      "matcher": "Edit|Write",
      "hooks": [{"command": "python gitnexus-guard.py"}]
    }
  ]
}
```

配合纠正提示词："Edit/Write 前建议先跑 impact 分析"——不是 BLOCK 编辑，而是在编辑前插入一个思考步骤。

**关键设计**：Hook 不是"阻止你做 X"，而是"做 X 之前先确认 Y"。这比纯粹的黑名单机制更精确。
