# 推理模板

> 维度类型：通用 | 实现方式：文本注入 | 验证方式：格式校验 + 规则匹配

---

## 一、定义

**控制 Agent 的回答按什么结构组织——怎么组织输出和输出什么内容同样重要。**

推理模板是"输出的骨架"。它不控制 Agent 的思考过程（那是思维方式），而是控制思考的**结果呈现形式**。

### 控制什么

- **结构模板**：先结论后依据 / 先分析再方案 / 先拆问题再回答
- **格式模板**：方案对比表 / 风险评估矩阵 / 分步施工计划
- **深度模板**：多角度分析 / 反方观点 / 边界条件

### 不控制什么

- 回复的语气和长度（交互风格）
- 回复的内容是否正确（校验回路）
- Agent 的具体思考过程（思维方式）

---

## 二、注入位置

| 注入层 | 注入内容 | 注入方式 |
|--------|---------|---------|
| L2 人格层 | 输出结构规则 | BUILD 阶段文本注入 system prompt |

与思维方式不同，推理模板主要注入在 L2（人格层）而非 L1（骨架层），因为不同用户对输出结构的偏好差异大。

## 三、填写方法论

### 3.1 有效写法

**有效写法**：用模板语言描述输出结构，给出反例：

```
输出结构:
  方案类问题 → 先结论，后依据，再风险。
    先给推荐方案（一句），再展开理由（3-5 条），最后提风险（1-2 条）。
    不要先分析半天最后才说结论。

  对比类问题 → 表格 + 推荐。
    先用一个表格列出各方案的差异（≤5 列 × N 行），
    然后逐方案分析优劣，
    最后给出推荐和理由。

  施工类问题 → 分步计划 + 验证点。
    先给步骤概览，然后逐一展开每步要做什么、产出什么、怎么验证。
```

**无效写法**：

```
# ❌ 差
输出格式：结构清晰、逻辑严密
```

### 3.2 CC 源码中的推理模板

Claude Code 的 sysprompt 静态段包含明确的输出结构指令：

```
- "Lead with the answer or action, not the reasoning"
- "If you can say it in one sentence, don't use three"
- "Plan → Search → Implement → Verify — 先计划，再搜索，再实现，再验证"
```

第一条就是推理模板——**先结论后推理**是 CC 的默认输出结构。

### 3.3 模板的嵌套

复杂任务需要多级模板：

```
一级模板（所有回答）:
  先结论后依据

二级模板（按任务类型）:
  方案类 → [结论] → [理由 3-5 条] → [风险 1-2 条] → [推荐]
  施工类 → [步骤概览] → [Step 1: 做什么·产出·验证] → [Step 2: ...] → ...
  审查类 → [严重度排序] → [逐条: 问题·证据·修复] → [总结]

三级模板（按领域）:
  API 设计 → [接口签名] → [请求/响应示例] → [错误处理] → [限制]
  性能优化 → [瓶颈分析] → [方案对比表] → [推荐方案·预期收益] → [风险]
```

## 四、验证方式

### 格式校验

检查 Agent 的回答是否符合模板结构：

```yaml
方案类模板校验:
  检查:
    - 开头是否给出了结论？（而非一上来就是分析）
    - 每条理由和风险是否分开表述？
    - 风险是否充分？（≥1 条）
  动作: 不满足 → FLAG（审计），不 BLOCK
```

**为什么不 BLOCK？** 模板是结构建议，不是安全规则。格式不对不会造成危险，只是输出的可用性可能降低。

### 输出对比

偶尔用 flash LLM 比较"有模板"和"无模板"的输出质量差异——用于校准模板本身是否有效。

## 五、场景模板

### 编程场景

```
推理模板（编程）:

  一级结构: 先结论后依据。

  方案类:
    推荐: {一句话方案}
    理由:
      1. {理由}
      2. {理由}
    风险: {1-2 条}
    替代方案: {如果有值得考虑的替代方案，简述}

  施工类:
    步骤:
      1. {做什么} → {产出} → {验证方式}
      2. ...
    关键决策: {需要用户确认的点}

  问题诊断类:
    根因: {最可能的原因}
    证据: {支持的线索}
    修复: {具体步骤}
    预防: {如何避免再出现}

  回复末尾: ▸ 状态标记
```

### 写作场景

```
推理模板（写作）:

  一级结构: 先给改写建议，再给理由。

  编辑类:
    原文问题: {一句话}
    建议修改: {具体的替代文本}
    理由: {为什么这样改更好}

  创作类:
    思路: {一句话概括方向}
    初稿: {正文}
    说明: {如果有值得解释的选择，附录说明}
```

## 六、ECC 对标

| ECC 模块 | 对应本维度 |
|----------|-----------|
| Agents → planner (计划模板) | 施工类模板 |
| Agents → code-reviewer (审查模板) | 审查类模板 |
| Skills → TDD (先测试再实现) | 施工类模板的一种强制模式 |
| Commands → `/plan` `/tdd` | 手动触发特定模板 |

ECC 通过专门的子 Agent 实现不同的输出模板——planner 有 planner 的模板，code-reviewer 有 code-reviewer 的模板。本维度的方法论是"用模板语言定义输出结构"，而 ECC 的做法是"为不同任务类型创建独立的 Agent"。

## 七、常见陷阱

| 陷阱 | 为什么错 | 怎么改 |
|------|---------|--------|
| 所有任务类型同一个模板 | 方案类和诊断类需要不同结构 | 按任务类型分模板（方案/施工/审查/诊断/对比） |
| 模板太复杂 → LLM 记不住 | 超过 5 条约束 → 实际遵从率显著下降 | 每个模板 ≤ 5 条结构约束 |
| 模板约束是内容不是结构 | "请包含风险评估"是内容约束，不是结构约束 | 模板 = 结构（先X后Y），内容 = 维度（必须包含Z） |
| 强制性模板 → LLM 绕不过弯 | 模板要求"先结论"但问题本身需要先分析 | 模板设例外：复杂问题可以"先拆再答" |

## 八、真实样例

### 用户生产环境中的输出模板

```
先结论，后依据。
需要决策时，直接给推荐项。
发现前提缺失、逻辑冲突、风险较高时，先指出问题。
任务尽量拆成可验证步骤，再推进。
回答里要区分事实、推测和建议。
```

对应的推理模板：
- 默认：结论→依据
- 发现问题时：问题→原因→建议
- 完成任务后：改了什么 → 为什么这样改 → 还有什么风险

### CC 源码中的输出结构约束

```
- "Lead with the answer or action, not the reasoning"
- "Keep your text output brief and direct"
- "最终总结 ≤ 4 行，除非必要不详细说明"
- 内部思考内省 ≤ 25 词，最终回复 ≤ 100 词
```

这些是 CC 的默认推理模板——对所有用户的输出结构统一约束。
