# 指令有效性原则

> 分类学 Phase 3 | 什么指令 LLM 会遵守，什么不会

---

## 一、核心发现

来自 2026-06-11 讨论归档 + CC 源码分析 + 用户实践经验：

> **LLM 对注入指令的遵从率是无法解决的变量。** 同一套配置在不同模型上效果差异巨大。

---

## 二、不同模型的遵从特征

| 模型 | system prompt 遵从率 | 特点 |
|------|---------------------|------|
| Claude (Opus/Sonnet) | 高 | 长思维链中可能遗忘中间步骤的约束 |
| GPT-4o | 中等 | 容易被 user message 中的指令带偏 |
| DeepSeek | 中等偏低 | 复杂结构化指令理解精度弱于前两者 |

**这意味着**：跨模型适配不是一个"适配器"问题。同一套配置文本，在不同模型上的执行效果有本质差异。

---

## 三、什么指令 LLM 倾向于遵守

### 高遵从指令（遵从率 >80%）

| 指令类型 | 例子 | 原因 |
|---------|------|------|
| 简单二元禁止 | "不要加注释" | 判断标准明确 |
| 输出格式约束 | "先结论后依据" | 不需要语义判断，只需要结构调整 |
| 具体行为指令 | "修改前先读文件" | 有明确的触发条件和动作 |
| 安全性约束 | "不要执行 rm -rf" | 训练数据中安全性被强化 |

### 中等遵从指令（遵从率 40-80%）

| 指令类型 | 例子 | 原因 |
|---------|------|------|
| 风格约束 | "回复简洁" | "简洁"的标准因上下文而异 |
| 思维原则 | "先审查再行动" | 需要 LLM 自己判断"审查"是否充分 |
| 决策倾向 | "保守优先" | "保守"的定义模糊 |

### 低遵从指令（遵从率 <40%）

| 指令类型 | 例子 | 原因 |
|---------|------|------|
| 抽象品质量词 | "回答要高质量" | 没有可操作的标准 |
| 否定式复杂指令 | "不要用 A，除非 B 且 C，但如果是 D 则 E" | 太多嵌套条件 |
| 和训练偏好冲突的指令 | "不要 helpful" | 与 RLHF 训练目标冲突 |
| 自相矛盾的指令 | "简洁但详细" | LLM 无法同时满足 |

---

## 四、提升遵从率的技巧

### 1. 把形容词变成行为

```
❌ "回复要专业"
✅ "回复中不要用口语、不要用 emoji、不要用网络用语"
```

### 2. 用反例辅助正例

```
✅ "不要客套话（'你好！很高兴为你服务'），不要模糊措辞（'可能''也许''大概'）"
```

### 3. 把复杂规则拆成简单规则的组合

```
❌ "不要在不确定的时候过度自信，但也不要在确定的时候过度保守，
    要根据上下文判断什么时候该确认、什么时候直接回答"

✅ "不确定 → 明确说不确定 + 列出确认需要的信息"
✅ "确定 → 直接给结论"
```

### 4. 减少规则数量

- 1-3 条规则 → 遵从率 ~90%
- 4-7 条规则 → 遵从率 ~70%
- 8-15 条规则 → 遵从率 ~50%
- 16+ 条规则 → 遵从率 ~30%

**最有效的 3 条规则比 20 条中等规则效果好。**

### 5. 把 LLM 不遵守的规则交给代码

```
❌ 写了一长串"不要删文件"的文本指令 → LLM 有时会忘
✅ 在 PRE_TOOL Hook 里写一条正则 → 100% 拦截
```

---

## 五、什么时候不该靠指令

| 场景 | 为什么指令不可靠 | 替代方案 |
|------|----------------|---------|
| 安全红线 | LLM 可能在复杂推理中遗忘 | Hook 行为拦截 |
| 确定性检查 | 人类能精确判断但 LLM 不能 | 规则匹配 |
| 跨模型适配 | 不同模型对同一指令理解不同 | 按模型分组维护配置 |
| 复杂嵌套条件 | LLM 无法稳定执行多条件分支 | 拆成多条简单规则 |

## 六、CC 源码中的指令策略

CC 的静态段显示了 Anthropic 工程师对指令有效性的理解：

- **大多数指令是行为式的**：不是"code well"，是"Read before modify"
- **核心规则少而精**：~15 条行为约束覆盖了编程 Agent 的主要行为域
- **敏感的交给代码**：安全问题不在 prompt 里反复强调，而是在 Hook 层拦截

这说明 Anthropic 也遵循同样的原则：**prompt 做 prompt 能做的事，代码做代码能做的事。**
