# CLAUDE.md 编写方法论

> 分类学 Phase 3 | 从用户生产级 CLAUDE.md ×3 + CC 源码中萃取

---

## 一、CLAUDE.md 的本质

CLAUDE.md 是一段文本，注入 system prompt。LLM 读到它，照着做。

它不是配置文件，不是代码，不是 API 调用。它就是**文本——所以它的能力和限制都是文本的能力和限制**：
- 能做的：定义规则、描述偏好、给出原则
- 不能做的：强制执行（需要 Hook）、交叉验证（需要校验回路）、跨模型适配（每个模型对同一段文本的理解不同）

---

## 二、三层 CLAUDE.md 架构

从用户的生产环境提炼出的三层分法：

```
全局 CLAUDE.md          项目 CLAUDE.md          技能/场景 CLAUDE.md
(C:\Users\xsfwq\.claude\)  (.\CLAUDE.md)          (skills/)
─────────────────────    ──────────────────      ──────────────
  我是谁                  当前在做什么             这个场景怎么处理
  我怎么思考              什么阶段                  特定领域的规则
  我怎么表达              什么规则不能破            临时覆盖主配置

  优先级: 最高             优先级: 中              优先级: 场景内最高
  生命周期: 长期稳定        生命周期: 随项目演变      生命周期: 技能执行期间
```

### 2.1 全局 CLAUDE.md — 写什么

- **思维方式**：怎么处理信息、怎么做决策
- **交互风格**：语气、长度、语言
- **行为策略**：绝对不能做的事
- **审查规则**：回答前必须检查什么

**不写**：特定项目的信息（项目名、技术栈、当前阶段）——那是项目 CLAUDE.md 的事。

### 2.2 项目 CLAUDE.md — 写什么

- **项目定位**：这个仓库做什么
- **目录约定**：关键目录、入口位置
- **常用命令**：启动、测试、构建
- **工作约定**：特殊流程、特殊限制
- **当前阶段**：处于什么阶段，什么不能跳

**不写**：全局规则（已经在全局 CLAUDE.md 里了）、个人偏好（也是全局的事）。

### 2.3 技能/场景 CLAUDE.md — 写什么

- **触发条件**：什么时候激活这个技能
- **临时规则**：该场景下的特殊行为约束
- **恢复机制**：技能结束后回到什么状态

## 三、编写原则

### 原则 1：行为指令，不是形容词

```
❌ "注意安全"          — 形容词，LLM 不知道什么是"安全"
✅ "不要执行 rm -rf"   — 行为指令，LLM 可以直接判断
```

### 原则 2：有"不"，也有"怎么做"

```
❌ "不要盲改文件"                           — 只有禁令，LLM 被拦后不知道怎么办
✅ "修改文件前必须先读取当前内容"             — 有替代行为
```

### 原则 3：优先级显式声明

```
❌ 规则 20 条，平等排列                   — 冲突时 LLM 不知道谁优先
✅ 行为策略 [覆盖] 交互风格               — 明确优先级
```

### 原则 4：可以验证，不是只能信任

```
❌ "Agent 应该在回答前审查"                  — 完全靠 LLM 自觉
✅ "回答前审查" + Hook 注入状态自检问题      — 有验证机制
```

### 原则 5：最小规则集

```
❌ 50 条规则 → LLM 记不住 → 实际遵从 ~30%
✅ 前 3 条最重要 → LLM 能记住 → 实际遵从 ~80%
```

如果规则超过 10 条，按优先级排列，前 3 条是最关键的。

## 四、CC 源码中的方法论验证

CC 源码泄露暴露的静态段规则，与上述原则完全一致：

| CC 规则 | 对应原则 |
|---------|---------|
| "DO NOT ADD **ANY** COMMENTS unless asked" | 行为指令，不是形容词 |
| "Never assume a library is available — check first" | 有"不"，也有"怎么做" |
| "Lead with the answer, not the reasoning" | 行为指令 |
| "Prefer editing existing files over creating new ones" | 有替代行为 |

Anthropic 的工程师已经验证了这些原则——他们选择的规则都是行为指令式的，不是形容词式的。

## 五、常见的有效句式

| 目的 | 句式 | 例子 |
|------|------|------|
| 禁止 | "不要/绝不 + 具体行为" | "不要盲改文件" |
| 要求 | "必须先 + 行为，再 + 行为" | "必须先读文件，再编辑" |
| 优先 | "优先 + 行为，避免 + 行为" | "优先编辑已有文件，避免新建" |
| 不确定时 | "不确定就 + 行为" | "不确定就停下来问" |
| 冲突消解 | "A [覆盖] B" | "行为策略 [覆盖] 交互风格" |

## 六、维护节奏

- **每周花 5 分钟**：看哪些规则 LLM 遵守了、哪些没遵守
- **发现一次错误**：加一条规则
- **发现三次同类错误**：规则可能需要调整（LLM 不理解当前写法）
- **规则超过 20 条**：审查是否可合并、是否可降级、是否已过时
