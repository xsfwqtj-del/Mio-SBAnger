# 配置进化机制

> 分类学 Phase 3 | CLAUDE.md 不是静态的——它应该随 Agent 对用户的理解而演化

---

## 一、核心理念

CLAUDE.md 是活的。好的配置不是一次写成然后永远不变的——它在每次对话中被验证，在每次发现中被修正，在每次归纳中被丰富。

但进化的方向必须可控。配置不应该"自动变异"——每一次改变都应该有明确的触发原因和可追溯的来源。

---

## 二、触发配置更新的条件

### 触发源 1：错误（最强信号）

```
Agent 犯了一个配置本可以防止的错误
  → session-to-memory → 错误类记忆
  → 问：当前配置的哪个维度可以防止这个错误？
  → 写入对应维度
  → 验证：下次类似场景，错误是否复现？
```

### 触发源 2：用户纠正

```
用户说"不对，你应该..."
  → 这条纠正 = 一条未写进配置的偏好
  → 判断：这是一次性的还是重复性的？
  → 如果是重复性的 → 写入对应维度
```

### 触发源 3：模式识别（≥3 次）

```
同一类行为连续出现 ≥3 次
  → session-to-memory → 模式类记忆
  → 问：这个模式是好的还是坏的？
  → 好的 → 固化为配置（"应该继续这样做"）
  → 坏的 → 加规则防止（"不要这样做"）
```

### 触发源 4：外部知识注入

```
从 CC 源码分析 / ECC / 酒馆 / 他人的实践中学到了新东西
  → 不是"照抄别人的配置"
  → 而是"别人的做法映射到我分类学的哪个维度，我的配置在这个维度上是否需要调整"
```

---

## 三、配置变更的审计

每次修改 CLAUDE.md 时，记录：

```markdown
## YYYY-MM-DD 更新

- 触发: {错误/纠正/模式/知识}
- 来源: {会话 ID / memory 文件 / 外部参考}
- 维度: {行为策略 / 交互风格 / ...}
- 旧: {改之前是什么}
- 新: {改之后是什么}
- 预期效果: {改了之后 Agent 应该有什么不同}
```

**为什么需要审计**：如果一条规则加了之后效果变差（Agent 变得过于保守 / 频繁误解规则），你需要能回滚。

---

## 四、不要进化的东西

| 不要进化 | 原因 |
|---------|------|
| 安全红线 | 出厂默认的安全红线不可被"放松" |
| 出厂默认值 | 出厂默认只在发布新版时更新 |
| 未经 ≥3 次验证的模式 | 一次出现的"偏好"可能是噪音 |
| 和现有配置矛盾的规则 | 至少要先解决冲突再加新规则 |

---

## 五、进化的节奏

```
日常使用: 不改配置，只观察和记录
每周 5 分钟: 翻看最近的 session-to-memory 产出，挑 1-2 条更新
每月 15 分钟: 审查全部配置，去掉过时的，合并重复的，调整优先级
每季度/项目节点: 重新跑分类学自检问卷，看是否有新维度被遗漏
```

这不是"维护负担"——如果 5 分钟/周就能维持一套精准的行为配置，ROI 远超"每次从零告诉 Agent 怎么做"。

---

## 六、闭环验证

改了一条配置后，在下周检查：

1. **Agent 行为是否按预期改变了？** 直接观察 3-5 轮包含相关场景的对话
2. **有没有副用作？** 加了"不要 X"是否导致 Agent 在其他场景也过度回避？
3. **是否需要调整粒度？** 规则太宽 → 需要具体化；规则太窄 → 需要泛化

**如果改了一条规则但 2 周内没有观察到任何行为变化 → 这条规则可能没有生效（注入位置不对 / 被其他规则覆盖 / LLM 不理解）。**
