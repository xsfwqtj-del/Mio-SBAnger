# ADR-0004: 进化循环 v0.1.0 → v2.1 重构

**日期**: 2026-06-11
**状态**: accepted
**决策者**: 烬汐

## 上下文

v0.1.0 的进化循环（`evolution-loop-spec.md`）采用望闻问切搜写装记验纠进十二步闭环，基于中医四诊 + STaR 自举循环设计。经过实际审视发现以下问题：

1. **触发太窄**：触发词是"进化"——只覆盖项目自检场景，不覆盖"用户有个需求想做"的日常场景
2. **单人搜索不可靠**：一个子 Agent 搜完就写方案，没有交叉验证——模型随机性意味着可能搜到偏的方向
3. **事后打分不如事中收敛**：验用 3 子 Agent 打分，不通过就回退重跑——全流程重跑，浪费 token
4. **子 Agent 过度使用**：从切到进全是子 Agent——频繁调子 Agent 导致缓存失效
5. **沉淀太分散**：进拆到 error-diagnosis、出厂默认、CONTEXT、ADR 四个位置
6. **记的执行者矛盾**：第一节说主 Agent 负责记，但立意后主 Agent 不应自己动手
7. **流程图并行关系画错**：切搜写装画成并行，实际是顺序

## 决策

### D1: 十二步 → 九步，重新定义流程结构

**旧**：望→闻→问→立意→切→搜→写→装→记→验→纠→进

**新**：理解与拓展→立意确认→竞争搜索→比对审计→最终审计→结果汇报→备份→执行→决策树

核心结构变化：
- 前半段（步骤 1-2）：主 Agent 理解需求 + 用户确认立意
- 中段（步骤 3-5）：子 Agent 竞争搜索 + 比对审计 + 收敛循环
- 后半段（步骤 6-9）：主 Agent 汇报 + 执行 + 沉淀

### D2: 搜索从单人变竞争

≥3 个子 Agent 独立搜索，空上下文（不知道其他 Agent 在做什么）。

**理由**：模型有随机性。3 个独立搜索方向一致 = 高置信度。方向不一致时，整合上下文再研究，循环到收敛——不给错误结果出现的机会（替代旧版"验不通过就回退重跑"）。

### D3: 偏离审查融入审计

步骤 4 和 5 都包含偏离审查——结论偏离立意 = 错误答案，不是"另一个有效方向"。

**理由**：旧版验只检查"改好了没"，不检查"结论是否跑偏了"。偏离是比执行失败更隐蔽的错误模式。

### D4: 用户确认后主 Agent 自己完成

步骤 7-9（备份、执行、决策树）由主 Agent 自己完成，不调子 Agent。

**理由**：频繁调子 Agent 导致缓存失效。轻量操作（备份、记决策树、执行已有方案）不需要独立上下文审查。

### D5: 沉淀统一为 ADR + 决策树

- 步骤 6 产出 ADR（为什么做这个决策）
- 步骤 9 产出决策树更新（决策落地改了什么）
- 都是 append-only，标注时间戳，只新增不替换

**理由**：旧版进分散到 4 个位置——新会话找信息要在多个文件中跳转。两个文件、统一格式、时间戳索引 = 可追溯的决策链。

## 替代方案

### 保留旧十二步，只在细节上修

**淘汰理由**：问题不是细节问题——是结构问题（触发太窄、单人搜索、事后打分）。修细节改不了结构。

### 完全不调子 Agent，主 Agent 一条龙

**淘汰理由**：主 Agent 自我审查的可靠性问题——同一张嘴不能又做手术又做麻醉评估。竞争搜索必须有独立上下文。

## 后果

### 积极的
- 触发从"进化"扩展到任意用户需求——日常使用场景全覆盖
- 竞争搜索 + 收敛循环替代事后验证——不给错误出现的机会
- 缓存友好——用户确认后不再调子 Agent
- 决策树 append-only——永远可追溯
- 偏离审查——防止最常见的隐蔽错误模式

### 消极的
- 子 Agent 调用量可能更大（N≥3，上不封顶 vs 旧版 1 个搜 Agent）
- 收敛循环可能多轮——如果方向持续不一致，token 消耗高于旧版
- 步骤 4 收敛上限需要明确——无限循环的风险

### 风险
- **收敛循环上限**：已设"多轮不收敛→标注分歧→如实汇报"，但"多轮"的具体数字待实践标定
- **偏离判定精度**：偏离审查由 LLM 执行，可能误判"合理的范围扩展"为"偏离"
