## 〇、进化是什么

**让一个项目能自己持续变好的机制。**

不是一次性检查，不是诊断报告——是理解需求、竞争研究、收敛审计、用户确认、执行沉淀的完整闭环。

主 Agent 统调度，子 Agent 只在搜索和审计阶段介入。任何对项目有意义的改动都应当走此流程。

---

## 一、角色分工

只有两个角色：**用户**和**主 Agent**。

| 角色 | 做什么 | 不做什么 |
|------|--------|---------|
| **用户** | 提需求；立意确认；执行前确认方案 | 不动手、不研究、不验证 |
| **主 Agent** | 理解拓展需求、立意确认、调度子 Agent、结果汇报、备份、执行、写 ADR 与决策树 | 立意确认后不做方向判断、不自己研究 |
| **搜索子 Agent** | 按分配的策略标签独立搜索，产出结构化结论 | 不知道其他 Agent 的存在和进展 |
| **审计 Agent** | 合规检查、偏离审查、收敛判断、最终审计 | 不参与搜索；必须是全新 Agent，不能是搜索池中的任何一个 |

---

## 二、核心原则

| 原则 | 含义 |
|------|------|
| 主 Agent 只调度 | 立意确认后主 Agent 不做研究判断，只派发任务和汇报 |
| 搜索 Agent 相互隔离 | 各自独立，不知道其他 Agent 的存在和进展 |
| 策略差异保证独立性 | 每个搜索 Agent 持有不同策略标签，收敛才有意义——不是靠随机种子 |
| 审计 Agent 独立 | 必须是全新 Agent，不能是搜索池中的任何一个 |
| 收敛才落地 | 未收敛不进入执行。满 3 轮仍不收敛 → 分歧上抛用户 |
| 备份先于执行 | 改动前必须留快照 |
| 只新增不替换 | ADR 和决策树都是 append-only，标注时间戳 |
| 进化自己也可以被进化 | 每次跑完循环，检查循环本身有没有要修的地方 |

### Step 1：理解与拓展（主 Agent）

分两步走——先读目标，再对照现状。

**① 回验项目立意**

先读项目的主文档（index.md / README / CLAUDE.md 开头），提取项目自己声称的目标：

> 这个项目声称自己是什么？要达成什么？

这是整个进化循环的参照系。后续所有观察都围绕一个问题：**项目离自己的立意还有多远？**

**② 四诊对照**

拿着项目立意，做四诊：

| 诊 | 问什么 |
|----|--------|
| **望** | 对照立意，项目目前实际是什么状态？差距在哪？ |
| **闻** | 有哪些隐性需求——项目立意暗示了但还没做的事？ |
| **问** | 用户说"进化 X"，真正想解决的问题是什么？（不一定是字面意思） |
| **切** | 有哪些约束、禁区、前置条件？ |

**产出**：拓展需求 = 用户意图 + 立意差距 + 边界条件

---

### Step 2：立意确认（主 Agent ↔ 用户）

向用户反问，格式如下：

```
你的需求是不是：[拓展后的需求]？
特别想确认：[最不确定的那一条]
```

用户确认 → 形成**立意**。

**立意一经确认，全流程冻结，不得修改。** 偏离立意的结论 = 错误，无论结论本身是否合理。

> **分界线**：立意确认后，主 Agent 只负责调度和汇报，不做方向判断。

---

### Step 3：独立搜索（子 Agent ×N，N≥3）

主 Agent 同时派出 N 个搜索子 Agent。每个 Agent 收到：
- 完整立意
- **唯一策略标签**（各 Agent 标签必须不同）
- 必要的项目上下文

每个 Agent 不知道其他 Agent 的存在、策略、进展。

**标准策略集**（N=3 时取前三，N 更大时向后扩展）：

| 优先级 | 标签 | 搜索视角 |
|--------|------|---------|
| 1 | 成功案例型 | 同类问题被成功解决的案例，提取可复用模式 |
| 2 | 反例型 | 同类方案失败或带来副作用的案例，识别风险 |
| 3 | 边界条件型 | 技术/实现层面的约束与上限，判断可行性边界 |
| 4 | 竞品参照型 | 成熟方案的设计决策，提取可借鉴处 |
| 5 | 第一性原理型 | 从问题本质推导，不依赖案例 |

**每个 Agent 产出格式**：

```
策略标签：[...]
搜索结论：[基于此策略的发现]
推荐方向：[具体的方向建议]
置信度：高 / 中 / 低
主要风险：[1-3 条]
```

**为什么策略标签比空上下文更好**：空上下文只保证了"不被别人带偏"，但不能保证多样性——5 个 Agent 可能都不约而同搜到同一个方向。策略标签从任务分配层面保证了**结构化多样性**——每个 Agent 被要求从不同的视角看问题，收敛才是真的收敛。

---

### Step 4：比对审计（审计 Agent，可循环，最多 3 轮）

> ⚠️ 审计 Agent 必须是全新独立 Agent，不能是 Phase 2 搜索池中的任何一个。

拿到所有搜索子 Agent 的产出，依次执行：

**① 合规检查**
- 每个 Agent 是否按所分配的策略标签执行？
- 有无明显偏题、逻辑跳跃或信息缺失？

**② 偏离审查**
- 各 Agent 的推荐方向是否与立意对齐？
- 标记所有偏离立意的结论（标记为无效，不参与方向判断）

**③ 收敛判断**

满足以下**全部条件**视为收敛：
- 有效方向（未被偏离标记）全部一致，或差异仅在实现细节层面
- 无有效方向明显互斥

| 判断结果 | 处置 |
|----------|------|
| ✅ 收敛 | 进入 Step 5 |
| ❌ 方向分歧 | 启动**调整合 Agent**（见下） |
| ❌ 某 Agent 偏离立意 | **打回该 Agent**，带修正指令重搜，结果纳入本轮审计 |

**调整合 Agent**（分歧时触发）：
- 接收所有分歧方向（合并上下文——不在分叉中持续分裂）
- 研究分歧根源，尝试形成统一方向或判定孰优孰劣
- 产出新的统一推荐 → 送回 Step 4 审计

**偏离打回 vs 直接丢弃**：偏离不丢弃，给修正指令打回重搜。这个 Agent 在搜索过程中积累的上下文有价值——可能只是方向偏了一点，修正后仍能贡献有效信息。

**终止条件**：
- 收敛 → 正常退出进入 Step 5
- 满 3 轮仍未收敛 → 主 Agent 将分歧方向列表上报用户，由用户选择后继续

---

### Step 5：最终合规审计（同审计 Agent）

对收敛结果做终检：
- 方向与立意完全对齐？
- 操作清单逻辑自洽、无遗漏？
- 是否存在未处理的关键风险？

通过 → 主 Agent 接收结果，进入汇报。

---

### Step 6：结果汇报（主 Agent → 用户）

```
方向：[X]
选择理由：[Y]
要做的操作：
  1. [操作 A]
  2. [操作 B]
  ...
已知风险：[风险列表]
```

同步写入 **ADR 条目**（方向级决策，只写一次，不在执行阶段修改）：

```
时间戳：[timestamp]
立意：[确认的需求]
选定方向：[...]
选择理由：[...]
放弃的方向：[...] 及原因
信息来源：搜索 Agent ×N，策略标签：[列表]
```

**ADR 和决策树的分工**：
- ADR：为什么走这条路（方向级）
- 决策树：走的过程中遇到什么、怎么选的（执行级）

等待用户确认。

---

### Step 7：备份（主 Agent）

对当前项目状态留快照。
- 轻量项目：文件副本
- Git 项目：commit 当前状态
- 重型项目：待后续设计（全量快照、数据库级备份）

---

### Step 8：执行（主 Agent）

主 Agent 按操作清单逐项执行。

执行过程中同步维护**决策树**（执行级，append-only + 时间戳）：

```
[timestamp] 执行：[操作 X]
  → 选择 [方案 A]，原因：[...]（备选：[方案 B]，排除原因：[...]）
[timestamp] 遭遇：[异常或分叉]
  → 处置：[...]
```

## 四、产出边界

| 产出 | 谁写 | 何时写 | 是否可修改 |
|------|------|--------|-----------|
| 立意 | 主 Agent + 用户 | Step 2 | ❌ 冻结 |
| 搜索结论 | 搜索子 Agent ×N | Step 3 | — |
| ADR | 主 Agent | Step 6 | ❌ 只写一次 |
| 决策树 | 主 Agent | Step 8（持续） | ❌ 只追加 |

## 五、设计来源

| 机制 | 来源 | 我们怎么用 |
|------|------|-----------|
| 策略标签驱动的竞争搜索 | Multi-agent ensemble + structured diversity | Step 3：N≥3 子 Agent，不同策略标签，隔离上下文 |
| 收敛循环 | Debate-style alignment, Iterative consensus | Step 4：方向不一致时调整合 Agent 整合上下文再研究，循环到收敛或满 3 轮 |
| 偏离审查 | Constitutional AI 对齐检查 | Step 4+5：结论偏离立意 = 错误，打回重搜而非直接丢弃 |
| 决策树 + ADR | ADR (Nygard, 2011) | Step 6+8：只新增不替换，时间戳，来源+理由+放弃的方向 |
| 人类监督点 | ANCHOR | Step 2 立意确认 + Step 6 结果汇报——两个关键决策点 |
| 四诊框架 | 中医辨证论治 | Step 1：望(状态)→闻(隐性)→问(本质)→切(约束)，不跳诊 |
| 失败即梯度 | STaR / Failure-as-Gradient | Step 4 偏离打回：不是惩罚，是给修正信号让 Agent 重新对齐 |

## 六、项目接入

一个项目接入进化循环，最少需要：

| 条件 | 说明 |
|------|------|
| 信息源 | 搜去哪里找（WebSearch / GitHub / Context7 / 项目内部文档） |
| 落地工具 | 装怎么改（代码项目直接 Edit，配置类项目调用对应技能） |
| 决策存储 | ADR 和决策树写哪（两个文件，append-only） |
| 项目上下文 | 主 Agent 理解需求需要读什么（CONTEXT.md 或等价物） |

分类学是第一个接入的项目。分类学的落地工具是 writer/tools/discovery 技能，记忆系统是 error-diagnosis + CONTEXT.md + change-log。

## 七、边界与约束

- **Step 1-2 不动手搜索**。只理解，不研究。
- **Step 3-5 必须调子 Agent**。尊重角色分工——主 Agent 立意后不自己研究。
- **收敛上限**：Step 4 最多 3 轮。满 3 轮不收敛 → 上抛用户，不无限循环。
- **审计 Agent 独立性**：必须是全新 Agent，不能复用搜索池中的任何一个。
- **Step 8 执行**：方案已有，不重规划。如需细化调 agent 的 plan 工具。
- **子 Agent 最小化**：用户确认后（Step 7-8）主 Agent 自己完成，不频繁调子 Agent 导致缓存失效。
- **轻量路径**：简单需求（改一个配置值、修一个路径引用）→ 主 Agent 可直接理解→立意→确认→执行，跳过竞争搜索和审计。判断标准：是否需要外部信息。
- **ADR + 决策树分离**：方向级决策写 ADR（Step 6），执行级细节写决策树（Step 8）。轻量改动可合并。
- **用户可随时中断并重新立意**。
- **进化循环本身也可以被进化**——发现循环有问题，用循环改进循环。

## 八、名词解释

| 术语 | 含义 |
|------|------|
| 进化 | 项目的自循环完善机制——从需求理解到执行沉淀的完整闭环 |
| 立意 | Step 2 用户确认后的精确目标 + 边界声明。全流程冻结，后续所有步骤的锚点 |
| 策略标签 | 分配给搜索 Agent 的唯一搜索视角（成功案例/反例/边界条件/竞品参照/第一性原理）。用结构化多样性保证收敛有意义 |
| 隔离上下文 | 每个搜索 Agent 只知道自己的策略标签和立意，不知道其他 Agent 在做什么 |
| 比对审计 | Step 4——合规检查 + 偏离审查 + 收敛判断 |
| 偏离 | 结论偏离了立意。在审计中标记为无效，打回重搜（给修正指令，不直接丢弃） |
| 收敛 | 有效方向全部一致且无互斥，差异仅在实现细节层面 |
| 调整合 | 分歧时将不同方向合并到一个上下文，研究分歧根源，尝试统一 |
| 决策树 | append-only 的执行级改动记录，标注时间戳——项目变好的轨迹 |
| ADR | Architecture Decision Record——方向级决策记录，"为什么走这条路" |
| 主 Agent | 统调度者。理解需求、立意确认、汇报结果、执行、写 ADR 与决策树 |
| 搜索子 Agent | 被调度者。按策略标签独立搜索，不知道其他 Agent 的存在 |
| 审计 Agent | 独立 Agent。合规检查、偏离审查、收敛判断。必须是全新 Agent |
