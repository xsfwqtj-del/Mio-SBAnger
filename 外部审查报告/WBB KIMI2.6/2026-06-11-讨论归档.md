> 参与者：用户（MioForever 项目负责人）、Senior Developer（高级开发工程师）
> 时间：2026-06-11 02:56 ~ 03:56（约 1 小时）
> 主题：审查 AI Agent 项目架构设计，最终得出"不做系统，只写文本"的结论

## 一、起点：审查 MioForever 蓝图

### 1.1 项目背景

MioForever 是一个全能个人陪伴型 AI 助手项目，核心概念是**"马具工程"**——给 AI Agent 戴上可配置的约束系统，让它更可控、更贴合用户。

蓝图中定义了项目的三大核心：
1. **配置导入**：用户如何定义和导入"马具配置"
2. **注入执行**：配置如何在 LLM 推理前注入 system prompt
3. **校验**：LLM 回复后如何检验是否遵守了规则

其余一切（多 Agent 框架、GUI、生态分享、认知提炼）都是叠在这三个核心之上的应用层。

已有设计基础：
- 配置 = 文本注入 + 验证规则（天然成对）
- 三层优先级：用户本地 > 导入配置包 > 出厂默认
- 8 个可扩展配置维度（Section）
- 配置热切换

### 1.2 讨论：三个核心模块的设计建议

**审查结论**（三个模块分别建议）：

**配置导入**：
- 建议弃目录结构，用单文件 `.mio`（ZIP 压缩包），内含 manifest + config
- config 用自由 key-value，不硬编码 8 Section
- 每项配置必须含 `prompt`（文本注入）和 `validation`（验证规则）
- 用 JSON Schema 做导入校验
- 用"层叠覆盖栈"实现优先级

**注入执行**：
- 采用物理四层模型：stable → volatile → brief → hard
- 8 Section 是逻辑模型，四层是物理模型，必须显式维护映射表
- 放弃"流式输出中途纠正"——技术上不可行（标准 API 是单向 SSE 流）
- 纠正流程改为：完成 → 扫描 → 不满足 → 重调

**校验**：
- 三层架构：L1 语法层（本地）→ L2 规则层（本地）→ L3 语义层（调 LLM）
- 语义层不稳定，建议用**异构模型评判**（pro 生成 + flash 评判）
- 验证规则声明式配置：类型、标准、阈值、动作
- 回环必须设硬上限（工具 max=10，拦截/判定 max=3）

## 二、风险追问：什么最可能无法实现？

### 2.1 第一轮：记忆网（MAGMA 四图）

用户问：如果项目落地，哪个模块无法实现的概率最高？

**结论：记忆网的慢通道因果推断。**

理由：
1. **LLM 无法稳定做因果推断**：flash 级模型在开放域因果推断上准确率很低，伪因果泛滥，类型判定不稳定
2. **存储层完全没设计**：用什么数据库？查询性能？图规模上限？三种拓扑结构（时间链、语义网、因果图）的查询模式完全不同
3. **检索精选可能不成立**：如果因果边质量低，"沿四边扩散"可能不如直接用向量检索

**建议**：检索精选先用"向量 + 时间窗口"实现，四图作为后台实验管线异步跑，不阻塞主流程。

### 2.2 第二轮：配置注入的执行力

用户追问：配置导入到配置执行呢？

**结论：LLM 对注入指令的遵从率是无法解决的变量。**

| 模型 | system prompt 遵从率 |
|------|---------------------|
| Claude | 高，长思维链会遗忘中间步骤 |
| GPT-4o | 中等，容易被 user message 带偏 |
| DeepSeek | 中等偏低，复杂结构化指令理解精度弱 |

同一套配置在不同模型上效果差异巨大，这不是适配器能解决的。

**建议**：每份配置声明兼容模型列表；BUILD 阶段加"指令冲突检测"。

## 三、方向转向：从"马具约束"到"意图转译"

### 3.1 用户提出新方向

用户假设推翻全部蓝图，只聚焦于一个功能：

> **把用户说的话转译成更能让 Agent 理解和执行的话。**

**为什么这个方向更锋利？**

原蓝图假设"LLM 不听话，需要约束"。但真问题可能是"用户没说清楚，LLM 听错了"——LLM 不是不听话，是听懂了错误的话。

### 3.2 核心设计：意图编译器（Intent Compiler）

核心链路：
```
用户输入（自然语言）
    ↓
意图编译器
    ├── 意图解析（提取真实需求、隐含假设）
    ├── 上下文补全（检索记忆、推断 expertise）
    └── 结构化转译（输出 LLM 最优理解的格式）
    ↓
结构化指令 → LLM
```

**关键设计决策**：
- 转译不是"扩写"，是"结构化"（YAML/JSON 中间表示）
- 三层转译渐进增强：L1 模板填充（本地）→ L2 轻量解析（flash）→ L3 深度推理（pro）
- 逆向校验：转译完成后用自然语言翻译回给用户确认

### 3.3 可进化数据库的加入

用户提出：再加一个可进化数据库。

重新定义为三层：
- **L1 偏好记忆**：key-value 存储用户习惯（"不用 React"），确定性覆盖
- **L2 模式识别**：从重复行为中推断模式（"用户通常要完整交付物"）
- **L3 认知图**：远期，只对高频实体做小规模图化

**关键约束**：L1/L2 不依赖 LLM，本地规则就能做。L3 才是 LLM 的"智能进化"。

### 3.4 从个人数据库到集体智慧网络

用户进一步提出核心设想：

> **本地规范性提取每一次用户输入，转换为固定模式（时间、前因、原句），在所有人公用的大数据库中匹配相同场景，找出好评最高的转译完的语句，返回本地发送给大模型。**

这就是**意图转译网络（Intent Translation Network, ITN）**的雏形。

## 四、ITN 架构的完整设计

### 4.1 核心概念

- **场景签名（Scene Signature）**：用户输入的匿名化、结构化指纹。六维度：`intent` `domain` `complexity` `temporal` `causal_chain` `tone`
- **转译结果（Translation Result）**：发给 LLM 的结构化指令，不是自然语言
- **匹配度（Match Score）**：基于维度相似度计算，不是字符串相似

### 4.2 系统架构

```
用户设备（本地）
    ├── 输入捕获
    ├── 场景签名生成（轻量、本地、隐私安全）
    ├── 本地缓存/个人模板库
    └── 签名上传（匿名化，不上传原始输入）
            ↓
云端服务
    ├── 签名接收网关
    ├── 匹配引擎（精确 → 模糊 → 粗粒度）
    ├── 结果排序器（好评度 + 新鲜度 + 置信度）
    └── 集体转译数据库
            ↓
用户设备（本地）
    ├── 格式化器（结构化 → 文本）
    ├── LLM 适配器（Claude/GPT/DeepSeek 各自格式）
    └── 反馈收集器（隐式行为追踪）
```

### 4.3 关键设计点

| 设计点 | 说明 |
|--------|------|
| 隐私安全 | 不上传原始输入，只上传结构化签名。即使泄露也无法反推 |
| 反馈机制 | 纯隐式：直接执行(+2)、小幅编辑(+1)、切换候选(-1)、删除重来(-2) |
| 模型适配 | 按 `model_family` 分组存储和匹配，不同模型有不同转译池 |
| 回环终止 | 工具回环 max=10，拦截/判定回环 max=3 |
| 冲突消解 | 配置包冲突检测，BUILD 阶段提前告警 |

### 4.4 完整数据库 Schema

设计了四张核心表：`scene_signatures`（签名表）、`translation_results`（转译结果表）、`feedback_records`（反馈记录表）、`user_reputation`（用户可信度表）。

### 4.5 酒馆（SillyTavern）的借鉴

从酒馆的 Prompt 构建管线中提取了可借鉴经验：
- 自包含单文件配置（`.png` 嵌入 JSON）
- 多层有序 prompt（system → 角色 → 动态 → 对话历史）
- World Info 条件触发注入
- 宏替换系统（`{{user}}` `{{char}}`）
- 层级分离原则（规则/角色/场景/示例）

## 五、层层质疑与诚实推翻

### 5.1 第一轮质疑：复杂场景的意图理解

用户指出 ITN 文档没有解决**"大项目局部拆解"**的问题：

> 用户说"帮我把那个接口改成 WebSocket"——"那个接口"在大项目语境中才有意义，脱离大项目毫无信息量。

**第一次尝试解决**：设计了"意图悬空检测 + 项目上下文图 + 意图补全"三层架构。

**被推翻的原因**：项目图从哪来？
- 让用户手动维护 → 没人会用
- 从代码仓库自动解析 → 只能接 GitHub，且准确率有限
- 从对话历史自动提取 → 冷启动时为空
- 让 LLM 猜 → 猜错了是灾难

**代词消解在开放域是未解决的 NLP 问题。**

### 5.2 第二轮质疑：分词器有必要吗？

用户问：如果只做嵌在 LLM-Agent 中间这层，分词器有必要吗？

**分析**：
- 如果走"本地规则路径"→ 需要分词器（jieba + spaCy 混合）
- 如果走"LLM 中间层路径"→ 不需要分词器，直接用轻量 LLM 做意图解析

**结论**：MVP 直接走路径 B（LLM 做意图解析），分词器可以删掉。等验证出价值后，再把高频意图换成本地规则来降本。

### 5.3 第三轮质疑：还有类似已做过的项目吗？

梳理了部分重叠的已有项目：

| 项目 | 重叠点 | 差异点 |
|------|--------|--------|
| DSPy（斯坦福） | 也做"意图 → 结构化指令" | 面向开发者写代码，不是实时对话中间件 |
| Semantic Router（Aurelio AI） | 签名 → 匹配 → 路由 | 只决定"走哪条路"，不做"怎么转译" |
| PromptPerfect / PromptLayer | 也在改进 prompt | 假设用户已写好 prompt，只做润色 |

**ITN 的独特壁垒**：跨模型适配、集体智慧、实时反馈闭环、隐私优先——但这四个壁垒都建立在"能拿到数据"的前提下。

### 5.4 第四轮质疑：数据从哪来？

用户尖锐指出：

> "能拿到的数据：该用户在这个插件内的输入和输出。违法的吧？"

**确实违法。**

- 拦截用户输入输出做数据分析，违反 GDPR、个人信息保护法、CCPA
- 即使写了"用户授权"，实际执行中授权条款站不住脚
- "匿名化"后上传在技术上做不到真正匿名（输入内容本身就是身份标识符）

**ITN 的数据飞轮在法律和隐私层面无法启动。**
- 没有数据就没有匹配
- 集体智慧无法积累
- 网络效应永远不会出现

### 5.5 第五轮质疑：这些都悬空

用户追问：你设计的那些"把悬空引用映射到项目图实体"能做到完美吗？

**不能。** 并诚实承认：如果两个文件（CLAUDE.md）已经解决了 80% 的问题，那一堆模块（底盘、行为引擎、记忆网、配置分享生态）到底想做什么？

**答案是：想做"动态的 CLAUDE.md"——自动从对话中学习、跨对话记忆、行为校验、配置分享。**

但每个愿望的实现路径上全是坑：
- 自动学习 → 需要大量用户数据，隐私不允许
- 跨对话记忆 → 需要记忆网，工程复杂度极高
- 行为校验 → LLM 遵从率不可控
- 配置分享 → 生态冷启动

## 六、最终结论：答案就是 CLAUDE.md

### 6.1 归一：所有方案的共同骨架

```
贴合度 = f(身份, 偏好, 上下文, 表达方式)
```

所有设计文档都是在这个公式里挑一个维度做深：

| 文档 | 主攻维度 |
|------|---------|
| MioForever 蓝图 | 身份 + 偏好（8 Section 配置） |
| ITN 文档 | 表达方式（集体智慧匹配） |
| 分词器设计 | 表达方式（结构化转译） |
| 项目上下文 | 上下文（大项目绑定） |

### 6.2 最终发现

用户自己的两个文件就是答案：

```
全局 CLAUDE.md ──→ 我是谁、我怎么思考、我怎么表达
项目 CLAUDE.md ──→ 当前在做什么、什么阶段、什么规则不能破
```

**这就是"让 LLM 更贴合我心意"的终极简化版。**

没有云端数据库，没有意图匹配，没有集体智慧，没有反馈闭环——只有一段文本，塞进 system prompt，LLM 读到，照着做。

### 6.3 为什么这比 ITN 更好

| 维度 | ITN（集体智慧网络） | 一张 CLAUDE.md 卡片 |
|------|-------------------|-------------------|
| 启动成本 | 需要云端 + 数据 + 匹配算法 | 0，现在就能用 |
| 隐私 | 需要上传用户数据 | 完全本地 |
| 贴合度 | 理论上高，实际上冷启动为 0 | 由用户自己写，100% 贴合 |
| 维护 | 系统维护 + 数据维护 | 偶尔更新卡片 |
| 效果 | 不确定 | 立竿见影 |

### 6.4 诚实的建议

**别做 MioForever 了。**

把力气花在：
1. **写好全局 CLAUDE.md**（用户已经写了，而且写得很好）
2. **为每个重要项目写项目级 CLAUDE.md**（用户也已经写了）
3. **建立"卡片模板库"**——不同场景（编程、写作、学习、规划）的 CLAUDE.md 模板，一键切换
4. **定期复盘和迭代**——每周花 5 分钟看哪些规则 LLM 遵守了、哪些没遵守，调整文本

**这就是个人开发者能做到的最优解。** 所有超过"文本卡片"的复杂度，都是把简单问题复杂化。

## 七、讨论的价值

这场讨论的价值不在于产出了什么系统，而在于**诚实推翻了三个层次的幻想**：

| 层次 | 幻想 | 推翻者 |
|------|------|--------|
| 技术层 | "用集体智慧匹配最优转译" | 隐私法不允许数据收集 |
| 产品层 | "自动理解用户复杂意图" | 开放域代词消解是未解问题 |
| 架构层 | "马具工程能约束 LLM 行为" | LLM 遵从率本身就是不可控变量 |

最终回到一个朴素的真理：

> **最简单的工具往往就是对的工具。**

一段写在 Markdown 里的规则文本，胜过一万行分布式系统的代码。
