# 蓝图审查报告

## 一、矛盾（5条）

### 1. 治理体系定位冲突 — 框架图 vs 第七章
- **位置**：第一章第74-78行（框架图）vs 第七章第432-433行
- **矛盾**：框架图将「治理体系（烬汐居）」画在 MioForever 整体生态框内，与基础 Agent、配置分享层并列。但第七章明确说「烬汐居不是 Mio 的治理模块，是 Mio 的元范式孵化器」，图注标注「不在 Agent 设计内」。
- **影响**：影响对 MioForever 整体分层结构的理解。
- **修复**：框架图中将治理体系移出整体生态框，以指向箭头表示产出注入关系。

### 2. 嵌套关系文本与框架图不一致 — 第82行 vs 框架图
- **位置**：第一章第82行 vs 第48-79行框架图
- **矛盾**：文本写「基础 Agent ☐ 治理体系内」，暗示基础 Agent 被包含在治理体系内部。框架图却将两者绘制为平级独立模块。且 ☐（U+2610）作为嵌套符号与全文框线体系不一致。
- **影响**：第一章的基础架构描述自相矛盾，误导后续章节的大局观。
- **修复**：改为「治理体系、基础 Agent、配置分享层三者平级」，或修正为「基础 Agent ☐ 整体生态」（如原文本有笔误）。

### 3. Phase 8 状态标注互斥 — 📐 vs 🔍
- **位置**：Phase 8（第495-498行）vs 5.6 开放API（第401-405行）
- **矛盾**：Phase 8（配置分享生态）标注 📐（设计完成待施工），但其子模块 5.6 开放API 标注 🔍（待设计）。📐 与 🔍 在状态图例中互斥。
- **影响**：蓝图自身的状态追踪不可靠，后续施工阶段无法依赖状态判断进度。
- **修复**：Phase 8 状态改为 📐/🔍（如同 Phase 5 表述），或明确标注 5.6 为后继项目。

### 4. 底盘 vs 3.1 主引擎状态矛盾
- **位置**：第89行（底盘）vs 第134-135行（3.1 主引擎）vs 第471行（Phase时序）
- **矛盾**：底盘标注 🔍（待设计），3.1 Mio 主引擎标注 🔧（施工中）且强依赖底盘5项能力。时序箭头暗示顺序依赖，但依赖方状态超前于被依赖方。文档以「寄生 Hermes」解释但状态系统无法体现这一区别。
- **影响**：Phase 1 和 Phase 2 的关系定义模糊，底盘和主引擎的依赖链无法通过状态追踪。
- **修复**：3.1 状态改为「🔧(寄生验证)/🔍(待移植)」，或在 Phase 2 注释注明 v0.4 寄生运行、待底盘就位后移植。

### 5. 「配置回流」仅为概念定义，无对应模块落实
- **位置**：第16-21行（三股力量）、第84行（核心数据流终点）
- **矛盾**：「配置回流」被定义为三股力量之一、核心数据流终点（会话落盘→session-to-memory→配置回流），但全文无模块或过程名为「配置回流」或承担此职责。最近的 4.5 反馈闭环+底盘⑤也没有被定义为回流管道。
- **影响**：三股力量的架构声明缺少落地保障，核心数据流在端到端层面断裂。
- **修复**：方案A—将引用替换为「反馈闭环→配置注入管道」；方案B—新增模块正式定义配置回流的输入/输出及与 4.5 的关系。

## 四、建议修复优先级

### 第一优先 — 开工前必须修复

蓝图本身的一致性决定了后续所有施工能否依赖它作为设计文档：

1. **矛盾#1（治理体系定位）** + **矛盾#2（嵌套关系）** — 第一章是蓝图的总纲，读完第一章如果各元件的关系都是错的，后面全部建立在不稳定的地基上。这两个修完框架图就定型了。
2. **矛盾#3（Phase 8 状态互斥）** — 蓝图的状态系统如果不可信，后续无法用它追踪 Phase 进度，施工排期的依据就是错的。
3. **矛盾#5（配置回流悬空）** — 三股力量是架构声明的核心之一，核心数据流断裂意味着 session-to-memory 的输出端没有定义对接方。

### 第二优先 — Phase 1 底盘前必须补

底盘是有具体输入/输出的数字件，合约签不清就看不清它提供什么、主引擎依赖什么：

4. **缺口#2（四层注入未展开）** — 底盘⑤是主引擎和认知子系统的依赖核心，四层注入的定义直接影响记忆注入、配置注入、反馈闭环的设计。
5. **缺口#15（API 密钥管理）** — 实践层硬需求，没有这个底盘无法运行。
6. **前瞻#1（成本预算引擎）** + **前瞻#3（降级策略）** — 这两条在底盘层嵌入成本最低，延迟到 Phase 2 以后改造成本高 3-5 倍。

### 第三优先 — Phase 2 主引擎移植前必须补

7. **缺口#1（8 Section Profile 未定义）** — P0 缺口，配置体系的核心抽象。Profile 运行时、穿搭组合、Pack Loader 全部依赖它，不定义的话后面 Phase 3-5 都无法开工。
8. **缺口#4（Hook 生命周期接口签名）** — 行为引擎的核心载体，不改影响工具系统、反馈闭环、插件系统的全部接口定义。
9. **前瞻#2（可观测性三件套）** — 主引擎移植过程中必须同时建立排查能力。

### 第四优先 — 持续补充，不阻塞 Phase 排期

10. 剩余 P1 缺口（#3、#5-11）— 功能细节层面的空白，修改或新增模块时补上即可，不阻塞整体排期。
11. 剩余 P2 缺口（#12-18）— 语义级细化，不阻塞开发。
12. 前瞻#5-13 — 架构前瞻，部分需要 Phase 推进到一定程度才具备实施条件，保持追踪即可。
