## 二、蓝图审计

---

### S 级 — 会导致整个系统失败的根问题

---

**S1：底盘选型悬空，整个蓝图的起点是开放问题**

- **问题**：Phase 1 的核心任务是「拆四个备选→取零件→自组装」，但四个候选（Pi SDK/HanaAgent/Hermes/Reasonix）没有评估结论。蓝图在「不清晰」一栏明确写了「零件验收标准（'能用'的定义）」。交给陌生开发者，第一个问题就是：我从哪里开始？按什么标准判断「能用」？
- **影响**：Phase 2–9 全部依赖底盘。没有底盘，行为引擎没地方挂 Hook，Pack Loader 没地方注入配置，整个蓝图是空中楼阁。
- **修复建议**：蓝图中补充底盘评估矩阵，至少包含：LLM 循环可拆性/工具系统接口/TypeScript vs Python 立场/许可证/可维护性。给出一个推荐选项，并定义「零件验收」的最低标准（例如：可独立运行一次 LLM 调用、可注入自定义 Hook、可模型切换）。

---

**S2：四层注入与 8 Section 关系未定义——配置体系的核心接口是歧义的**

- **问题**：底盘⑤写了「四层注入：stable / volatile / brief / hard」；三.4 和〇.B 写了「8 Section Profile」。两套模型同时存在，没有定义映射关系。实现底盘⑤的开发者和实现三.4 的开发者会做出两个互不兼容的东西。
- **影响**：Pack Loader、Profile 运行时、BUILD 阶段的注入逻辑全部依赖这个接口。这是配置体系的主干接口，歧义在这里等于整个配置分享层（五）施工前地基已裂。
- **修复建议**：二选一，不能并存不定义。方案一（推荐）：明确 8 Section 是逻辑模型，四层注入是物理层，定义映射（例：L0→hard、L1→stable、L2/8-Section→volatile、L3简报→brief）。方案二：废弃四层注入命名，全部用 8 Section 驱动 BUILD，四层只作为内部实现细节不对外暴露。

---

**S3：流式输出中「注入纠正不中断流式」技术上不可实现**

- **问题**：[4] LLM 推理步骤写道「触发纠正 → 在同一轮内继续推理（不中断流式）」。标准 LLM API 的流式是单向流——一旦开始，不能在同一次 API 调用中途注入新内容并继续。要纠正必须 abort + 重新调用。
- **影响**：思维链验证（〇.B.1 的核心举例）和推理模板验证的实现机制失效。如果开发者直接按蓝图施工，到实现层会发现这个设计做不到，不得不重新设计这个环节。
- **修复建议**：将「同一轮内不中断」改为「流式完成后扫描 → 不满足 → 注入纠正提示词 → 重调用 pro」。或者如果「不中断」的意思是「用户感知不到中断」（两次调用对用户透明），必须在蓝图中明确说明这是实现细节，不是 API 语义。

---

**S4：三个回环无终止条件**

- **问题**：〇.3 明确定义了三个回环（工具回环/拦截回环/判定回环），但没有任何回环定义：最大迭代次数、超时机制、死循环检测。
- **影响**：触发场景——LLM 生成被拦截 → 注入纠正提示词 → LLM 重新生成 → 再次被拦截 → 无限循环。生产环境中会导致 API 成本无限消耗和进程卡死。三个回环都有这个问题，但拦截回环（[6a]→[4]）和判定回环（[5]→[4]）风险最高。
- **修复建议**：每个回环添加 `max_iterations`（建议：工具回环 10，拦截回环 3，判定回环 3），超出后强制返回带错误标记的响应，写入审计日志。这是最小安全保护，必须在底盘层实现。

---

### A 级 — 架构级缺陷，会导致关键模块失效或需要重大返工

---

**A1：session-to-memory v2.4.0 输出格式与 MAGMA 四图接口不兼容**

- **问题**：四.1（✅ v2.4.0）输出「11 分类记忆文件」（平铺结构）。四.4 MAGMA 快通道第①步要求「session-to-memory 提取碎片 → 事件节点」（图节点格式）。蓝图没有定义这个转换层，也没有说谁来做这个转换。
- **影响**：Phase 4（检索精选注入）和 Phase 5（记忆网）的施工会在这里卡住。开发者要么改 v2.4.0（破坏已有功能），要么自己加适配器（但蓝图没说）。
- **修复建议**：明确声明：v2.4.0 保持现有接口不变（11 分类文件输出稳定），新增独立 `EventNodeAdapter` 层负责将记忆条目转换为 MAGMA 图节点，作为 MAGMA 快通道的第 0 步。

---

**A2：V2 语义合理性「毫秒级本地判定」与「语义理解」逻辑矛盾**

- **问题**：底盘③和三.2 声称 V2 是「本地毫秒级」，同时 V2 的内容是「方向性质疑：Agent 有权对用户假设提出独立挑战」。判断用户方向性假设是否存在问题，本质是语义理解任务，不是规则匹配，无法在本地毫秒级完成。
- **影响**：开发者无法实现这个设计。要么做成简单关键词匹配（能毫秒，但不是语义级），要么调 flash LLM（能语义，但不是毫秒，且增加延迟和成本）。两种实现结果完全不同，蓝图没有给开发者做决定的依据。
- **修复建议**：拆分 V2 定义：V2a（本地毫秒级）= 工具调用方向的结构化规则匹配（例如：检测到「删除所有」类操作 → 触发 CONFIRM）；V2b（语义级质疑）= flash LLM 判断，明确标注为可选、有额外成本。不能把两个不同实现合并在一个「V2」里。

---

**A3：慢通道 worker 可靠性机制缺失**

- **问题**：四.4 慢通道是「后台 worker + 队列驱动」，但没有定义：队列实现（内存 = 进程重启丢失 / 持久化 = 需要额外依赖）、worker 崩溃恢复（事件 ID 处理到一半怎么办）、幂等性（同一事件 ID 被重复消费的处理）、与主进程的生命周期绑定关系。
- **影响**：记忆网的完整性依赖慢通道。如果 worker 不可靠，图会有残缺的边和孤立节点，且这种退化是悄无声息的——检索结果质量下降但没有报错。
- **修复建议**：至少定义：队列持久化方案（SQLite 单表足够，两列：event_id / status）；处理状态三态（pending / processing / done）；失败重试上限（3 次后标记 failed，人工干预）。

---

**A4：ADR-0002 废弃状态未确认**

- **问题**：蓝图说 ADR-0002 已被 MAGMA 取代，方向偏差。但没有说明 ADR-0002 对应的代码在代码库中的当前状态（已删除 / 已归档 / 仍在运行）。
- **影响**：施工 MAGMA 四图时，开发者可能发现旧图代码还在运行，不知道该删还是该绕开。如果 ADR-0002 代码还在产出数据，和 MAGMA 数据会在存储层发生冲突。
- **修复建议**：在蓝图中用一行明确声明 ADR-0002 代码的当前状态，以及 MAGMA 施工是从零开始还是在旧代码基础上改造。

---

**A5：llm_wiki 在蓝图中既是依赖又是待设计——Phase 5 目标无法自洽**

- **问题**：Phase 5 目标是「会学习的 Agent」，六.知识管线把 llm_wiki 列为慢通道产出。但 llm_wiki 的设计完全缺失：没有触发条件、没有输入格式、没有输出格式，施工状态是「🔍 待定」。
- **影响**：Phase 5 施工时，「会学习」的语义是不完整的——开发者知道做四图，但不知道 llm_wiki 怎么做，会把这个缺口带到交付物里。
- **修复建议**：要么给 llm_wiki 补充最小设计（触发时机：图节点达到 N 个 / 输入：某一实体的全部关联边 / 输出：Markdown 页面 + 版本号）；要么从 Phase 5 目标中删除 llm_wiki，标注为 Phase 5.5 或 SOMEDAY，不要让一个完全未设计的模块挂在当前 Phase 的完成条件上。

### B 级 — 实现级风险，不处理会在施工中反复返工

---

**B1：KV 缓存命中策略没有具体机制**

- 问题：「保留 KV 缓存命中」在蓝图中是设计原则，但没有定义具体机制。L3 简报拼入 user message 末尾的设计理论上可以保持 prefix 缓存命中，但需要验证目标 LLM 提供商的具体缓存行为（Gemini / Anthropic / OpenAI 各不同）。
- 影响：KV 缓存命中是成本预算的核心假设，实现不当会让每轮成本高出数倍。
- 修复建议：针对目标提供商给出具体规则：system prompt L0+L1 完全静态不可修改、L3 简报只追加到 user message 末尾不插入 system prompt、L2 动态内容的注入边界。

---

**B2：约束松弛度状态生命周期未定义**

- 问题：strict / flexible / creative 三级约束松弛度贯穿全文，但没有声明：存储位置（会话级 / 全局 / Profile Section）、默认值、修改权限（用户 / 系统 / 外部配置）、会话中能否动态切换。
- 影响：[5] POST_THINK、[6a] PRE_TOOL 都依赖这个值，如果不同模块各自假设不同的默认值，行为会不一致。
- 修复建议：声明约束松弛度是 Profile 中一个可配置的字段（不注入 prompt，纯代码参数），存储在会话 Profile 快照中，BUILD 阶段加载，会话期间不可热切换，默认值 flexible。

---

**B3：.mio-pack 版本兼容性缺失**

- 问题：manifest.json 有「版本+依赖」字段，但没有定义兼容性策略：旧格式包在新版 Mio 中的处理方式、插件 API 版本变更时的行为。
- 影响：配置分享生态的长期可维护性。用户保存的 .mio-pack 在 Mio 升级后可能失效，且无任何提示。
- 修复建议：最小版本策略：manifest.json 加 `schema_version` 字段、不兼容时加载失败并返回具体错误提示、不做自动迁移（迁移是中阶用户责任）。

---

**B4：session-to-memory 已知缺陷未进入蓝图**

- 问题：蓝图用「✅ 已浇筑（v2.4.0）」标注 session-to-memory，但 manifest 断点恢复逻辑缺失、分片重叠策略未解决这两个已知问题没有进入蓝图的任何位置。
- 影响：交给陌生开发者，他会认为这个模块是干净的。实际施工时踩到这两个坑，不知道是自己做错了还是原来就有问题。
- 修复建议：在四.1 施工状态中追加：「已知未修复：manifest 断点恢复逻辑（#XXX）/ 分片重叠策略（#XXX）」，或者把状态从「✅」改为「🔧 已浇筑（有已知缺陷）」。

---

### C 级 — 细节缺陷，不紧急但会产生长期维护负担

---

**C1：出厂配置是 fallback 但 fallback 完整性未验证**

- 问题：〇.B.4 保证「任何配置项未设置，退回出厂默认，确保 Mio 永远不会裸奔」。但 8 Section 各项的出厂默认值没有列出，不知道哪些 Section 有默认值、哪些是可选的。
- 影响：新用户首次使用，某个 Section 没有出厂默认，对应 Hook 点找不到配置，行为未定义。
- 修复建议：补充出厂配置最小完整集，8 Section 每项显式声明默认值（「不限制」也是一种声明）。

---

**C2：多 Agent 框架与底盘资源隔离未设计**

- 问题：三.5 说多 Agent 实例有「独立上下文/记忆」，但没有定义多个 Agent 共享同一底盘时的资源隔离：Hook 总线命名空间、工具注册表隔离、成本预算是共享还是独立。
- 影响：Phase 6 施工时会遇到底盘没有预留隔离机制，要么返工底盘要么多 Agent 共享全局状态。
- 修复建议：在底盘⑤或行为引擎设计中，声明 Agent 实例 ID，作为 Hook 注册和工具注册的命名空间 key。

---

**C3：Galgame 选项引擎「预留接口」没有规格**

- 问题：三.5「Galgame 选项引擎：预留接口作为 ValidationLoop 的可选扩展」，但接口规格完全未定义。
- 影响：「预留」没有具体规格，等于没有预留。开发者不知道留什么槽。
- 修复建议：哪怕一行：输入格式（Agent 当前状态 + 选项列表 JSON）、输出格式（用户选择 → 触发哪种 Hook 行为）。

---

**C4：烬汐居出厂配置注入节奏未定义**

- 问题：七.说烬汐居产出「出厂配置（.mio-pack）」注入 Mio，但没有定义注入时机（编译期打包 / 启动时加载 / 运行时热更新）、用户能否覆盖出厂配置、出厂配置更新时已部署实例是否自动更新。
- 影响：烬汐居是蓝图的治理起点，但它的产物如何进入系统是一个未定义的路径。
- 修复建议：声明出厂配置是在 Build 时打包进安装包、还是作为网络资源在启动时拉取，以及用户本地配置覆盖出厂配置的优先级已在〇.B.4 定义，只需把这个机制扩展到「出厂配置更新」场景。
